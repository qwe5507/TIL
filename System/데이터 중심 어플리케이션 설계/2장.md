# 2장. 데이터 모델과 질의 언어

다루는 주제
- 데이터 모델: 관계형 / 문서 / 그래프
- 질의 언어: 선언형 vs 명령형, 맵리듀스, 사이퍼, 스파클, 데이터로그

## 이 장 핵심

데이터 모델은 코드 작성법만 바꾸는 게 아니라 문제를 보는 방식 자체를 바꾼다. 그래서 모델 선택이 중요하다.
관계형/문서/그래프 셋 다 현역이고 정답은 없다. 데이터의 "관계 모양"에 따라 갈린다. 일대다·트리면 문서, 다대일·다대다가 많으면 관계형, 사방으로 연결되면 그래프.
질의 언어는 선언형(SQL, 사이퍼)이 명령형보다 대체로 유리하다. "어떻게"를 엔진한테 맡길 수 있어서.

## 2.1 관계형 모델과 문서 모델

지금 제일 익숙한 SQL은 1970년 에드거 코드(Edgar Codd)가 낸 **관계형 모델**이 뿌리다. 데이터를 관계(relation, = 테이블)로 두고, 그 안에 순서 없는 튜플(tuple, = 로우)을 담는 구조.

처음엔 "이걸 효율적으로 구현할 수 있겠냐"는 의심을 받았는데, 1980년대 후반쯤 RDBMS + SQL이 정형 데이터의 표준이 됐다. 그 사이 경쟁자가 여럿 있었다 — 네트워크 모델, 계층 모델(70~80년대 초), 객체 DB(80년대 말~90년대 초), XML DB(2000년대 초). 다들 한때 떠들썩했지만 결국 관계형이 살아남았다.

원래 영역은 비즈니스 데이터 처리(은행 거래 같은 트랜잭션 처리 + 급여/보고 같은 일괄 처리)였는데, 지금은 웹 게시판·소셜·커머스·게임·SaaS까지 거의 다 관계형으로 돌아간다.

### NoSQL의 탄생

2010년대 들어 관계형의 독주를 흔든 게 NoSQL이다. 특정 기술을 가리키는 말이 아니라 2009년 어느 밋업 트위터 해시태그였다. 나중에 "Not Only SQL"로 다시 해석됐고.

NoSQL을 쓰게 된 동기:

- 관계형으로는 버거운 대규모 데이터셋 / 높은 쓰기 처리량을 감당할 확장성
- 상용 제품보다 무료 오픈소스 선호
- 관계형이 잘 못 하는 특수한 질의
- 관계형 스키마의 제약이 답답하고, 더 유연한 표현력이 필요

현실적으로는 한 앱 안에서 관계형이랑 NoSQL을 섞어 쓰는 경우가 많다. 이걸 **다중 저장소 지속성(polyglot persistence)** 이라고 부른다.

> 핵심 거래 데이터는 RDB, 캐시는 Redis, 이벤트/로그성은 또 다른 데. NoSQL이냐 RDB냐를 종교처럼 고를 게 아니라 데이터 성격별로 맞는 걸 붙이는 게 현실. 이 장 내내 "상황 따라 다르다"가 반복되는데 그게 결론임.

### 객체 관계형 불일치 (impedance mismatch)

요즘 코드는 대부분 객체지향인데, 객체를 관계형 테이블에 욱여넣으려면 그 사이를 번역하는 계층이 꼭 필요하다. 이 어긋남을 **임피던스 불일치(impedance mismatch)** 라고 한다. Hibernate, ActiveRecord 같은 ORM이 이 변환 보일러플레이트를 줄여주긴 하는데, 두 모델 사이의 근본적 차이를 완전히 없애진 못한다.

책에선 링크트인 이력서(빌 게이츠)를 예로 든다. 프로필 하나에 경력 여러 개, 학력 여러 개, 연락처 여러 개가 달리는 전형적인 일대다(one-to-many) 구조.

> 관계형 스키마로 표현한 링크트인 프로필(빌 게이츠). users 테이블 중심으로 positions / education / contact_info 등이 user_id로 연결됨.

이런 이력서류 데이터는 사실 JSON 문서 하나로 표현하는 게 훨씬 자연스럽다. MongoDB, RethinkDB, CouchDB, Espresso 같은 문서 DB가 이걸 지원한다. JSON으로 두면 좋은 점은 **지역성(locality)** 이다. 관계형에선 프로필 하나 꺼내려고 여러 테이블 조인을 해야 하는데, 문서는 관련 정보가 한 덩어리에 다 있어서 질의 한 번이면 끝난다. 일대다 관계가 트리 구조로 명시적으로 드러나는 것도 장점.

### 다대일과 다대다 관계

위 예제에서 지역(region)이나 업계(industry)를 자유 텍스트가 아니라 **ID**로 둔 게 포인트다. 왜 굳이 ID로 빼냐면:

- 프로필 간 표기/철자를 일관되게 유지
- 동명 도시 같은 모호함 회피
- 이름 바뀔 때 한 곳만 고치면 됨 (갱신 편의성)
- 현지화 지원 (목록을 언어별로 번역)
- 검색 품질 (지역 계층 정보 활용)

ID로 둘지 텍스트로 둘지는 결국 **중복**의 문제다. ID는 그 자체로 의미가 없으니 바뀔 일이 없다. 반대로 텍스트를 그대로 박아넣으면 그걸 쓰는 모든 레코드에 사람이 읽는 정보가 복제되고, 값이 바뀌면 전부 갱신해야 한다(쓰기 오버헤드 + 일부만 갱신되는 불일치 위험). 이 중복을 제거하는 게 **정규화(normalization)** 의 핵심.

문제는 정규화하려면 다대일(many-to-one) 관계가 필요한데, **문서 모델은 여기에 약하다**. 관계형은 조인이 쉬워서 ID로 다른 테이블 로우를 참조하는 게 자연스럽지만, 문서 DB는 트리 구조라 조인 지원이 보통 빈약하다. 조인이 없으면 앱 코드에서 여러 번 질의해서 흉내 내야 한다.

게다가 앱은 시간이 지나면 데이터가 점점 서로 얽힌다. 처음엔 조인 없는 깔끔한 문서였다가, 기능 붙이다 보면 조직·학교를 별도 엔티티로 빼야 하고(그림 2-3) 추천서 같은 다대다(many-to-many)가 생긴다(그림 2-4).

> 회사 이름은 단순 문자열이 아니라 회사 엔티티로의 링크. 회사명에 마우스 올리면 회사 정보 카드가 뜨는 스크린샷.

> 다대다 관계로 이력서를 확장. 두 사용자(정점)가 조직·학교를 공유 참조하고 추천서로 엮인다.

> **메모.** "지금은 단순하니까 그냥 텍스트로 박자"가 나중에 발목 잡는 전형이 이거다. 값이 바뀔 수 있으면 ID로 빼서 참조하고, 안 바뀌면 박아도 된다. crypto 쪽으로 치면 코인 표시명도 바뀔 수 있으니 내부적으론 ID로 관리하는 게 안전. 정규화 = "중복을 줄여서 갱신 사고를 막는 것"으로 외우면 됨.

### 문서 데이터베이스는 역사를 반복하고 있나?

"다대다를 어떻게 표현할 거냐"는 논쟁은 NoSQL보다 훨씬 오래된 거다. 1970년대 IMS까지 거슬러 올라간다.

**① 계층 모델 (hierarchical, IMS)**

JSON처럼 트리 구조. 일대다는 잘 되는데 다대다·조인이 어려웠다. 지금 문서 DB가 겪는 문제랑 판박이.

**② 네트워크 모델 (CODASYL)**

레코드가 다중 부모를 가질 수 있음. 대신 최상위에서부터 연결을 따라가는 **접근 경로(access path)** 를 프로그래머가 직접 추적해야 했다. n차원 공간을 손으로 항해하는 느낌. 질의 바꾸려면 코드를 다 뜯어고쳐야 함.

**③ 관계형 모델**

그냥 데이터를 다 펼쳐놓는다. 접근 경로를 사람이 정하는 게 아니라 **질의 최적화기(query optimizer)** 가 알아서 정한다. 그래서 새 질의/새 색인을 추가해도 기존 걸 안 건드려도 된다.

문서 DB는 일대다 중첩 레코드라는 점에서 계층 모델로 한 발 돌아간 셈이다. 다대다는 외래 키(관계형) 또는 문서 참조(document reference)로 처리하고, 읽을 때 조인/질의로 확인한다.

> 관계형이 이긴 진짜 이유가 여기 있다. "접근 경로를 사람이 짜냐 vs 옵티마이저가 짜냐". 한 번 옵티마이저를 잘 만들어두면 그 위의 모든 앱이 공짜로 혜택을 본다. 인덱스 하나 추가하고 쿼리는 안 고쳐도 빨라지는 게 이 구조 덕분.

### 관계형 vs 오늘날의 문서 DB

| | 문서 모델 | 관계형 모델 |
|---|-----------|-------------|
| 강점 | 스키마 유연성, 지역성, 앱 객체 구조와 가까움 | 조인, 다대일·다대다 지원 |
| 약점 | 조인 약함, 다대다 표현 불리 | 객체↔테이블 변환(임피던스 불일치) |
| 잘 맞는 데이터 | 문서 같은 트리(일대다) | 관계가 얽힌 데이터 |

어느 쪽이 코드를 더 간단하게 만드냐는 데이터 모양에 달렸다. 문서 같은 트리면 문서 모델, 다대다가 많으면 관계형, 사방으로 연결되면 그래프. 문서 구조를 억지로 여러 테이블로 쪼개는 걸 **찢기(shredding)** 라고 하는데, 이걸 하면 스키마도 앱 코드도 복잡해진다.

**스키마 유연성** 얘기가 중요하다. 문서 DB를 흔히 "스키마리스(schemaless)"라고 부르는데 이건 오해다. 읽는 코드가 어떤 구조를 가정하고 있으니 암묵적 스키마는 있는 거고, 단지 DB가 강제를 안 할 뿐이다.

| | 쓰기 스키마 (schema-on-write) | 읽기 스키마 (schema-on-read) |
|---|------------------------------|------------------------------|
| 방식 | 스키마 명시적, 쓸 때 DB가 검사 | 구조 강제 안 함, 읽을 때 해석 |
| 비유 | 정적 타입 언어 | 동적 타입 언어 |
| 대표 | 관계형 DB | 문서 DB |
| 유리한 때 | 모든 레코드 구조가 동일 | 레코드마다 구조 제각각 / 외부가 구조 결정 |

이게 제일 체감되는 순간이 필드를 바꿀 때다. 예를 들어 `name` 한 필드를 `first_name`/`last_name`으로 쪼갠다고 하면,

- 문서 DB: 새 문서는 새 구조로 쓰기 시작하고, 읽는 코드에서 옛 문서를 처리(`if 옛날거면 name 쪼개기`)하면 끝.
- 관계형: `ALTER TABLE ... ADD COLUMN` 후 `UPDATE`로 마이그레이션. 대부분 RDB는 ALTER가 밀리초 단위로 빠른데, MySQL은 ALTER 시 테이블을 통째 복사해서 큰 테이블이면 한참 멈출 수 있다(따로 도구로 회피).

**질의를 위한 데이터 지역성**: 전체 문서를 자주 통째로 읽는다면 한 덩어리로 저장된 문서 모델이 유리(디스크 탐색 한 번). 단 함정이 있다 — 문서의 작은 일부만 필요한데도 DB는 보통 전체 문서를 통째로 불러오고, 갱신할 때도 전체를 재작성하는 경우가 많다. 그래서 **문서는 작게 유지하라**가 권장. 지역성은 문서 모델만의 게 아니다. 구글 Spanner의 중첩 테이블, 오라클 multi-table index cluster, Bigtable의 column-family도 같은 목적.

요즘은 관계형 DB도 XML/JSON을 지원하고(PostgreSQL, MySQL, IBM DB2 등), 문서 DB도 조인 비슷한 걸 흉내 낸다. 관계형과 문서가 서로 수렴하는 중이고, 결국 하이브리드가 미래라는 게 저자 입장.

> schema-on-read를 "스키마가 없다"로 받아들이면 사고 난다. 스키마는 코드 안에 숨어 있을 뿐이라, MongoDB 같은 데 막 던져넣다 보면 나중에 "이 필드 있는 문서 없는 문서가 섞여" 있는 지옥을 본다. DB가 안 막아주는 만큼 앱이 책임진다는 뜻. 마이그레이션 부담을 DB에서 앱으로 옮긴 것뿐이다.

## 2.2 데이터를 위한 질의 언어

관계형 모델이 가져온 또 하나의 변화가 질의 방식이다. SQL은 **선언형(declarative)**, IMS·CODASYL은 **명령형(imperative)** 이었다.

명령형은 "이렇게 하나하나 해라"를 다 적는다. 상어만 골라내려면 배열을 루프 돌면서 family가 'Shark'인 것만 새 배열에 담는 식. 순서대로 연산을 지시한다.

선언형은 결과의 조건만 말한다.

```sql
SELECT * FROM animals WHERE family = 'Shark';
```

어떤 색인을 쓸지, 어떤 순서로 실행할지는 안 적는다. 그건 질의 최적화기가 정한다. 선언형이 좋은 이유:

**①** 명령형 API보다 간결하다.
**②** 엔진 내부 구현이 숨겨져 있어서, 질의를 안 고치고도 DB 성능을 올릴 수 있다(색인 추가 등).
**③** 병렬 실행에 유리하다. "어떻게"를 안 박아놨으니 엔진이 여러 코어/장비에 쪼개 돌릴 여지가 있다. 명령형은 순서를 못 박아서 병렬화가 어렵다.

### 웹에서의 선언형 질의

책에서 든 웹 비유가 직관적이다. CSS/XPath는 "selected 클래스 안의 p를 파란색으로"처럼 선언형이고, 같은 걸 JavaScript로 DOM 직접 조작하면 코드가 길고 깨지기 쉬운 명령형이 된다. SQL이 명령형 질의보다 나은 거랑 같은 이치.

### 맵리듀스 질의

**맵리듀스(MapReduce)** 는 여러 장비에서 대량 데이터를 분산 처리하는 모델(구글이 퍼뜨림). MongoDB, CouchDB가 일부 지원한다. 선언형도 완전 명령형도 아닌 어중간한 중간 지점에 있다. map과 reduce 두 함수로 질의 로직을 짠다.

중요한 제약: map/reduce는 **순수 함수(pure function)** 여야 한다. 넘겨받은 데이터만 쓰고, 추가 질의 금지, 부수효과 금지. 그래야 DB가 임의 순서로 어디서든 실행하고 장애 시 재실행해도 안전하다.

예를 들어 "월별 상어 관찰 수 집계"를 SQL로 쓰면 평범한 `GROUP BY` 한 방인데, 몽고DB 맵리듀스로 쓰면 map에서 (연월 → 마릿수)를 emit하고 reduce에서 합치는 식이라 함수 두 개를 신경 써서 짜야 한다. 한 질의 쓰자고 두 함수 조율하는 게 더 번거롭다.

그래서 MongoDB 2.2가 **집계 파이프라인(aggregation pipeline)** 이라는 선언형 질의를 따로 추가했다. 모양은 JSON 기반이라 SQL이랑 문법은 다른데 하는 일은 SQL의 부분 집합과 비슷하다. 결국 NoSQL이 슬그머니 SQL을 다시 발명하고 있는 셈.

> 맵리듀스 직접 짤 일은 요즘 거의 없지만 "순수 함수라서 분산·재실행이 안전하다"는 발상은 알아둘 가치가 있다. Kafka Streams나 분산 처리에서 멱등성·재처리 안전성 따질 때 같은 사고가 나온다.

## 2.3 그래프형 데이터 모델

다대다가 흔하고 연결이 복잡해지면 관계형으로도 버겁다. 그럴 땐 데이터를 그래프로 모델링하는 게 자연스럽다. 그래프는 **정점(vertex, 노드)** 과 **간선(edge, 관계)** 두 가지로 이뤄진다. 소셜 그래프(사람-친구), 웹 그래프(페이지-링크), 도로망(교차로-도로) 같은 게 대표적.

핵심은 그래프가 **동종 데이터에 안 묶인다**는 점이다. 페이스북은 사람·장소·이벤트·체크인·댓글을 전부 한 그래프에 담는다. 정점 종류가 달라도 단일 저장소에 일관되게 넣을 수 있다.

이 절 예제는 아이다호 출신 루시와 프랑스 본 출신 알랭이 결혼해 런던에 사는 그래프다.

<img width="558" height="323" alt="image" src="https://github.com/user-attachments/assets/a1d0ce91-1a09-4b03-a9b6-b32eeed8e231" />

> 그래프 구조 데이터 예제. 네모 상자가 정점(사람/도시/주/국가/대륙), 화살표가 간선(born_in, lives_in, within, married).

그래프를 다루는 방식은 크게 두 갈래 — **속성 그래프**(Neo4j, Titan, InfiniteGraph)와 **트리플 저장소**(Datomic, AllegroGraph). 질의 언어로는 사이퍼, 스파클, 데이터로그를 본다.

### 속성 그래프 (property graph)

정점은 (고유 식별자 / 유출 간선 집합 / 유입 간선 집합 / 속성 키-값)으로, 간선은 (식별자 / 꼬리 정점 / 머리 정점 / 관계 레이블 / 속성)으로 구성된다.

재밌는 건 이걸 관계형 테이블 두 개(vertices, edges)로도 저장할 수 있다는 거다. edges 테이블에 tail_vertex / head_vertex / label / properties를 넣고, 양쪽 정점 칼럼에 색인을 걸면 그래프를 앞뒤로 순회할 수 있다. 특정 타입을 강제하는 스키마가 없어서, 한 그래프에 온갖 종류의 정점·간선을 섞어 담아도 깔끔하게 유지된다. 그래서 발전성(요구사항 변화 수용)이 좋다.

### 사이퍼 질의 언어 (Cypher)

속성 그래프용 선언형 질의 언어. Neo4j가 만들었다. (이름은 영화 매트릭스 등장인물에서 따온 거고 암호학 cipher랑은 무관)

"미국에서 태어나 유럽에 사는 사람 이름 찾기" 같은 질의를 MATCH로 패턴을 그리듯 쓴다. 대충 이런 모양:

```cypher
MATCH (p) -[:BORN_IN]-> () -[:WITHIN*0..]-> (:Location {name:'United States'}),
      (p) -[:LIVES_IN]-> () -[:WITHIN*0..]-> (:Location {name:'Europe'})
RETURN p.name
```

`:WITHIN*0..`이 "WITHIN 간선을 0번 이상 따라가라"는 뜻(정규식 `*`랑 같은 느낌). 도시 → 주 → 국가처럼 계층을 몇 단계 타고 올라갈지 미리 안 정해도 된다.

### SQL의 그래프 질의

관계형에 그래프를 넣고 SQL로 질의할 수도 있긴 하다. 그런데 어렵다. 그래프 질의는 정점에 닿기 전에 간선을 몇 번 탈지 미리 모르기 때문에 조인 수를 고정할 수 없다. 이걸 풀려고 **재귀 공통 테이블 식(recursive CTE, `WITH RECURSIVE`)** 을 쓴다.

문제는 분량이다. 사이퍼로 **4줄**짜리 질의가 SQL 재귀 CTE로는 **29줄**이 된다. 같은 일을 하는데.

> "그래프 데이터엔 그래프 질의 언어"라는 게 이 4줄 vs 29줄 비교 하나로 설득된다. 가변 깊이 순회(친구의 친구의 친구..., 권한 상속, 카테고리 계층)가 잦으면 관계형으로 우겨넣지 말고 그래프를 고려하라는 신호. 반대로 그게 드물면 관계형으로 충분.

### 트리플 저장소와 스파클

트리플 저장소는 속성 그래프랑 거의 같은 걸 다른 용어로 말하는 거다. 모든 정보를 **(주어, 서술어, 목적어)** 세 부분으로 저장한다. 예: (짐, 좋아한다, 바나나). 주어는 그래프의 정점이고, 목적어는 둘 중 하나다 — 원시 값(문자열·숫자)이면 속성이 되고, 다른 정점이면 그건 간선이다. 표기 형식으로 Turtle / N3가 있다.

**시맨틱 웹과 RDF**: 트리플 저장소 글을 읽다 보면 시맨틱 웹 얘기가 딸려 나온다. 핵심 아이디어는 "웹사이트가 사람용 텍스트뿐 아니라 기계가 읽을 데이터도 같이 게시하면 전 인터넷이 하나의 거대한 DB가 된다"는 것. 이를 위한 게 **RDF(Resource Description Framework)** 다. 2000년대 초에 과대평가됐다가 시들었지만, 시맨틱 웹과 별개로 RDF 트리플 자체는 앱 내부 모델로 쓸 만하다. RDF에선 서로 다른 출처 데이터를 섞어도 충돌 안 나게 주어·서술어·목적어를 URI로 쓴다(접속 가능한 주소일 필요는 없고 그냥 네임스페이스).

**스파클(SPARQL)**: RDF용 트리플 질의 언어. 사이퍼가 사실 스파클의 패턴 매칭을 참고해서 만들어진 거라 둘이 닮았다. 같은 질의를 사이퍼보다 더 간결하게 쓴다.

### 초석: 데이터로그 (Datalog)

사이퍼·스파클보다 훨씬 오래된 언어(1980년대 학계). Datomic, Cascalog가 쓴다. 잘 안 알려졌지만 뒤에 나온 질의 언어들의 토대라 알아둘 값어치가 있다.

트리플을 `(주어, 서술어, 목적어)` 대신 `서술어(주어, 목적어)` 꼴로 쓴다. 예: `within(idaho, usa)`. 그리고 SELECT로 바로 묻는 게 아니라 **규칙(rule)** 을 정의해놓고 단계적으로 질의를 쌓아 올린다. 규칙은 다른 규칙을 부르거나 재귀적으로 자기를 부를 수 있어서, 복잡한 질의를 작은 조각으로 나눠 재사용할 수 있다. 프롤로그(Prolog) 해봤으면 익숙한 모양.

<img width="602" height="204" alt="image" src="https://github.com/user-attachments/assets/b0fd737b-1224-4bc8-ab0e-232a9ef72c6e" />

> 데이터로그 규칙을 반복 적용해 "아이다호가 북아메리카에 속함"을 찾아가는 과정. 규칙 1 적용 후 → 규칙 2 한 번 → 규칙 2 두 번, 테이블이 점점 채워진다.

일회성 간단 질의엔 불편한데, 데이터가 복잡할 땐 규칙을 조합·재사용할 수 있어서 강력하다.

## 2.4 요약

역사적으로 데이터를 하나의 큰 트리(계층 모델)로 표현하려다 다대다가 안 풀려서 관계형이 나왔다. 그러다 관계형도 안 맞는 데가 있다는 게 드러나면서 NoSQL이 두 갈래로 등장했다.

- **문서 DB**: 데이터가 자기 문서 안에 다 들어 있고, 문서 간 관계가 거의 없는 경우
- **그래프 DB**: 반대로 모든 게 서로 연결될 수 있는 경우

- 세 모델(문서·관계형·그래프)은 다 현역. 한 모델로 다른 걸 흉내 낼 순 있지만 결과가 엉성해서 목적별로 고른다.
- 문서·그래프의 공통점: 보통 스키마를 강제 안 함(schema-on-read). 단 스키마가 없는 게 아니라 앱 코드 안에 숨어 있는 것.
- 질의 언어도 모델마다 다름: SQL, 맵리듀스, 집계 파이프라인, 사이퍼, 스파클, 데이터로그. CSS/XPath도 (DB 언어는 아니지만) 선언형 사례.
- 책이 안 다룬 모델: 게놈 서열 유사도 검색(GenBank), 입자물리 페타바이트급(LHC), 전문 검색(full-text) 같은 특화 영역.

> 핵심: 데이터 모델은 취향이 아니라 데이터의 관계 모양에 따라 정해진다. 새 기능 설계할 때 "이거 일대다야, 다대다야, 아니면 사방으로 연결돼?"를 먼저 물으면 RDB / 문서 / 그래프 중 뭘 쓸지가 거의 자동으로 나온다. 그리고 가능하면 선언형으로 — 최적화 여지를 엔진에 남겨두는 게 이득.
