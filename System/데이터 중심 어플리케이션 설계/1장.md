# 1장. 신뢰할 수 있고 확장 가능하며 유지보수하기 쉬운 애플리케이션

## 한눈에 보기

- **데이터 시스템** = DB · 캐시 · 검색 색인 · 큐를 앱 코드로 엮은 복합체
- **신뢰성** = fault(부품 고장)가 생겨도 failure(서비스 다운)로 안 번지게 → HW / SW / 사람 결함
- **확장성** = 부하 늘 때 성능 유지 → 부하를 숫자로(부하 매개변수) → 성능을 분포로(백분위) → 확장 방식 결정
- **유지보수성** = 운용성 + 단순성 + 발전성

---

## 들어가기 전에

요즘 만드는 애플리케이션은 대부분 **계산 중심(compute-intensive)** 이 아니라 **데이터 중심(data-intensive)** 이다.
CPU 성능이 발목을 잡는 경우는 거의 없고, 진짜 골치 아픈 건 데이터의 양 / 복잡도 / 변화 속도다.

데이터 중심 앱은 보통 아래 같은 표준 구성 요소(building block)를 조합해서 만든다.

- **데이터베이스** — 나중에 다시 찾을 수 있게 데이터 저장
- **캐시** — 비싼 연산 결과를 기억해서 읽기 속도 개선
- **검색 색인(search index)** — 키워드나 다양한 조건으로 필터/검색
- **스트림 처리(stream processing)** — 다른 프로세스로 비동기 메시지 전달
- **일괄 처리(batch processing)** — 주기적으로 누적된 대량 데이터 분석

**왜 당연해 보일까?** 평소에 별생각 없이 쓸 만큼 이 도구들이 잘 만들어졌기 때문. 근데 막상 들여다보면 안 단순하다.

- 앱마다 요구사항이 달라서 → DB도 캐시 방식, 검색 색인 방식이 제각각
- 하나의 도구로 다 안 됨 → 여러 도구를 엮어야 함

> 💡 DB 하나 갖다 쓰는 걸로 끝이 아니라 캐시·검색·큐를 직접 엮어서 한 시스템처럼 굴리는 게 요즘 구조. 문제는 이것들 사이 동기화(캐시 무효화, 색인 갱신)를 대신 해주는 게 없다는 거. 다 앱 코드 몫이고, 버그도 보통 거기서 난다.

## 데이터 시스템에 대한 생각

**DB / 큐 / 캐시는 원래 다른 범주로 본다.** DB와 메시지 큐는 둘 다 한동안 데이터를 저장한다는 점은 비슷한데, 접근 패턴이 완전히 달라서 성능 특성도 구현 방식도 다르다.

**그런데 왜 이걸 다 묶어서 "데이터 시스템(data system)"이라고 부를까?**

- 새 도구들이 특정 사용 사례에 최적화되면서 → 전통적 분류 경계가 흐려짐
  - Redis = 메시지 큐로도 쓰는 데이터스토어
  - Apache Kafka = 지속성(durability)을 보장하는 메시지 큐
  - → "이건 DB, 이건 큐" 하고 딱 자르기 애매해짐
- 단일 도구로 처리 + 저장을 다 못 하는 요구사항 등장 → 작업을 잘게 나눠 여러 도구를 애플리케이션 코드로 엮음

**캐시·색인 동기화는 누구 책임?**
메인 DB와 분리된 캐시 계층(Memcached 등)이나 전문(full-text) 검색 서버(Elasticsearch / Solr)를 둘 때, 이걸 메인 DB와 동기화시키는 건 보통 **애플리케이션 코드의 몫**이다.

<img width="570" height="375" alt="image" src="https://github.com/user-attachments/assets/c88ed0e5-3161-404e-86c7-c2e16a52d80a" />

> 다양한 구성 요소를 결합한 데이터 시스템 아키텍처 예시
> 클라이언트 요청 → 애플리케이션 코드 → 캐시 / 기본 DB / 검색 색인 / 메시지 큐로 분기되는 그림

**복합 데이터 시스템(composite data system)** = 이렇게 여러 도구를 묶어 하나의 서비스처럼 보이게 만든 것.

- 외부 클라이언트에는 API로 내부 구현을 숨김
- 쓰기 때 캐시 무효화·갱신 같은 보장을 애플리케이션 코드가 책임
- → 이제 개발자는 **앱 개발자이자 데이터 시스템 설계자**다

이 책 전체를 관통하는 세 가지 관심사가 여기서 등장한다.

- **신뢰성(Reliability)** — 뭔가 잘못돼도 시스템이 계속 제대로 동작
- **확장성(Scalability)** — 부하가 늘어도 대처할 방법이 있음
- **유지보수성(Maintainability)** — 시간이 지나도 여러 사람이 생산적으로 작업 가능

## 신뢰성 (Reliability)

소프트웨어에 대한 일반적인 기대치는 대충 이렇다.

- 사용자가 기대한 기능을 수행한다
- 사용자의 실수나 예상 못 한 사용법을 어느 정도 허용한다
- 예상된 부하/데이터 양에서 충분한 성능을 낸다
- 허가되지 않은 접근과 오남용을 막는다

이걸 한 문장으로 줄이면 **"무언가 잘못되더라도 지속적으로 올바르게 동작함"** 이 신뢰성이다.

### 결함(fault) vs 장애(failure) — 이 구분이 핵심

- **결함(fault)**: 시스템의 한 구성 요소가 사양에서 벗어난 상태
- **장애(failure)**: 사용자에게 필요한 서비스를 제공하지 못하고 시스템 **전체**가 멈춘 상태

이걸 견디고 대처할 수 있는 시스템 → **내결함성(fault-tolerant)** 또는 **탄력성(resilient)** 있다고 함.

- "모든 종류의 결함을 견딘다"는 건 불가능 (블랙홀이 지구 서버를 다 삼키면 답 없음)
- → **특정 유형의 결함**에 대한 내성만 이야기하는 게 맞다
- 결함 확률을 0으로 만드는 것도 불가능 → 목표는 **결함이 곧바로 장애로 번지지 않게** 설계하는 것

> 💡 fault(부품 하나 고장) ≠ failure(서비스 전체 다운). fault 확률을 0으로 만드는 건 포기하고, fault → failure로 안 번지게 막는 게 룰.

**일부러 결함을 일으켜 내결함성을 테스트**하기도 한다.

- Netflix **카오스 몽키(Chaos Monkey)**: 운영 중에 무작위로 프로세스를 죽여봄
- 이유: 결함 처리 코드는 가끔만 동작 → 안 건드리면 진짜 장애 때 작동 안 하는 경우가 많음 → 평소에 검증
- **단, 보안은 예외**: 예방이 해결책보다 나은 영역(공격자가 데이터 손상시키면 되돌릴 수 없음) → 일부러 유도하는 접근이 안 맞음

### 하드웨어 결함

디스크 고장, RAM 결함, 정전, 누가 케이블 잘못 뽑기... 장비 많이 다루면 늘 일어나는 일이다.

- 하드디스크 평균 장애 시간(MTTF)은 약 10~50년. 디스크 10,000개짜리 클러스터면 평균 하루에 한 개씩은 죽는다고 봐야 함.
- 대응: 구성 요소에 **중복(redundancy)** 추가. RAID, 이중 전원, 핫 스왑(hot-swap) CPU 등.
- 예전엔 단일 장비 중복으로 충분했는데, 데이터 양과 연산 요구가 늘면서 더 많은 장비를 쓰게 됐고 하드웨어 결함률도 따라서 증가.
- AWS 같은 클라우드는 가상 인스턴스가 예고 없이 사라지기도 한다. 신뢰성보다 **유연성/탄력성**을 우선해서 설계됐기 때문.
- 그래서 흐름은 "하드웨어 중복" → "소프트웨어 내결함성으로 전체 장비 손실도 견디기" 쪽으로 이동 중. 운영상 장점도 있음(한 노드씩 순회하며 재부팅/패치 가능 → 전체 중단시간 없이 업데이트).

### 소프트웨어 오류

**하드웨어 결함과 다른 점:**

- 하드웨어 결함 = 무작위적·독립적 (한 디스크 죽었다고 옆 디스크도 죽진 않음)
- 소프트웨어 결함 = **체계적 오류(systematic error)** → 노드 간 상관관계가 있어서 한 번에 여러 노드를 동시에 무너뜨림

예시:

- 특정 입력에서 모든 서버 인스턴스가 죽는 버그 (예: 2012년 6월 30일 윤초 때 리눅스 커널 버그로 많은 앱이 동시에 멈춤)
- CPU/메모리/디스크/대역폭을 과도하게 잡아먹는 폭주 프로세스
- 느려지거나 잘못된 응답을 반환하는 의존 서비스
- 한 구성 요소의 작은 결함이 다른 곳의 결함을 부르는 **연쇄 장애(cascading failure)**

이런 버그는 특정 상황이 와야 터져서 오랫동안 잠복함. 빠른 해결책은 없고 아래로 줄여나가는 수밖에 없다.

- 가정/상호작용을 신중히 따져보기
- 빈틈없는 테스트
- 프로세스 격리, 죽은 프로세스 재시작 허용
- 프로덕션에서 동작 측정·모니터링

### 인적 오류

시스템을 설계·구축·운영하는 건 결국 사람인데, 사람은 미덥지 않다.

- 한 연구: **운영자의 설정 오류가 중단의 주원인**
- 하드웨어 결함은 중단 원인의 10~25% 정도밖에 안 됨

**사람이 실수해도 신뢰성 있게 만들려면:**

- **오류 가능성을 최소화하는 설계** — 잘 설계된 추상화/API/관리 인터페이스. 단, 너무 제한적이면 사람들이 우회하려고 해서 역효과.
- **사람이 실수하는 곳과 장애 나는 곳을 분리** — 실제 데이터에 영향 없는 비프로덕션 **샌드박스(sandbox)** 제공.
- **철저한 테스트** — 단위 테스트부터 통합·수동 테스트까지. 특히 정상 동작에선 거의 안 일어나는 **코너 케이스(corner case)** 에 유용.
- **빠르고 쉬운 복구** — 설정 변경 빠르게 롤백(roll back), 새 코드 점진 배포, 데이터 재계산 도구 제공.
- **모니터링** — 성능 지표와 오류율을 보는 **원격 측정(telemetry)**. 문제 추적과 장애 해석에 필수.
- **조직 차원의 교육과 실습**.

> 💡 장애 원인 1위가 사람 실수라는 게 포인트. 그래서 신뢰성은 '코드 잘 짜기'보다 '실수해도 빨리 되돌리기'에 가깝다. 롤백 빠르고 배포 안전하고 모니터링 잘 되는 게 곧 신뢰성. 코드 리뷰만큼 배포 파이프라인이 중요하다.

### 신뢰성은 얼마나 중요할까?

원전이나 항공 관제 얘기만이 아니다.

- 비즈니스 앱에서 버그 → 생산성 저하 / 매출 손실 / 평판 타격으로 직결
- "중요하지 않은" 앱도 사용자에 대한 책임이 있음 (예: 부모가 아이 사진 다 맡겨놓은 사진 앱)
- 시제품 개발, 이익률 낮은 서비스 등에선 **신뢰성을 의도적으로 희생**할 수도 있음
  - 핵심은 "지금 신뢰성을 깎고 있다"를 **자각하고** 그 시점을 아는 것

## 확장성 (Scalability)

지금 잘 돌아간다고 미래에도 잘 돈다는 보장은 없다. 흔한 원인이 **부하 증가**(동시 사용자 1만 → 10만 → 100만).

**워딩 정리:**

- "X는 확장 가능하다", "Y는 확장성이 없다" 같은 단정적 표현은 의미 없음
- 확장성을 논한다 = 아래 질문을 다루는 것
  - "시스템이 특정 방식으로 커지면 어떻게 대처할 것인가?"
  - "추가 부하를 처리하려면 자원을 어떻게 투입할 것인가?"

### 부하 기술하기 — 부하 매개변수(load parameter)

부하를 먼저 숫자로 표현할 수 있어야 "부하가 두 배 되면?" 같은 질문을 논할 수 있다. 이 숫자들이 **부하 매개변수(load parameter)**.

- 웹 서버의 초당 요청 수
- DB의 읽기 대 쓰기 비율
- 동시 활성 사용자 수
- 캐시 적중률
- ...무엇을 부하 매개변수로 잡을지는 시스템마다 다름

#### 트위터 사례 (2012년 11월 데이터 기준)

두 가지 주요 동작:

- **트윗 작성(tweet)**: 평균 초당 4.6k, 피크 때 12k 이상
- **홈 타임라인(timeline)**: 초당 300k 요청

초당 12,000건 쓰기 자체는 별거 아니다. **진짜 문제는 트윗 양이 아니라 팬 아웃(fan-out)** — 한 사용자가 많은 사람을 팔로우하고, 한 사용자를 많은 사람이 팔로우하는 구조. 구현 방식은 크게 둘이다.

**접근 방식 1 — 읽을 때 조인 (pull)**

- 홈 타임라인 요청할 때마다 → 내가 팔로우하는 사람들을 찾고 → 그들의 트윗을 다 모아 → 시간순 정렬
- 관계형 DB라면 이런 식:

```sql
SELECT tweets.*, users.* FROM tweets
  JOIN users   ON tweets.sender_id   = users.id
  JOIN follows ON follows.followee_id = users.id
  WHERE follows.follower_id = current_user
```

>  트위터 홈 타임라인 구현을 위한 간략한 관계형 스키마
> tweets / users / follows 테이블 관계도

**접근 방식 2 — 쓸 때 캐시에 삽입 (push, fan-out on write)**

- 각 사용자마다 홈 타임라인 캐시를 미리 만들어 둠
- 누가 트윗 작성 → 그 사람을 팔로우하는 모든 사람의 캐시에 새 트윗을 꽂아넣음
- 읽을 땐 미리 계산돼 있으니 저렴

<img width="585" height="221" alt="image" src="https://github.com/user-attachments/assets/cfca969e-266f-4814-b5b7-4c6aaff41766" />

>  팬 아웃을 위한 트위터 데이터 파이프라인
> 트윗 작성 → 각 팔로워에게 전송, 사용자당 최대 31M 팔로워

트위터의 선택:

- 처음엔 접근 1을 썼지만 홈 타임라인 읽기 부하를 못 버텨서 접근 2로 전환.
- 평균적으로 트윗 게시 요청보다 홈 타임라인 읽기가 수백 배 많아서, **쓰기 시점에 일을 더 하고 읽기 시점엔 일을 덜 하는 게** 유리했음.
- 단점: 트윗 하나 쓰면 팔로워(평균 약 75명) 수만큼 캐시 쓰기가 발생. 평균 4.6k 트윗 → 초당 345k 캐시 쓰기.
- 진짜 문제는 **편차**. 팔로워 3천만 넘는 사용자가 트윗 하나 쓰면 캐시 쓰기가 3천만 건. 5초 안에 전부 반영하기가 도전 과제.
- 그래서 결국 **혼합형(hybrid)** 으로 감. 대부분은 접근 2로 처리하되, 팔로워 엄청 많은 유명인 트윗만 접근 1처럼 읽을 때 합쳐줌.

> 💡 읽기가 쓰기보다 압도적으로 많으면, 쓸 때 미리 다 계산해두고 읽을 땐 그냥 꺼내 쓰는 게 이득. 트위터가 타임라인을 미리 만들어두는 이유가 이거다. 단 유명인처럼 팬아웃이 터지는 케이스는 따로 빼서(하이브리드) 처리. 피드·알림 설계할 때 그대로 써먹는 패턴이라 기억해둘 것.

### 성능 기술하기 — 처리량 vs 응답 시간

부하를 기술했으면 "부하가 늘 때 무슨 일이 일어나는지"를 본다. 두 가지 관점:

1. 자원을 그대로 두고 부하만 늘리면 → 성능이 어떻게 영향받나?
2. 성능을 유지하려면 → 자원을 얼마나 늘려야 하나?

성능을 보는 두 지표:

- **처리량(throughput)** — 초당 처리 레코드 수, 일정 작업의 전체 소요 시간. 하둡 같은 일괄 처리 시스템에서 주로 본다.
- **응답 시간(response time)** — 클라이언트가 요청 보내고 응답받기까지의 시간. 온라인 시스템에서 중요.

> ⚠️ **지연 시간(latency) ≠ 응답 시간(response time)**
> - **응답 시간**: 클라이언트 관점에서 본 시간. 실제 처리 시간(서비스 시간) + 네트워크 지연 + 큐 지연 전부 포함.
> - **지연 시간**: 요청이 처리되기를 기다리는 시간. 서비스를 기다리며 휴지(latent) 상태인 시간.
> 자주 혼용하는데 같은 게 아님.

#### 응답 시간은 "분포"로 봐야 한다

- 같은 요청을 반복해도 매번 응답 시간이 다름 → 단일 숫자가 아니라 **측정값의 분포**로 봐야 함
- 대부분은 빠른데 가끔 엄청 오래 걸리는 **특잇값(outlier)** 이 있다
  - 원인: 백그라운드 컨텍스트 스위치, 네트워크 패킷 손실/TCP 재전송, GC 일시정지, 페이지 폴트, 디스크 진동 등

<img width="614" height="184" alt="image" src="https://github.com/user-attachments/assets/0535ce9a-a7af-4b5c-ba58-a8d2b211f1b1" />

>  100개 샘플 요청에 대한 응답 시간
> 회색 막대 그래프, 평균 / 중앙값 / 95·99분위 표시

**평균(arithmetic mean)은 좋은 지표가 아니다.** 얼마나 많은 사용자가 실제로 지연을 겪었는지 안 알려주기 때문.

대신 **백분위(percentile)** 를 쓴다:

- **중앙값(median, p50)** — 정렬했을 때 가운데 값. "요청 절반은 이 시간보다 빠르고, 절반은 느리다." 사용자가 보통 얼마나 기다리는지 보려면 이게 좋음. 단, 사용자는 한 페이지에서 여러 요청을 보내니까 적어도 하나가 중앙값보다 느릴 확률은 50%보다 훨씬 높다는 점 주의.
- **상위 백분위 — p95, p99, p999** — 값이 얼마나 나쁜지(꼬리)를 본다. p95가 1.5초면 "100개 중 95개는 1.5초 미만, 5개는 1.5초 이상".

#### 꼬리 지연 시간(tail latency)이 왜 중요한가

상위 백분위(꼬리)는 사용자 경험에 직접 영향을 준다.

- 아마존은 내부 서비스 응답 시간 요구를 **99.9분위(1,000건 중 1건)** 로 기술한다.
- 이유: 응답 가장 느린 고객은 보통 구매를 많이 해서 데이터가 제일 많은 사람 — 즉 **가장 소중한 고객**일 때가 많기 때문.
- 응답 시간 100ms 증가 → 판매량 1% 감소, 1초 느려지면 고객 만족 지표 16% 하락 같은 관찰도 있음.
- 단, p99.99(만 건 중 1건)까지 최적화하는 건 비용 대비 이득이 적다. 통제 불가능한 임의 이벤트에 쉽게 좌우되니까.

백분위는 **SLO(서비스 수준 목표)** / **SLA(서비스 수준 협약서)** 에 자주 쓰인다.
예) "응답 중앙값 200ms 미만 + p99 1초 미만일 때 정상, 가용 시간 99.9%"

#### 큐 대기와 선두 차단(head-of-line blocking)

- 큐 대기 지연이 높은 백분위의 상당 부분을 차지
- 서버는 병렬로 소수 작업만 처리 가능(예: CPU 코어 수 제한) → 느린 요청 몇 개가 뒤 요청을 막음 = **선두 차단(head-of-line blocking)**
- 뒷 요청은 빨리 처리됐어도 앞에서 기다린 시간 때문에 클라이언트엔 느려 보임
- → **클라이언트 쪽에서 응답 시간을 측정**하는 게 중요

> 💡 부하 테스트 흔한 함정: 클라가 응답 받은 다음에 다음 요청 보내게 짜면, 실제보다 줄이 짧아져서 결과가 뻥튀기됨. 응답이 느려도 계속 쏴야 진짜 부하가 나온다.

#### 꼬리 지연 증폭(tail latency amplification)

- 한 사용자 요청을 처리하려고 백엔드를 여러 번 호출 → 그중 **하나만 느려도** 전체 요청이 느려짐
- 병렬 호출이어도 가장 느린 호출이 끝나길 기다려야 함
- 호출 수가 많을수록 느린 호출이 끼어들 확률 ↑

>  단일 백엔드 호출이 전체 사용자 요청을 느리게 만드는 상황
> 웹 앱 → 백엔드 1~7 호출, 그중 487ms짜리 하나가 발목 잡는 그림

**백분위 계산 팁:**

- 대시보드에 응답 시간 백분위를 계속 띄우려면 효율적 계산 필요 (목록 전부 정렬은 비쌈)
- forward decay, t-digest, HdrHistogram 같은 근사 알고리즘 사용
- 주의: 백분위는 산술 평균하면 안 됨 → **히스토그램을 합치는 게** 올바른 방법

### 부하 대응 접근 방식

부하 매개변수가 커져도 성능을 유지하려면?

- **용량 확장(scaling up / 수직 확장)** — 더 강력한 장비로 이동. 간단하지만 고사양 장비는 매우 비쌈.
- **규모 확장(scaling out / 수평 확장)** — 여러 대의 낮은 사양 장비로 부하 분산. 이런 걸 **비공유(shared-nothing) 아키텍처**라 부른다.
- 현실은 둘의 조합. 적절한 사양 장비 몇 대가 다수의 저사양 장비보다 단순하고 저렴할 때가 많다.

- **탄력적(elastic) 시스템** — 부하 증가를 감지하면 자원을 자동으로 추가. 부하 예측이 어려울 때 유용.
- 반대는 **수동 확장** — 사람이 분석해서 장비를 추가. 더 단순하고 예상치 못한 일이 적음.

- **무상태(stateless) 서비스**는 여러 장비로 분산 배포하기 쉽다.
- **상태 유지(stateful) 데이터 시스템**을 분산하는 건 복잡도가 확 올라감. 그래서 "확장이 강제될 때까지 DB는 단일 노드로 유지"가 최근까지의 통념이었다.

**대규모 시스템 아키텍처는 그 앱에 특화돼 있다.** 모든 상황에 맞는 만능 확장 소스(magic scaling sauce)는 없음.

- 결정 요소: 읽기 양, 쓰기 양, 저장 데이터 양, 데이터 복잡도, 응답 시간 요구사항, 접근 패턴 등의 조합
- 예) 1kB짜리 초당 10만 건 vs 2GB짜리 분당 3건 → 데이터 처리량이 같아도 설계는 완전히 다름

> 💡 '확장성 = 수평확장'으로 외우면 틀린다. 먼저 '어떤 부하가 늘 거냐'를 정하고 거기 맞춰 설계하는 거고, 그 가정이 틀리면 확장한다고 한 게 다 헛심이 됨. 그래서 트래픽도 안 나오는 초기엔 확장 고민보다 기능부터 빨리 내는 게 맞다.

## 유지보수성 (Maintainability)

소프트웨어 비용의 대부분은 초기 개발이 아니라 **지속되는 유지보수**에 들어간다.

- 유지보수 = 버그 수정, 운영 유지, 장애 조사, 새 플랫폼 적용, 기능 추가, 기술 채무 상환 등
- 다들 레거시 유지보수를 싫어하지만, 처음부터 고통을 최소화하도록 설계할 수 있다(아니 그래야 함)

**핵심 설계 원칙 세 가지:**

- **운용성(operability)** — 운영팀이 원활하게 운영할 수 있게 만들어라
- **단순성(simplicity)** — 복잡도를 제거해 새 엔지니어가 이해하기 쉽게 (UI 단순성과는 다른 개념)
- **발전성(evolvability)** — 나중에 시스템을 쉽게 바꿀 수 있게 (유연성/수정 가능성/적응성)

### 운용성: 운영의 편리함 만들기

> "좋은 운영은 나쁜(불완전한) 소프트웨어의 한계를 우회하기도 하지만, 아무리 좋은 소프트웨어라도 나쁘게 운영하면 신뢰할 수 없다."

운영 일부는 자동화할 수 있고 또 해야 하지만, 자동화를 세팅하고 제대로 도는지 확인하는 건 여전히 사람 몫이다.

**좋은 운영팀이 책임지는 일:**

- 시스템 상태 모니터링, 나쁘면 빠르게 복원
- 장애/성능 저하 원인 추적
- 보안 패치 포함해 최신 상태 유지
- 시스템 간 영향 파악, 변경이 사고 치기 전에 차단
- 미래 문제 예측 및 선제 해결(예: 용량 계획)
- 배포/설정 관리 모범 사례와 도구 마련
- 복잡한 유지보수 태스크(플랫폼 마이그레이션 등) 수행
- 운영 절차 정의로 안정적 환경 유지
- 인사 이동에도 시스템에 대한 조직의 지식 보존

**데이터 시스템이 운용성을 높이려면:** 좋은 모니터링으로 런타임 동작 가시화 / 자동화·표준 도구 통합 지원 / 개별 장비 의존성 회피(특정 장비 내려도 전체 영향 없게) / 좋은 문서 / 자기 회복(self-healing) + 필요 시 수동 제어 / 예측 가능한 동작.

### 단순성: 복잡도 관리

- 작은 프로젝트는 단순하게 짜지만, 커지면 복잡하고 이해하기 어려워짐 → **커다란 진흙 덩어리(big ball of mud)**
- **복잡도 증상**: 상태 공간 급증, 모듈 간 강한 결합(tight coupling), 일관성 없는 명명/용어, 성능 해결용 해킹, 임시방편 특수 사례(special-casing) 등
- 복잡한 소프트웨어 → 변경 시 버그 위험 ↑, 예산·일정 초과 → 그래서 **단순성은 핵심 목표**

**단, 단순하게 = 기능 줄이기가 아니다.** 줄여야 할 건 **우발적 복잡도(accidental complexity)**.

- 우발적 복잡도 = 문제 자체(사용자에게 보이는 본질)에 내재한 게 아니라 **구현에서만 생기는** 복잡도
- 제거 도구 = **추상화(abstraction)**. 깔끔한 외관 아래 세부 구현을 숨기고 여러 앱에서 재사용 가능
  - 예) 고수준 언어 → 기계어/CPU 레지스터/시스템 호출을 추상화
  - 예) SQL → 디스크·메모리 자료구조, 동시 요청, 고장 후 불일치를 추상화
- 다만 분산 시스템에서 "복잡도를 관리 가능한 수준으로 유지하는" 좋은 추상화를 찾는 건 여전히 어려움

### 발전성: 변화를 쉽게 만들기

- 요구사항이 영원히 그대로일 가능성은 거의 없음 (새 사실 학습, 예기치 못한 사용 사례, 비즈니스 우선순위 변경, 플랫폼 교체, 법·규제 변경, 성장에 따른 아키텍처 변화 등)
- 조직 차원: **애자일(agile)** 패턴이 변화 대응 프레임워크 제공. TDD, 리팩토링도 여기 도움
- 데이터 시스템 수준: 변화를 쉽게 만드는 능력 = **간단함 + 추상화**와 밀접 (간단한 시스템이 수정하기 쉬움)
- 저자는 데이터 시스템 수준의 민첩성을 따로 **발전성(evolvability)** 이라 부름

> 💡 셋이 따로 노는 게 아니다. 단순하게 만들면 → 이해 쉬워지고 → 바꾸기 쉬워지고 → 운영도 편해짐. 하나만 고르라면 단순성. 복잡도 줄이는 게 나머지 둘을 공짜로 끌고 온다.

## 정리

요구사항은 두 종류로 나뉜다.

- **기능적 요구사항** — 무엇을 해야 하나 (저장/조회/검색/처리)
- **비기능적 요구사항** — 보안, 신뢰성, 법규 준수, 확장성, 호환성, 유지보수성 등

이번 장에서 자세히 본 세 가지(키워드만 다시):

- **신뢰성** — fault(HW 무작위·독립 / SW 체계적·상관 / 사람 실수)가 failure로 안 번지게. 내결함성으로 사용자에겐 결함을 숨김.
- **확장성** — 부하를 숫자로(부하 매개변수) → 성능을 분포로(중앙값·p95·p99, 꼬리 지연) → 부하 대응(수직/수평, 비공유, 탄력적). 트위터 팬아웃 = 부하 예시.
- **유지보수성** — 운용성(운영 편하게) + 단순성(우발적 복잡도 줄이기, 추상화) + 발전성(변경 쉽게).
