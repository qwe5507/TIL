# 1장. 시스템 설계의 기본
---

## 1.1 시스템 설계란

### 소프트웨어 시스템

특정 작업을 수행하려고 함께 동작하는 **컴포넌트 / 모듈 / 프로그램의 묶음**.
간단한 로컬 프로그램일 수도 있고, 네트워크에 걸쳐 있는 분산 시스템일 수도 있다.

언어 선택, TDD 할지 말지, 릴리즈 후 유지보수 전략까지 — 전부 설계의 일부다.

### 분산 소프트웨어 시스템

중앙 집중형과 달리 **여러 노드가 네트워크를 통해 하나의 목표를 위해 협력**하는 구조.

- 통신: RPC, 메시지 전달(message passing), 발행/구독(pub-sub)
- 예시: 클라우드 플랫폼(AWS/GCP), P2P, 분산 DB, CDN(유튜브·트위치)


```
사용자 ─→ [CDN: 정적 리소스]
       ─→ [DNS → 로드밸런서 → 앱 서버 1..N]
                                 ├→ 캐시
                                 ├→ 프록시 → DB 클러스터 (Db1 ~ Db6)
                                 └→ 블록 스토리지
```

분산이 붙는 순간 신경 쓸 게 확 늘어남 — 네트워크 통신, 일관성, 가용성, 장애 허용성, 보안 등등. 이게 2장 이후의 주제다.

### 시스템 설계의 의미

요구 사항(기능적 + 비기능적)을 받아서 **구조·구현·유지보수 방식을 설명하는 청사진(blueprint)** 으로 바꾸는 작업.

책에서 제시한 설계 절차:

1. **요구 사항 분석** — 데이터 조회/저장 패턴까지 깊게 들여다봄
2. **상위 수준 아키텍처 설계** — 컴포넌트·모듈·인터페이스 정의
3. **하위 수준 상세 설계** — 각 컴포넌트의 내부 동작, 알고리즘
4. **UI 설계** — 백엔드와 소통할 프론트 인터페이스
5. **API 설계** — 프론트↔백 간 규약
6. **DB 설계** — 파일 저장소부터 RDBMS(MySQL), NoSQL(HBase, Cassandra)까지

> 핵심 목표: "이해하기 쉽고 유지보수·확장이 용이하며, 성능·신뢰성·보안 요구를 충족하는 설계"
> 그리고 요구 사항이 바뀌어도 유연하게 대응할 수 있어야 한다.

## 1.2 시스템 설계의 두 가지 유형

### 1.2.1 상위 수준 (High-Level Design)

> "나무를 심기 전에 공원 부지를 먼저 정한다"는 비유로 시작함. 꽤 적절한 비유.

상위 수준에서 고려하는 것 4가지.

**① 시스템 아키텍처**

전체 구조와 컴포넌트 간 통신 방식 — 즉 아키텍처 패턴 선택.

- **모놀리식(monolithic)**: 모든 구성 요소를 하나로 묶은 애플리케이션
- **클라이언트-서버**: 클라이언트가 하나 이상의 서버에 요청
- **마이크로서비스**: 작고 독립적인 서비스들로 분리, 네트워크로 통신
- **이벤트 주도(event-driven)**: 비동기 이벤트/메시지 기반 상호작용

아키텍처를 고를 때 봐야 할 기준:
- 확장성: 사용자·데이터·기능이 늘었을 때 버틸 수 있는가
- 유지 보수성: 버그·디버깅·성능 개선이 얼마나 수월한가
- 신뢰성: 중단 없이 동작하는가
- 지연 시간: 응답에 어떤 영향을 주는가

> 포인트는 **"왜 그 아키텍처인지"** 를 근거와 함께 정리하는 것.

**② 데이터 흐름**

"데이터 흐름을 고려하지 않는 설계는 고추장 없는 비빔밥" — 책 표현이 재밌어서 그대로 옮김.

- **수집**: API 호출 / 실시간 스트리밍 / 배치
- **저장**: 접근 빈도, 검색 성능, 일관성 여부로 결정
- **처리**: 필요한 컴퓨팅 자원, 병목 현상 예측
- **검색**: 응답 시간, 캐시 전략, 부하 분산

**③ 확장성(Scalability)**

- **수직 확장(scale-up)**: 한 노드의 CPU/메모리/스토리지를 업그레이드
- **수평 확장(scale-out)**: 인스턴스를 추가해 부하를 분산

> 책의 식당 비유가 한 번에 와닿았다.
> - 수직 확장 = 주방을 더 좋은 장비로 교체 (오븐 업글, 냉장고 추가)
> - 수평 확장 = 주방을 아예 하나 더 만들어 병렬 운영
>
> 수직은 한계가 있고, 대규모 서비스에선 수평이 사실상 필수. 로드밸런싱·캐시·DB 샤딩·스테이트리스 서비스 설계가 다 수평 확장을 위한 도구다.

**④ 장애 허용 시스템(Fault-tolerant)**

일부가 죽어도 시스템이 계속 돌아가게 만드는 능력.
신경 써야 할 것: 복제(replication), 중복성(redundancy), 점진적 성능 저하, 모니터링, 자가 복구.

→ 상위 수준 설계는 **전체 아키텍처에 집중**하고 구현 세부 사항은 잘 안 들어감.

### 1.2.2 하위 수준 (Low-Level Design)

> "공원 부지 정했으면 이제 나무를 심을 차례."

하위 수준은 **구현 디테일**에 집중. 성능·메모리·유지보수성을 최적화하는 알고리즘, 데이터 구조, API를 고르는 작업.

**① 알고리즘**

- 시간 복잡도 / 공간 복잡도 / 둘 사이의 트레이드오프

> 인상 깊었던 문장: **"성능 좋은 기기를 비싸게 사기보다 알고리즘을 잘 최적화하는 게 훨씬 나은 선택"**
> 말은 쉬운데 현장에선 이게 안 돼서 장비 업글로 때우는 경우 허다함.

**② 데이터 구조**

고려할 포인트:
- 데이터 접근 방식 (빈도, 읽기/쓰기/업데이트)
- 쿼리 성능 (검색·삽입·삭제의 시간 복잡도)
- 메모리 사용량

자주 쓰는 것: 배열, 연결 리스트, 해시 테이블, 트리, 그래프.

**③ API**

설계할 때 고려할 4가지:
- **일관성**: 누가 봐도 이해·사용이 쉬워야 함
- **유연성**: 기존 기능에 영향 없이 확장 가능해야 함
- **보안**: 인증·인가·유효성 검사
- **성능**: 지연 최소화, 자원 효율

> **API 버저닝 파트는 따로 메모.**
>
> 백엔드 엔드포인트를 보면 `/api/v1/users` 처럼 v1을 붙이는 이유 = **하위 호환성 보장** 때문.
>
> 1. 초기 API를 v1으로 배포
> 2. 요구 변경 시 v1을 유지한 채 **v2를 새로 만듦** (예: `name` → `user: { name }` 로 구조 변경)
> 3. 기존 클라이언트는 v1 계속 사용
> 4. 내부 서비스에서 v1 호출부를 v2로 하나씩 이전
> 5. 전부 옮겨지면 v1 제거
>
> 버전 없이 바로 v1 구조를 바꿔버리면 기존 클라이언트가 전부 깨진다. 당연한 얘긴데 설계 단계에서 이걸 강제하는 습관이 중요.

**④ 코드 최적화**

- **리팩토링**: 기능 유지하면서 구조 개선 → 가독성·유지보수성 ↑
- **루프 언롤링(loop unrolling)**: 반복을 묶어 오버헤드 감소
- **메모이제이션(memoization)**: 계산 결과 캐싱
- **병렬 처리**: 작업을 쪼개서 동시 실행

루프 언롤링 예시
```javascript
// 일반 반복문
const sandwichOrders = [3, 5, 2, 6, 4];
let totalSandwiches = 0;
for (let i = 0; i < sandwichOrders.length; i += 1) {
  totalSandwiches += sandwichOrders[i];
}

// 루프 언롤링 (2개씩 묶어 처리)
let totalSandwichesUnrolled = 0;
for (let i = 0; i < sandwichOrders.length; i += 2) {
  totalSandwichesUnrolled += sandwichOrders[i] + (sandwichOrders[i + 1] || 0);
}
```

> 카페 비유: 샌드위치 하나씩 만들지 말고 빵 3개 깔아놓고 한 번에 3개 만드는 식.
> - **장점**: 반복 횟수 감소 → 성능 ↑
> - **단점**: 데이터가 불규칙하면 남는 재료(=쓸모없는 계산)가 생김, 코드 복잡도 ↑
>
> 결론: **데이터 크기·규칙성이 일정할 때** 효과적.

## 1.3 업계에서 시스템 설계가 갖는 중요성

설계를 제대로 하면 따라오는 이점:

- **요구 사항에 대한 명확한 이해**: 설계하는 과정 자체에서 요구가 정리됨
- **협업 시너지**: 팀원·이해관계자(영업·마케팅 등)가 같은 그림을 공유
- **설계 검토·피드백 용이**: 문서가 있으니 토론이 쉬워짐
- **높은 확장성**: 트래픽 증가에도 구조적으로 대응
- **성능 확보**: 병목을 사전에 차단
- **유지 보수성**: 수명 ↑, 관리 비용 ↓
- **비용 효율성**: 버그·재작업 감소

한 줄로: **좋은 설계 = 요구사항 충족 + 비즈니스 성과의 기반**.

## 1.4 산업별 실제 사례

일상의 서비스 거의 대부분이 탄탄한 설계 위에 돌아간다는 점을 강조.

| 산업 | 설계가 특히 중요한 이유 |
|------|------------------------|
| 금융 | 거래 내역·계좌 관리, 안정성/신뢰성/효율성 필수 |
| 전자상거래 | 대량 트랜잭션, 재고·배송·고객 정보 처리 |
| 의료 | 전자 의료 기록, 규제 준수, 높은 신뢰성 |
| 제조업 | 생산 공정 제어, 장비 모니터링, 재고 관리 |
| 운송업 | 물류·배송 추적·경로 최적화, 대용량 데이터 처리 |

## 1.5 요약

- **시스템 설계** = 아키텍처·컴포넌트·모듈·인터페이스·상호작용을 정의하는 일련의 과정
- 보통 **상위 수준 → 하위 수준** 순서로 진행
  - 상위: 아키텍처, 데이터 흐름, 확장성, 장애 허용성
  - 하위: 알고리즘, 데이터 구조, API, 코드 최적화
- 좋은 설계의 효과: 효율성 ↑, 비용 ↓, 오류·재작업 ↓
- 금융·이커머스·의료·제조·운송 등 거의 모든 산업에서 핵심 역량

다음 장(2장)에서는 **분산 시스템의 속성** — 일관성, 가용성, 파티션 허용성, 지연 시간, 내구성, 신뢰성, 장애 허용성을 다룰 예정.
