# 11장. X 서비스 설계

## 이 장 핵심

X는 시스템 설계 면접의 결정판. **읽기/쓰기 비율 비대칭**, **팬아웃(fan-out)** 문제, **타임라인 생성** 같은 굵직한 주제가 다 들어 있다. 9~10장이 단일 핵심 문제에 집중했다면, 11장은 4개 서비스가 메시지 큐로 연결된 **종합 분산 시스템 설계**.

```
[Client] →[LB]→[API GW]→ ┌─[Tweet Service]   → DB + Object Storage
                          ├─[User Service]    → User/Follow Tables
                          ├─[Timeline Service]← Kafka ← Tweet
                          └─[Search Service]  → Elasticsearch
```

## 11.4 시스템 규모 산정

### 가정

| 항목 | 값 |
|------|------|
| 전체 사용자 | 1억 명 |
| DAU | 2000만 명 |
| 사용자당 일별 트윗 | 5개 |
| 트윗 평균 크기 | 200B (텍스트만) |
| 미디어 포함 트윗 비율 | 20% (평균 1MB) |
| 트윗 보관 기간 | 5년 |

### 저장소

**트윗 저장소**
- 일별: 2000만 × 5 × 200B = **20GB/일**
- 연간: 20GB × 365 ≈ 7.3TB/년
- 5년: **36.5TB**

**미디어 저장소**
- 일별: 2000만 × 5 × 20% × 1MB = **200TB/일**
- 연간: 73PB/년
- 5년: **365PB**

**사용자 저장소** (1인당 1MB) — 1억 × 1MB = **100TB**

**합계** ≈ **365PB** (미디어가 압도적)

### 대역폭

**트윗 전송**: 100억 건/일 × 200B = **2TB/일**
(2000만 DAU × 5개 트윗 × 평균 팔로워 100명 = 100억 건)

**미디어 전송**: 20억 건/일 × 1MB = **2PB/일**

**총 대역폭**: 트윗 2TB + 미디어 2PB ≈ **2PB/일**

### 처리량

| 항목 | 계산 | 결과 |
|------|------|------|
| 트윗 쓰기 QPS | 2000만 × 5 / 86,400 | ≈ **1200 QPS** |
| 미디어 업로드 QPS | 1200 × 20% | ≈ **240 QPS** |
| **타임라인 팬아웃 QPS** | 1200 × 평균 팔로워 100 | ≈ **12만 QPS** |

> **팬아웃 12만 QPS가 진짜 핵심.** 한 사람이 트윗 하나 쓰면 평균 100명의 타임라인에 추가해야 하니까. 트윗 쓰기 자체보다 100배 무겁다.

### 캐시 크기

- 일별 트윗 조회: 2000만 × 100번 = **20억 건/일**
- 캐시 적중률 80% × 트윗 200B = **약 320GB** 캐시 필요

### 11.6.4 캐싱 전략

대규모 서비스의 성능은 캐시가 좌우. 책에 나오는 캐싱 전략 5가지.

| 전략 | 설명 |
|------|------|
| **시간 기반 슬라이딩 윈도우** | 최근 N시간(예: 24시간) 트윗만 캐시 |
| **인기도 기반** | 좋아요/리트윗 점수가 임계치 넘으면 캐시 |
| **하이브리드** | 최근 2시간 + 인기 트윗 |
| **예측 기반** | ML 모델로 인기 끌 트윗 미리 캐시 |
| **사용자 기반** | 팔로워 많은/공식 계정 트윗 캐시 |

### 캐시 제거 정책

| 정책 | 기준 |
|------|------|
| **LRU** | 가장 오래 안 쓴 것부터 제거 |
| **TTL** | 유효기간 만료 (인기 트윗은 더 길게) |
| **LFU** | 조회 빈도 낮은 것 제거 |
| **크기 기반** | 최대 크기 도달 시 LRU/LFU와 조합 |
| **우선순위 기반** | 트윗 영향력/작성자 영향력 기반 |

### 캐시 운영 패턴

- **다층 캐시** — Hot/Warm/Cold 계층으로 분리
- **캐시 워밍(cache warming)** — 재시작 시 곧 자주 조회될 데이터 미리 채움
- **캐시 버전 관리** — 트윗 수정/삭제 시 stale 데이터 정리
- **유형별 캐시 분리** — 트윗/프로필/타임라인 별도 운용

## 개인 메모

### 팬아웃 전략 (Fan-out) — 이 장의 진짜 핵심

책에서 다루는 방식은 **Push 모델 (Fan-out on Write)** 인데, 실무에선 두 가지가 더 있음:

| 방식 | 동작 | 장점 | 단점 |
|------|------|------|------|
| **Push (Fan-out on Write)** | 트윗 작성 시 팔로워 타임라인에 미리 분배 | 조회 빠름 | 쓰기 비싸, 팔로워 1000만인 셀럽은 재앙 |
| **Pull (Fan-out on Read)** | 조회 시 팔로잉 사용자들의 트윗을 합침 | 쓰기 단순 | 조회 비싸, 매번 N개 사용자 트윗 fetch |
| **Hybrid** | 일반 사용자는 Push, 셀럽은 Pull | 양쪽 최적화 | 구현 복잡 |

**셀럽 문제 (Celebrity Problem)** — 책엔 안 나오는데 면접 단골. 팔로워 1억인 계정이 트윗 하나 올리면 즉시 1억 명 타임라인 업데이트해야 함. 그래서 셀럽 트윗은 **Pull**로, 일반 사용자는 **Push**로 가는 **Hybrid** 가 실제 Twitter 구조.

### Snowflake ID

트윗 ID는 그냥 UUID로 만들면 정렬이 안 됨. 시간순 정렬 필요한 도메인 — Twitter가 만든 **Snowflake ID** 표준 사용:
- 64bit = `timestamp(41) + worker_id(10) + sequence(12)`
- ID만 봐도 시간 정렬 가능 + 분산 환경에서 충돌 없음

### Timeline 캐시 구조

Timeline Service의 캐시는 **사용자별로 트윗 ID 리스트만** 저장 (내용은 Tweet Service에서). 보통 Redis sorted set으로
```
ZADD timeline:user_123 1700000000 "tweet_id_1"
ZADD timeline:user_123 1700000005 "tweet_id_2"
ZREVRANGE timeline:user_123 0 49   # 최근 50개
```

### 메시지 큐 = 백본

마이크로서비스 4개를 결합도 낮게 묶어주는 게 Kafka. **트윗 한 번 작성** → 큐로 한 번 발행 → Timeline/Search 등 N개 서비스가 각자 처리. 새 서비스 추가 시 큐만 구독하면 됨.

### Elasticsearch의 한계

검색 결과가 약간 stale할 수 있음 (인덱싱 지연). Twitter도 검색이 1초~수초 지연되는 경우 종종 보임. **실시간성 vs 일관성 트레이드오프**. 일관성 강하게 가려면 동기 인덱싱 → 쓰기 지연 ↑.

### Object Storage 비용

미디어가 365PB까지 간다는 계산은 S3 비용으로 환산하면 어마어마함. 실제로는
- **Lifecycle Policy** — 오래된 미디어는 자동으로 Glacier로 이동 (저렴한 콜드 스토리지)
- **CDN (CloudFront 등)** — 인기 미디어는 에지에 캐싱해서 S3 비용/지연 둘 다 줄임

### 다이렉트 메시지 (DM)

책에선 살짝만 다루는데 DM은 실제로 별도 서비스로 분리하는 게 정석:
- E2E 암호화 가능성
- 다른 데이터 모델 (chat thread 개념)
- 다른 접근 제어 (private)

### "Read-heavy" 시스템

X는 **읽기 트래픽이 쓰기보다 100배 이상**. 그래서 모든 설계의 핵심이 "어떻게 읽기 비용을 줄일 것인가". 답은 결국 **캐시 + 미리 계산해두기 (precomputation)**. Timeline을 미리 만들어두는 Push 모델이 그래서 채택됨.

### Trending 분석

Apache Storm/Flink 외에 요즘은 **Kafka Streams** (7장에서 본 그것)도 많이 씀. JVM 앱 안에 임베디드라 운영 부담 적음.
