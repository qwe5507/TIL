
## 이 장 핵심

캐시는 자주 접근하는 데이터의 사본을 사용자와 더 가까운 위치에 임시 저장하는 기술. 분산 캐시는 이 캐시를 여러 노드에 분산시켜 대규모 환경에서도 같은 효과를 내도록 만든 것.

```
[클라이언트] → [캐시 노드 1] ─┐
              [캐시 노드 2] ─┼→ [백엔드 / DB]
              [캐시 노드 3] ─┘
                (분산 캐시)
```

캐시는 디스크 I/O를 줄이고, 백엔드 부하를 낮추고, 응답 시간을 단축한다. 하지만 일관성·동기화 같은 새로운 문제를 끌고 온다.

---

## 6.1 캐싱 정의

캐싱은 자주 접근하는 데이터나 리소스의 사본을 더 빠른 위치에 두고 관리하는 기술. 메모리(RAM), 디스크, 네트워크 등 다양한 위치에서 구현 가능.

### 핵심 용어

| 용어 | 의미 |
|------|------|
| **캐시(Cache)** | 데이터/리소스 사본을 저장하는 공간. 하드웨어 캐시(CPU 캐시), 소프트웨어 캐시(인메모리 캐시) 등 |
| **캐시된 데이터(Cached Data)** | 캐시에 저장된 사본. 보통 DB나 디스크처럼 느리지만 영구 저장되는 곳에서 가져옴 |
| **캐시 히트(Cache Hit)** | 요청한 데이터가 캐시에 있을 때. 원본까지 안 가도 됨 |
| **캐시 미스(Cache Miss)** | 요청한 데이터가 캐시에 없을 때. 원본에서 가져와 캐시에 저장 |
| **캐시 제거(Eviction)** | 저장 공간이 가득 차서 기존 데이터를 삭제해야 할 때 발생 |

### 6.1.1 분산 캐시 정의

여러 서버 노드에 자주 쓰는 데이터를 메모리에 분산 저장해두는 기술. 디스크 I/O를 거치지 않아 응답 속도가 빨라진다.

```
일반(로컬) 캐시: 단일 노드/앱 내부에 저장
              [App + Cache]
              
분산 캐시: 여러 노드에 걸쳐 캐시 공유
              [App] → [Cache Cluster: Node1, Node2, Node3]
```

### 6.1.2 분산 캐시 vs 일반 캐시

| 비교 항목 | 일반 캐시 (로컬) | 분산 캐시 |
|----------|------------------|----------|
| **적용 범위** | 단일 시스템/노드 내부 | 네트워크로 연결된 여러 노드 |
| **아키텍처** | 로컬 메모리에 위치, 앱과 같은 기기 | 캐시 노드 클러스터, 노드 간 데이터 공유 |
| **확장성** | 단일 서버 환경에 적합 | 수평 확장 가능, 대규모 환경에 적합 |
| **활용 사례** | 단일 앱에서 데이터 빠르게 조회 | 대용량 데이터, 여러 시스템 공유, 마이크로서비스 |
| **일관성/동기화** | 단순. 캐시 무효화도 쉬움 | 노드 간 동기화 복잡 |

요약하면 둘 다 빠른 데이터 접근이 목표지만, 분산 캐시는 **네트워크와 다중 노드라는 추가 변수**가 끼어든다.

### 6.1.3 활용 사례

- 웹 애플리케이션 — 사용자 세션, 페이지 구성 요소, DB 쿼리 결과
- DB 쿼리 결과 캐싱 — 반복 쿼리 비용 절감
- 콘텐츠 전송 네트워크(CDN) — 정적 콘텐츠(이미지/동영상/CSS) 에지 서버 활용
- 세션 관리 — 웹 앱 확장성 ↑
- API 응답 캐싱 — 백엔드 부하 감소
- 실시간 분석 — 자주 조회되는 집계 데이터
- 메시지 큐 — 중간 결과 저장으로 처리 효율 ↑

### 6.1.4 장점

- **성능 개선** — 메모리 접근으로 속도 ↑
- **확장 용이성** — 노드만 추가하면 부하 흡수
- **백엔드 부하 감소** — DB/파일 시스템에 가는 요청 ↓. DDoS 같은 상황에서도 원본 보호
- **장애 허용성** — 데이터 복제로 일부 노드 장애에도 서비스 유지
- **일관된 접근 속도** — 데이터셋 크기와 무관하게 일정한 응답 속도
- **비용 절감** — 백엔드 추가 증설 필요성 ↓

### 6.1.5 한계점 ⚠️

분산 캐시는 만능이 아니다. 다음 단점들이 따라온다.

- **일관성 유지의 어려움** — 모든 캐시 노드가 최신 데이터를 갖도록 동기화하는 게 진짜 어렵다
- **캐시 무효화의 복잡성** — 원본이 바뀌었을 때 모든 노드의 캐시를 갱신/제거하는 비용이 큼
- **설정/관리 복잡성** — 노드 간 조율, 데이터 분산 전략 등 운영 부담 ↑
- **네트워크 부하** — 노드 간 통신이 새 부담. 지리적으로 분산되면 더 커짐
- **캐시 데이터의 오래됨(stale)** — 동기화가 늦어지면 옛 데이터를 반환할 수 있음
- **데이터 분배의 어려움** — 잘못된 분배 전략은 일부 노드 과부하 / 일부 노드 유휴를 만든다
- **메모리 사용량** — 캐시는 결국 메모리. 비용·자원 효율에 영향
- **솔루션 자체 비용** — 인프라 + 운영 + 라이선스
- **데이터 접근 패턴** — 무작위/저빈도 접근에는 캐시 효과가 거의 없음. 오히려 비용만 ↑
- **쓰기 작업이 많은 환경에 약함** — 읽기 위주에 적합. 쓰기 잦으면 동기화 비용으로 이점이 사라짐

> 캐시는 도입 전에 **접근 패턴 분석**이 필수. 80/20처럼 일부 데이터에 트래픽이 몰리는 환경이 아니면 효과가 크지 않다.

## 6.2 분산 캐시 설계

### 6.2.1 요구 사항 정의

**기능적 요구사항** (캐시는 본질적으로 키-값 저장소)

- `put(key, value)` — 키-값 쌍을 캐시에 저장
- `get(key)` — 키로 값 조회

**비기능적 요구사항**

- **높은 성능** — 짧은 지연 시간, 높은 처리량
- **높은 확장성** — 데이터·요청·사용자 수 증가에 대응
- **높은 가용성** — 노드 장애에도 서비스 지속

### 6.2.2 설계 과정

#### 데이터 구조 — 해시 맵 + 캐시 제거 정책

put/get을 O(1)로 처리하려면 **해시 맵(해시 테이블)** 이 자연스러운 선택. 그런데 메모리는 한정돼 있으니 가득 찼을 때 뭘 버릴지를 정해야 한다 — **캐시 제거 정책(eviction policy)**.

#### 캐시 제거 정책 종류

**삽입 순서 기반 (접근 시간 무시)**

| 정책 | 설명 | 사용 예시 |
|------|------|----------|
| **FIFO** (First In, First Out) | 가장 먼저 들어온 항목부터 제거 | 예약 시스템 — 들어온 순서대로 처리 |
| **LIFO** (Last In, First Out) | 가장 최근에 들어온 항목부터 제거 | 소셜 미디어 추천 스토리 |

**접근 기반**

| 정책 | 설명 | 사용 예시 |
|------|------|----------|
| **MRU** (Most Recently Used) | 가장 최근 사용 항목 제거 | "방금 본 건 한동안 다시 안 볼 거야" — 문서 편집 SW |
| **LRU** (Least Recently Used) | 가장 오래 사용 안 된 항목 제거 | 웹 브라우저 캐시 (대표 정책) |
| **LFU** (Least Frequently Used) | 사용 빈도가 가장 낮은 항목 제거 | 음악 스트리밍 서비스 |

**시간/용량 한도 기반** — 정해진 한도 C를 넘거나 일정 시간 N이 지나면 제거 정책 발동.

#### LRU 캐시 설계

LRU가 가장 흔하게 쓰이니 이걸 직접 설계해본다. 요구사항은
- 최대 N개 저장
- put / 삭제 모두 **O(1)**

핵심 트릭은 **이중 연결 리스트 + 해시 맵 조합**.

<img width="434" height="411" alt="image" src="https://github.com/user-attachments/assets/b796b554-8310-47a7-ac63-2564f9f387c1" />

**조회(get)할 때 일어나는 일**

- **Case 1: 캐시 미스 + 여유 있음**
  1. DB에서 valueObject 가져옴
  2. 새 노드 생성, 리스트 맨 앞에 추가
  3. 해시 맵에 `key → 노드 포인터` 추가
  
- **Case 2: 캐시 미스 + 가득 참**
  1. 리스트 끝(가장 오래된 노드) 삭제, 해시 맵에서도 키 제거
  2. 그 다음은 Case 1과 동일

- **Case 3: 캐시 히트 (이미 있는 키)**
  1. 해시 맵에서 노드 포인터 찾아 리스트 노드로 이동
  2. 그 노드를 리스트 맨 앞으로 이동

모든 작업이 O(1)이 되는 이유 — 해시 맵은 평균 O(1) 조회, 이중 연결 리스트는 노드 포인터만 있으면 O(1)에 위치 변경 가능.

> **이게 LeetCode 146번 "LRU Cache" 문제다.** 시스템 설계 면접에서 거의 무조건 등장하는 자료구조 조합이라 손으로 한 번 짜보면 좋다.

#### 시스템 통합 — 캐시 배치 전략

설계한 LRU 캐시를 어디에 둘지가 다음 문제. 두 가지 접근 방식이 있다.

**방법 1. 앱 서버에 캐시 함께 배치 (인-프로세스)**

<img width="499" height="241" alt="image" src="https://github.com/user-attachments/assets/9516594b-f167-45fa-a8a7-53fcf485482b" />

- **장점**: 같은 머신 내 프로세스 간 호출 → 매우 빠름
- **한계**: 머신 다운되면 그 캐시도 같이 사라짐. 앱 서버 늘릴 때 캐시도 같이 늘어나는데 캐시 데이터는 분리되어 있어 효율 떨어짐.

**방법 2. 앱 서버와 캐시를 분리 (스탠드얼론 / Standalone)**

<img width="539" height="338" alt="image" src="https://github.com/user-attachments/assets/e3c94534-fabd-4f8e-9f28-61fc5eadfa59" />

- **장점**: 앱 서버와 독립적으로 확장. 캐시 클러스터 자체적으로 관리.
- **한계**: 키 기반 데이터 분배 필요 → **일관된 해싱**(3장에서 다룬 것)이 적절한 해법. 인프라 비용 증가.

#### 두 방식 비교

| 항목 | 서버 통합형 | 스탠드얼론 |
|------|------------|-----------|
| 성능 | 네트워크 오버헤드 없음, 직접 접근 | 네트워크 거쳐야 함, 약간 지연 |
| 확장성 | 앱과 함께 확장. 너무 커지면 한계 | 캐시만 독립적으로 세부 조정 가능 |
| 자원 공유 | 메모리/CPU 같이 씀 → 효율적이지만 충돌 가능 | 캐시 분리, 충돌 적음 |
| 유연성 | 통합되어 있어 기술 선택 제약 | 캐시 솔루션 자유 선택 |
| 의존성/독립성 | 앱과 묶여 한쪽 문제가 다른 쪽 영향 | 영향 적음 |
| 운영 복잡성 | 배포 단순 | 별도 운영 필요 |
| 인프라 비용 | 적음 | 추가 발생 |
| 네트워크 지연 | 거의 없음 | 약간 있음 |

요약하면 **소규모/지연 민감 = 통합형, 대규모/유연성 우선 = 스탠드얼론**. 실무에서는 거의 스탠드얼론 방식을 쓴다.

#### 비기능 요구사항 점검

설계가 끝나면 처음 정한 비기능 요구사항을 다시 본다.

- **높은 성능**: 모든 연산 O(1) ✓
- **높은 확장성**: 캐시 노드 수 늘리고 샤딩 ✓
- **높은 가용성**: 한 서버 다운되면 그 데이터를 잃을 위험 ⚠️ → **데이터 복제로 해결**

복제 패턴은 보통 **주 복사본(primary) + 보조 복사본(secondary) 2~3개**. 보조는 읽기 부하 분산에도 활용. 단, 복제는 일관성 문제를 다시 끌어옴 (2장 트레이드오프 참조).

## 6.3 대표적인 분산 캐시 솔루션

가장 많이 쓰이는 두 솔루션 — **Redis**와 **Memcached**.

### 6.3.1 Redis (레디스)

문자열, 리스트, 집합, 해시 등 다양한 데이터 구조를 지원하는 **인메모리 데이터 저장소**. 단순 캐시를 넘어서 메시지 브로커, DB로도 활용 가능.

**활용 사례**
- 웹 애플리케이션 캐싱
- 실시간 분석
- 세션 저장소
- 리더보드, 카운팅 시스템 (정렬 집합 활용)

**확장성** — 데이터를 여러 노드로 나누는 **샤딩** 지원.

**커뮤니티** — 활발한 오픈 소스 + 풍부한 문서/사용자 기반.

### 6.3.2 Memcached (맴캐시드)

**단순한 키-값 메모리 캐시**. 가볍고 직관적. 분산 아키텍처로 확장도 가능.

**활용 사례**
- 웹 애플리케이션 캐싱
- 세션 저장소
- DB 쿼리 결과 캐시
- 단순 키-값 분산 저장

**확장성** — 일관된 해싱으로 클러스터에 노드 추가하여 수평 확장.

**커뮤니티** — 안정적인 코드 기반, 다양한 언어와 통합 쉬움.

### 6.3.3 Redis vs Memcached — 어떻게 고르나

| 기준 | Redis | Memcached |
|------|-------|-----------|
| **사용 목적** | 다양한 데이터 구조, 고급 기능 | 단순 키-값 캐시, 빠르고 간단 |
| **데이터 보존** | 영속화 옵션 있음(RDB/AOF) | 메모리 기반, 영구 저장 X |
| **데이터 구조** | 다양함(list, set, sorted set, hash...) | 키-값만 |
| **사용 편의성** | 다양해서 학습 시간 ↑ | 직관적, 빠르게 통합 |

**선택 기준**
- 복잡한 자료구조나 영속화 필요 → Redis
- 단순한 캐시 + 운영 단순성 우선 → Memcached

> 실무에선 Redis가 압도적으로 많다. Memcached는 정말 캐시만 필요한 단순한 환경에서 종종 쓰임. 다만 Redis가 더 무겁고 메모리 효율은 Memcached가 약간 나은 편이라, 초고성능 키-값 캐시만 필요하면 Memcached도 여전히 선택지.
