# 12장. 인스타그램 서비스 설계

## 이 장 핵심

11장과 마이크로서비스 구조는 비슷한데, **사진이라는 무거운 미디어**를 어떻게 처리·저장·전달할지가 새로 나오는 문제다. 객체 저장소(S3) + CDN + 비동기 처리 파이프라인 조합이 답.

```
[Client] →[LB]→[API GW]→ ┌─[사진 업로드 서비스] → DB + Object Storage
                          │         ↓ (메시지 큐)
                          │   [백그라운드 작업: 압축/리사이즈/썸네일]
                          ├─[뉴스 피드 서비스]
                          ├─[사용자 서비스]
                          ├─[상호 작용 서비스]   (좋아요/댓글/해시태그)
                          └─[다이렉트 메시지 서비스]
                                   ↓
                              [CDN으로 전달]
```

---

## 12.1 기능적 요구사항

- **사용자 등록/인증** — 회원가입, 로그인, 세션 관리
- **사진 업로드/공유** — JPEG/PNG 등 다양한 형식, 압축, 필터, 캡션, 태그
- **뉴스 피드** — 팔로우한 사람의 사진 실시간 표시, 무한 스크롤
- **상호 작용** — 좋아요, 댓글, 멘션(@)
- **다이렉트 메시지** — 1:1/그룹 비공개 메시지, 실시간 알림
- **검색/탐색** — 사용자/사진/해시태그, 인기/트렌딩/추천
- **알림** — 팔로우/좋아요/댓글/DM 푸시 알림, 사용자 설정 가능

---

## 12.2 비기능적 요구사항

- **확장성** — 대규모 사용자와 사진 트래픽, 서비스별 독립 확장
- **성능** — 사진 업로드/피드/상호 작용 즉각 응답. 비동기 처리로 응답 시간 보호
- **가용성** — 다중화, 로드 밸런싱, 백업/복구
- **신뢰성** — 데이터 무결성, 일관성, 트랜잭션
- **사용성** — 검색/필터/추천 기능으로 콘텐츠 발견 쉽게

| 엔티티 | 역할 |
|--------|------|
| **User** | 프로필 (X와 동일) |

## 12.4 시스템 규모 산정

### 가정

| 항목 | 값 |
|------|------|
| 전체 사용자 | 1억 명 |
| DAU | 1000만 명 |
| 사용자당 일별 업로드 | 평균 2장 |
| 사진 한 장 평균 크기 | 5MB |
| 보관 기간 | 5년 |
| 사용자당 평균 팔로워 | 500명 |
| 사진당 평균 좋아요/댓글 | 100개 / 10개 |

### 저장소

**사진 저장소**
- 일별: 1000만 × 2장 × 5MB = **100TB/일**
- 연간: 100TB × 365 ≈ 36.5PB/년
- **5년: 182.5PB**

**사용자 저장소** (1인당 1MB) — 1억 × 1MB = **100GB**

**메타데이터** (좋아요/댓글/해시태그, 사진당 1KB) — 일별 20GB → 5년 36.5TB

**총합 ≈ 182.5PB** (사진이 압도적)

### 대역폭

- **업로드**: 100TB/일 (1000만 × 2장 × 5MB)
- **전송**: 1000만 × 2장 × 평균 팔로워 500 × 5MB = **50PB/일**
- **총 대역폭**: 약 **50PB/일**

### 처리량

| 항목 | 계산 | 결과 |
|------|------|------|
| 사진 업로드 QPS | 2000만 / 86,400 | ≈ **230 장/초** |
| 좋아요 QPS | 230 × 100 | ≈ **23,000/초** |
| 댓글 QPS | 230 × 10 | ≈ **2,300/초** |

> 사진 업로드는 230 QPS로 적당해 보이지만 **각 업로드가 5MB 미디어**라 실제 부하는 큼. 상호 작용(좋아요/댓글)은 가벼운 대신 횟수가 많음.

#### 뉴스 피드 생성 과정

1. **팔로우 관계 확인** — Follow 테이블에서 팔로우 중인 계정 목록 조회
2. **사진 수집** — 각 계정의 최근 사진을 DB에서 가져옴 (메타데이터 포함)
3. **정렬 및 순위 도입** — 시간순/참여도/소유자와 친밀도/전체 참여도 등을 조합한 랭킹 알고리즘
4. **페이지네이션** — 한 번에 모두 안 가져오고 토큰(이전 페이지 마지막 사진 ID/타임스탬프) 기반으로 분할
5. **캐시 활용** — 새 피드는 Redis에 저장, 사진 업로드/좋아요/댓글 발생 시 캐시 즉시 업데이트
6. **중복 사진 처리** — 사진 ID로 중복 체크해 피드 깔끔하게

#### 실시간 업데이트

- **WebSocket / Long Polling** — 클라이언트와 지속 연결, 새 사진/상호 작용을 실시간 푸시
- **알림 서비스 연동** — 팔로우/좋아요/댓글 발생 시 푸시 알림

#### 피드 동기화 (여러 기기에서)

- **타임스탬프 기반 동기화** — 마지막 본 사진의 타임스탬프를 기준으로 그 이후만 받아옴
- **점진적 업데이트** — 전체 피드 재로딩 대신 새 사진만 가져옴

## 12.7 기타 고려 사항

### 해시태그 기능

- 해시태그를 별도 엔티티로 관리 + 사진과 다대다 관계
- 해시태그 기반 사진 검색 API
- 사진 업로드 서비스에 캡션에서 해시태그 자동 추출 기능

### 탐색 및 추천 기능

- 사용자 취향/관심사/과거 활동 기반 개인화 추천
- 인기 사진/트렌드 큐레이션 컬렉션
- 키워드 검색 (사진/사용자/해시태그)

### 스팸 및 악용 방지

- 봇 활동 패턴 탐지 (대량 좋아요/댓글/팔로우)
- **ML 기반 이미지 분류**로 부적절 콘텐츠 자동 탐지/삭제
- 사용자 신고/차단 기능

### 다국어 지원 및 지역화

- 다양한 언어/지역 설정 지원
- 사용자 생성 콘텐츠는 원본 언어로 저장, 필요 시 번역 기능

### 수익화 전략

- 스폰서 콘텐츠, 프로모션 게시물, 광고
- 기업 사용자 프로필 + 성과 분석 도구
- 전자상거래 플랫폼 통합 — 상품 태그, 앱 내 구매

### 모니터링/지표/관측성

- **ELK 스택 / Splunk** — 중앙 집중형 로그
- **Prometheus + Grafana** — 메트릭 수집/시각화, KPI 추적
- **Jaeger / Zipkin** — 분산 트레이싱으로 마이크로서비스 간 요청 흐름 추적
- **PagerDuty / OpsGenie** — 임계값 기반 자동 알림

## 개인 메모

### 11장(X) vs 12장(Instagram) 차이점

| 항목 | X | Instagram |
|------|----|-----------|
| 중심 데이터 | 텍스트 트윗 (200B) | 사진 (5MB) |
| 저장소 핵심 | DB (트윗 데이터) | Object Storage (S3) + CDN |
| 핵심 부하 | 팬아웃 12만 QPS | 미디어 업로드 230 QPS + 전송 50PB/일 |
| 처리 패턴 | 실시간 분배 | 비동기 미디어 처리 |
| 해시태그 | 인덱싱 (Elasticsearch) | 별도 엔티티 (정규화) |
| 검색 엔진 | Elasticsearch 중심 | 추천/탐색이 더 중요 |

### 이미지 처리 파이프라인 상세

책에선 "압축/리사이즈/썸네일"만 짧게 언급하는데 실제론 훨씬 복잡:

1. **포맷 변환** — HEIC/PNG → JPEG, 또는 WebP/AVIF로 (대역폭 절감)
2. **다해상도 생성** — `300x300`(썸네일), `600x600`(피드), `1080x1080`(원본 표시), 최근엔 4K도
3. **EXIF 처리** — 메타데이터 추출(위치/카메라) + 민감 정보 제거
4. **콘텐츠 모더레이션** — ML로 NSFW/폭력 탐지
5. **얼굴 인식** — 태깅 자동화
6. **퍼셉추얼 해싱** — 중복/스팸 이미지 탐지 (pHash, dHash)

이걸 다 순차 처리하면 느리니까 **DAG (방향성 비순환 그래프) 형태로 병렬 처리**.

### CDN 캐싱 전략

CDN은 단순히 "에지 서버 캐싱" 이상의 디테일이 있음:

- **Pull vs Push CDN** — Instagram은 보통 Pull (요청 들어올 때 origin에서 가져와 캐싱). Push는 비싼 미디어 미리 분산
- **Cache TTL** — 인기 사진은 길게, 신규는 짧게
- **Signed URL** — 일정 시간만 유효한 URL로 hotlinking 방지, 권한 제어
- **이미지 변환 CDN** — CloudFront + Lambda@Edge로 URL 파라미터에 따라 즉석 리사이즈 (`?w=300`)

### 객체 저장소의 비용 최적화

182.5PB는 S3 표준에 두면 어마어마함. 실제로는

- **S3 Standard** — 최근 사진 (자주 조회)
- **S3 Standard-IA** (Infrequent Access) — 30일~90일 지난 사진
- **S3 Glacier** — 1년 이상 거의 안 보는 사진 (백업 용도)
- **Lifecycle Policy** — 자동 이동

### Feed Ranking 알고리즘

책엔 "랭킹 알고리즘"이라고만 나오는데 실무에선 ML 기반이 표준:

- **EdgeRank** (Facebook의 초기 방식) — Affinity × Weight × Time decay
- **Two-tower 모델** — 사용자 임베딩 × 콘텐츠 임베딩
- **Multi-task learning** — 좋아요 확률, 댓글 확률, 체류 시간 등 여러 목표 동시 최적화
- **Real-time signals** — 최근 본 콘텐츠, 최근 상호 작용 등 즉시 반영

### Stories 같은 단기 콘텐츠

책엔 없는데 Instagram의 진짜 핵심 기능 중 하나. 24시간 후 자동 삭제되는 사진/영상:

- **TTL 기반 저장** — DynamoDB의 TTL 속성으로 자동 삭제
- **별도 피드** — 메인 피드와 분리 (UI도 다름)
- **조회 추적** — 누가 봤는지 기록 → 별도 카운터 테이블

### Explore/Discover 피드

팔로우 안 한 사용자의 콘텐츠를 보여주는 탭. 추천 시스템의 정수:

- 사용자 행동 로그 → ML 모델 학습 → 후보 생성 → 랭킹 → 필터링
- 단계별로 다른 시스템 (보통 candidate generation은 빠른 ANN 검색, ranking은 무거운 모델)

### 면접에서 빠뜨리면 안 되는 포인트

- **동기/비동기 분리** — 업로드 즉시 응답, 처리는 백그라운드
- **CDN 필수** — 미디어는 무조건 CDN
- **다해상도 사전 생성** — 클라이언트마다 다른 크기 제공
- **Lifecycle Policy** — 오래된 미디어는 콜드 스토리지로
- **Feed ranking은 ML 기반** — 단순 시간순이 아니라
