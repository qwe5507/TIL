# 9장. URL 단축 서비스 설계

## 이 장 핵심

겉으로는 단순해 보여도 "어떻게 겹치지 않는 짧은 키를 효율적으로 생성하느냐"가 핵심 문제. 4가지 접근법을 비교한 끝에 **분산 카운터 + Base62 인코딩** 방식으로 수렴한다.

```
[Client] → [LB] → [Read Server]  → [Cache] → [Redis]
              ↓
         [Counter Server (n)] ← [Range Server]
              ↓
         [Redis (long → short 저장)]
```

## 9.1 실제 활용 사례

설계 전에 "왜 필요한가"를 짚고 넘어감.

| 활용처 | 이유 |
|--------|------|
| 소셜 미디어 공유 | 글자 수 제한, 클릭 추적 |
| 제휴 마케팅 | CTR/캠페인 성과 분석 |
| 이메일 마케팅 | 가독성 ↑, 참여도 추적 |
| QR 코드 | 짧은 URL을 QR로 만들면 스캔 빠름 |
| 인쇄/오프라인 | 사람이 외워서 입력 가능 |
| 내부 공유 링크 | 사내 협업, 활용도 추적 |
| 모바일 앱 | 동적 링크 (앱 안 깔려있으면 스토어로) |
| 브랜드 링크 | 자체 도메인으로 신뢰도 ↑ |

### 9.1.1 기능적 요구사항

**핵심**
- 긴 URL → 단축 URL 반환
- 단축 URL → 원본 URL 반환
- 단축 URL은 **고유**하고 **길이도 일정**해야 함

**확장 기능**
- 입력 URL 유효성 검사
- 사용자 정의 URL(맞춤형) 지원
- 단축 URL 만료 (마지막 사용 후 6개월)
- 오류 처리
- 긴 URL 수정 가능
- URL 매핑 삭제
- 활용도 분석/모니터링
- 사용자 계정 관리

### 9.1.2 비기능적 요구사항

- **가용성** — 항상 안정적으로 동작
- **확장성** — 사용자 1억 명 이상, 요청 폭주 대응
- **응답 속도** — 읽기/쓰기 지연 최소
- **일관성**
  - 같은 단축 URL은 누가 봐도 같은 원본 반환
  - 같은 긴 URL은 같은 단축 URL이 생성됨
  - 한 사용자가 같은 긴 URL을 다시 단축해도 같은 결과
- **내구성** — 만들어진 URL은 손실되지 않음
- **신뢰성** — 장애·폭주에도 정상 동작

## 9.2 API 설계

핵심 기능을 위한 두 개의 엔드포인트.

```
POST /shorturl
  요청: { longUrl: "..." }
  응답: { shortUrl: "..." }
  함수: getShortUrl(user_id, longUrl)

GET /shorturls/{shorturl}
  응답: { longUrl: "..." }
  함수: getLongUrl(shortUrl) → String
```

## 9.3 계산으로 문제 규모 파악

시스템 설계 면접에서 빠지지 않는 단계. **back-of-envelope 계산**으로 시스템 규모를 가늠한다.

### 사용할 문자 종류

- a-z, A-Z, 0-9 = **총 62개** (Base62)
- 추가로 `-`, `.`, `_`, `~`를 더하면 66개
- 이 책에서는 62개 사용

### 서비스 규모 추정

```
전체 사용자          ≈ 10억
URL 생성 사용자(10%) ≈ 1억
하루 평균 URL 생성   = 1억 건 (사용자당 1개)
10년 저장 시 총 URL  = 1억 × 365 × 10 ≈ 3650억 개
```

### 단축 URL 길이

```
6자리: 62^6 ≈ 500억 개   → 부족
7자리: 62^7 ≈ 3.5조 개    → 충분
```

→ **7자리** 선택.

### 저장 공간

각 URL 레코드 크기:
- 단축 URL: 20B
- 원본 URL: 1000B
- created_at: 10B
- updated_at: 10B
- created_by: 20B
- **합계 ≈ 1060B → 1500B로 반올림**

```
3650억 × 1500B ≈ 500TB
복제본 3개  ≈ 1.5PB
```

### RPS (Requests Per Second)

```
쓰기 RPS = 1억 / 86,400초 ≈ 1억 / 100,000 = 1000 RPS
읽기 RPS = 쓰기의 100배 ≈ 100,000 RPS
```

> 쓰기 1K + 읽기 100K RPS. **서버 한 대로는 불가능**, 수평 확장 필수.

## 9.4 시스템 설계

### 9.4.1 핵심 문제: 고유한 단축 URL을 어떻게 생성하나

여러 방식을 비교해본다.

#### 방법 1: 무작위 생성

긴 URL을 받으면 무작위로 단축 URL 생성 → DB에 이미 있는지 확인 → 중복이면 재생성.

**문제점**
- **DB 조회가 매번 필요** — 쓰기마다 최소 한 번의 읽기. 중복이면 재시도. 지연 시간 ↑
- **동시성 충돌** — 두 요청이 같은 시점에 같은 short_url을 생성하면 한쪽 데이터가 덮어쓰임

```
시점 t1: req A: long_url1 → short_url1 (DB 조회: 없음)
시점 t2: req B: long_url2 → short_url1 (DB 조회: 없음, 우연히 같음!)
시점 t3: req A: short_url1 저장 → {short_url1 → long_url1}
시점 t4: req B: short_url1 저장 → {short_url1 → long_url2}  ← 덮어쓰기, long_url1 소실
```

**해결책**
- `putIfAbsent` 같은 원자적 연산을 지원하는 DB 사용
- 또는 저장 후 `getLongUrl(short_url)`로 검증 — 추가 읽기 비용
- → 실제 운영용으로는 비효율적

#### 방법 2: MD5 해시

긴 URL → MD5 (128bit) → 앞 42bit 추출 → 7자리 Base62 인코딩.

```javascript
const crypto = require('crypto');

function generateShortUrl(longUrl) {
  const hash = crypto.createHash('md5').update(longUrl).digest('hex');
  return hash.substring(0, 7);  // 앞 일곱 글자만 추출
}
```

**장점** — 같은 긴 URL은 항상 같은 단축 URL → 중복 저장 방지.

**문제점** — 128bit 중 일부(42bit)만 쓰니까 **충돌 확률 급증**. 서로 다른 URL이 같은 short_url을 가질 수 있다

비트를 더 쓰면 충돌은 줄지만 **단축 URL이 길어짐**. 핵심 가치(짧은 URL)가 무너짐.

#### 방법 3: 카운터 방식

1, 2, 3, ... 순차 증가. 충돌 0%.

**문제점**
- **SPOF** — 카운터 서버 한 대에 의존하면 죽으면 끝
- 여러 서버로 분산하려고 `호스트 ID(6bit) + 시간 ID(32bit) + 카운터(4bit)` 같은 방식 시도
  - 호스트 6bit = 최대 64대만
  - 카운터 4bit = 1ms당 16개 초과 시 충돌

→ 대규모 환경에는 부족.

#### 방법 4: 분산 카운터 방식 ⭐ (채택)

전체 시퀀스 3.5조(62^7)를 **범위(range)** 로 쪼개 여러 카운터 서버에 할당.

```
3.5조 / 1M(100만) = 350만 개의 범위

R1   [0     ~ 1M)
R2   [1M    ~ 2M)
...
R10  [9M    ~ 10M)
```

**구조**
```
[Counter Server s1]  ← R1 [0~1M) 할당
[Counter Server s2]  ← R2 [1M~2M) 할당
...
       ↑
       │ 범위 요청/할당
       │
[Range Server] ← 고가용성, 어떤 카운터에 어떤 범위 줬는지 저장
```

각 카운터 서버는 자기 범위에서 +1씩 증가하며 고유 숫자 생성 → **충돌 없음**.

#### Base62 변환

생성된 숫자(10진수)를 7자리 Base62 문자열로 인코딩.

문자 집합: `abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789`

예시: 카운터 = `9234529445` → Base62 변환
```
9234529445 ÷ 62 = 148907051, 나머지 23 → x
148907051  ÷ 62 = 2401748,   나머지 39 → N
2401748    ÷ 62 = 38702,     나머지 4  → e
38702      ÷ 62 = 624,       나머지 14 → o
624        ÷ 62 = 10,        나머지 4  → e
10         ÷ 62 = 0,         나머지 10 → k
```

→ `keoeNx` → 최종 URL: `urlshortener.com/keoeNx`

#### 방법 4의 남은 이슈와 해결

| 이슈 | 해결 |
|------|------|
| 카운터가 범위를 다 쓰면? | Range Server에 새 범위 요청 |
| 카운터 다운되면? | 그 범위(최대 100만 개)는 손실. 전체 350만 범위 중 1개라 큰 손해 X |
| 범위 손실 막으려면? | 카운터가 주기적으로 현재 값을 Range Server에 백업. 장애 시 백업+N에서 재개 |
| Range Server가 SPOF? | 읽기/쓰기 빈도가 낮은 서비스라 ZooKeeper 같은 고가용성 시스템으로 처리 |

### 9.4.2 데이터베이스 선택

저장할 데이터는 `short_url → long_url` 매핑 = **키-값 구조**.

→ **Redis** 선택. 빠르고, 확장성·내구성·장애 허용성 갖춤.

### 쓰기 처리 흐름

1. 클라이언트가 긴 URL 제출 → POST API
2. LB가 카운터 서버 중 하나로 라우팅
3. 카운터 서버가 범위에서 다음 숫자 생성 (필요 시 Range Server에서 새 범위 받음)
4. 7자리 Base62 변환 → 단축 URL
5. Redis에 `long_url → short_url` 저장

### 읽기 처리 흐름

1. 클라이언트가 단축 URL 제출 → GET API
2. LB가 Read Server로 라우팅
3. Read Server가 **캐시 먼저** 확인
   - 있으면 즉시 반환
   - 없으면 Redis 조회 → 캐시에 저장 → 반환

## 9.5 요구사항 검토

설계한 시스템이 처음 정의한 요구사항을 충족하는지 점검.

### 기능적 요구사항

| 요구사항 | 충족? |
|----------|-------|
| 긴 URL → 단축 URL | ✓ |
| 단축 URL → 원본 URL | ✓ |
| 단축 URL 고유 + 7자리 일정 | ✓ (분산 카운터 + Base62) |

### 비기능적 요구사항

| 항목 | 어떻게 만족했나 |
|------|----------------|
| **가용성** | 수평 확장, SPOF 없음 |
| **확장성** | 카운터/Read 서버 수평 추가만 하면 됨 |
| **지연 시간** | 쓰기: 별도 계산/조회 불필요. 카운터 수백 개 미리 메모리에 / 읽기: Redis 인메모리 + LRU 캐시 |
| **일관성** | 단축 URL당 DB에 단일 항목만 저장 |
| **내구성** | Redis 내구성 옵션 (AOF/RDB) |
| **신뢰성** | 장애/폭주에도 정상 작동 |

## 9.6 요약

- URL 단축 서비스의 핵심 = **충돌 없는 고유 키 생성**
- 무작위/MD5/단일 카운터 모두 한계가 명확
- **분산 카운터 + Base62 인코딩**이 정답에 가까움
- Redis 저장 + 카운터 서버 + Range Server + Read Server + Cache로 구성
