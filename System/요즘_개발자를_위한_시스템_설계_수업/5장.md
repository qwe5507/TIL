- 관계형 vs NoSQL 비교, 어떤 상황에 뭘 쓰는지
- **키-값 저장소를 처음부터 설계**해보는 과정 (확장성·복제·일관성·장애 허용성을 단계적으로 추가)
- 4가지 NoSQL 계열의 대표 솔루션 — DynamoDB(키-값), Cassandra/HBase(컬럼 패밀리), Neo4j(그래프)
- 시스템 설계 면접에서 키-값 저장소 질문이 나왔을 때의 접근법

이 장은 시스템 설계 면접에서 가장 자주 등장하는 주제 중 하나라 꼼꼼히 봐둘 가치가 있다.

## 5.1 데이터베이스

### 데이터베이스가 필요한 이유

단순 파일 저장과 비교했을 때 DB가 주는 것
- **데이터 구성** — 테이블/행/열로 체계적 분류
- **빠른 검색** — 인덱싱, 복잡한 쿼리
- **무결성** — 제약 조건, 검증 규칙
- **보안** — 인증, 권한, 암호화
- **일관성** — ACID 트랜잭션
- **확장성** — 수직/수평 확장
- **다중화/백업/복구**
- **데이터 관계** — 외래 키로 엔티티 간 연결
- **데이터 이력** — 시간에 따른 변화 추적
- **데이터 분석/공유**

> 모든 시스템 요구를 한 종류의 DB가 다 만족시킬 수는 없다. 도메인에 맞는 DB를 고르는 게 시스템 설계의 첫걸음.

### 5.1.1 DB 유형 — 크게 두 부류

```
데이터베이스
├── 관계형 (RDBMS) — 테이블/행/열, SQL, ACID
└── 비관계형 (NoSQL)
    ├── 키-값 저장소 — Redis, DynamoDB, Riak
    ├── 문서 지향 — MongoDB, CouchDB
    ├── 컬럼 패밀리 — Cassandra, HBase, ScyllaDB
    └── 그래프 기반 — Neo4j, Neptune, OrientDB
```

### 5.1.2 관계형 DB

1970년 에드거 코드(Edgar F. Codd)가 만든 모델. 데이터를 행과 열로 구성된 표 형식으로 관리.

- **테이블** — 고객, 제품, 주문 같은 개체나 개념 표현
- **행(records, tuples)** — 각 데이터 항목
- **열(columns, attributes)** — 데이터 유형
- **키** — 기본 키(primary key)로 행 고유 식별, 외래 키(foreign key)로 테이블 간 관계
- **정규화** — 중복 제거, 무결성 향상
- **SQL** — 표준 쿼리/조작 언어
- **ACID** — 원자성, 일관성, 독립성, 지속성
- **트랜잭션** — 여러 SQL 작업을 하나의 묶음으로

대표 RDBMS: MySQL, PostgreSQL, Oracle, SQL Server, SQLite, IBM Db2.

> 금융, 재고 관리, CRM 등 데이터 일관성·구조·신뢰성이 중요한 도메인에 주로 사용.

### 5.1.3 비관계형 DB (NoSQL)

NoSQL = "non SQL" 또는 "non relational". 테이블/행/열을 안 쓰고 더 유연한 구조로 데이터를 저장.

- **키-값 저장소** — 단순 검색에 효율적, 복잡한 쿼리는 불리
- **문서 지향** — JSON/XML 형식, 웹 앱에 많이 사용
- **컬럼 패밀리** — 관련 데이터를 컬럼 그룹으로, 수평 확장 유리
- **그래프 기반** — 노드와 엣지로 관계 표현

장점: 비정형 데이터 처리, 확장성, 데이터 모델 변경 유연.
단점: 일관성 약화, 신중한 데이터 모델링과 인덱싱 설계 필요.

### 5.1.4 관계형 vs NoSQL 비교 정리

| 항목 | 관계형 | NoSQL |
|------|--------|-------|
| **데이터 무결성** | 정규화로 중복 최소화 | 유연한 모델 |
| **보안** | 내장 보안 기능 강력 | DB마다 차이 |
| **트랜잭션** | ACID 기본 지원 | 대부분 미지원 |
| **성능** | 복잡한 조인 시 저하 | 대규모 작업에서 우수 |
| **메모리** | 행/열 고정, null도 공간 차지 | 유연 |
| **복잡성** | 조인, 관계 관리 부담 | 모델링 단순 |
| **수평 확장** | 어려움 | 강점 |

## 5.2 키-값 저장소

각 고유한 키가 특정 값과 연결되는 단순한 구조. **분산 해시 테이블(DHT)** 을 떠올리면 이해가 쉽다. put/get/delete 같은 단순 연산이 기본.

대규모 분산 시스템에서 자주 쓰는 이유
- **확장성** — 단순 구조라 노드 분산 쉬움
- **성능** — 같은 노드 안에서 처리되면 매우 빠름
- **유연성** — 고정 스키마 불필요
- **장애 허용성** — 복제·분할 설계 가능

### 5.2.3 키-값 저장소 설계 — 요구사항 정의

**기능적 요구사항**
- `Put(key, value)` — 저장 또는 업데이트
- `Get(key)` — 조회 (없으면 적절한 오류)
- `Delete(key)` — 삭제

**비기능적 요구사항**
- **확장성** — 노드 추가로 수평 확장
- **성능** — put/get 모두 짧은 지연
- **내구성** — 노드 장애에도 데이터 손실 없음
- **일관성** — 최신 쓰기가 모든 읽기에 반영
- **가용성** — 일부 노드 장애에도 작동
- **파티션 허용성** — 네트워크 장애에도 작동

이 비기능 요구사항 6가지가 다음 절들의 설계 과제를 만든다.

## 5.3 확장성과 데이터 복제 최적화

### 5.3.1 확장성 — 단순 모듈로 해싱의 문제

가장 단순한 방법은 노드 개수 m으로 나눈 나머지로 노드를 정하는 것.

```
node_id = hash(key) % m
```

문제는 **노드를 추가하거나 제거할 때 거의 모든 키가 다른 노드로 이동**해야 한다는 것. 비용이 너무 크고 지연 시간이 길어진다.

### 5.3.2 일관된 해싱 사용

3장에서 다룬 일관된 해싱을 활용. 노드와 키를 같은 해시 공간(원형 링)에 매핑하고, 키는 시계 방향으로 가장 가까운 노드가 담당.

```
       N4
    ╱      ╲
   N3       N1
   ╲       ╱
       N2
       
key → hash(key) → 링 위 좌표 → 시계방향 가장 가까운 노드가 담당
```

장점: 노드 추가/삭제 시 일부 키만 이동하면 됨. 부하도 평균적으로 고르게 분산.

문제: 부하가 항상 균등하진 않다. 특정 노드에 트래픽이 몰리는 **핫스팟(hotspot)** 발생 가능.

### 5.3.3 가상 노드 (Virtual Nodes)

핫스팟 해결을 위해 한 노드를 링 위 여러 위치에 배치. 같은 키에 여러 해시 함수(`h1, h2, h3`)를 적용.

```
서버마다 가상 지점 여러 개:
  N1 → s1, s11, s12, s13
  N2 → s2, s21, s22, s23
  ...
```

장점:
- 부하가 더 고르게 분산됨
- 노드 장애 시 그 노드의 부하가 한 노드에 몰리지 않고 여러 노드에 흩어짐
- 물리 장비 성능에 따라 가상 노드 수를 조정 가능 (성능 좋은 노드는 더 많이 배치)

### 5.3.4 데이터 복제 전략

가용성을 위해 데이터를 여러 노드에 복제. 두 가지 모델.

**주-종 모델 (Primary-Secondary)**

```
[주(쓰기)] → [종(쓰기 복제)]
           → [종(읽기 전용)]
```

주 저장소가 쓰기 처리, 종은 복제 + 읽기. 주 노드 장애 시 SPOF 위험.

**동등 모델 (Peer-to-Peer)**

```
[주] ⇄ [주]
  ⇅      ⇅
[주] ⇄ [주]
```

모든 노드가 주 역할. 모두에 복제하면 비용 크니 보통 3~5개만 복제. SPOF 없음, 가용성·내구성 ↑.

> 책 저자는 동등 모델을 추천한다. SPOF 회피와 가용성 측면에서 유리. 단, 일관성 처리 비용은 더 큼.

각 키의 복제본 위치는 **우선 목록(preference list)**으로 관리. 키 K를 받은 코디네이터 노드는 자기 + 후속 n-1개 노드를 우선 목록으로 삼는다.

## 5.4 get/put 구현

### 코디네이터(Coordinator) 노드

put/get 요청을 처리하는 노드. 보통 우선 목록의 첫 번째 노드.

클라이언트 → 코디네이터 도달 방식:
1. **로드 밸런서 경유** — 클라이언트 코드가 단순
2. **분할 인식 클라이언트 라이브러리** — 클라이언트가 직접 코디네이터로 라우팅, 홉 수 ↓ 지연 ↓

### 5.4.2 r과 w 사용 (Quorum)

2장에서 다룬 그것. 성공 기준이 되는 노드 수를 명시적으로 설정.

- `r` = 읽기 성공에 필요한 응답 노드 수
- `w` = 쓰기 성공에 필요한 응답 노드 수
- `n` = 복제본 수

| n | r | w | 결과 |
|---|---|---|------|
| 3 | 2 | 1 | r+w>n 위반 — 일관성 보장 안 됨 |
| 3 | 2 | 2 | 강한 일관성, 읽기/쓰기 균형 |
| 3 | 3 | 1 | 빠른 쓰기, 느린 읽기 |
| 3 | 1 | 3 | 빠른 읽기, 느린 쓰기 |

**`r + w > n`이어야 강한 일관성** — 읽기 셋과 쓰기 셋이 반드시 한 노드 이상에서 겹치기 때문.

읽기 시 get 지연은 **r개의 레플리카 중 가장 느린 것에 좌우**된다. r 값이 클수록 일관성보다 가용성을 우선시한 셈.

## 5.5 키-값 저장소의 장애 허용성

### 5.5.1 일시적 장애 — 힌트 전달 방식 (Hinted Handoff)

원래 가야 할 노드 A가 일시적으로 다운된 상황. 엄격한 쿼럼이면 작업이 막혀버림.

**느슨한 쿼럼(sloppy quorum)** + **힌트 전달**로 해결.

```
1. 클라이언트가 노드 A에 쓰기 요청
2. A가 일시적으로 다운 → 우선 목록의 다음 노드 D가 대신 처리
3. D는 데이터를 임시 저장하면서 "원래는 A 거임"이라는 힌트(hint)도 같이 기록
4. 노드 A 복구
5. D는 임시 저장한 데이터를 A에 전달 → A가 업데이트
6. D는 자기 임시 저장소에서 그 데이터 삭제
```

이렇게 하면 일시 장애가 있어도 가용성 유지.

### 5.5.2 영구적 장애 — Merkle Tree로 동기화

노드가 영구적으로 장애를 겪고 새 노드로 교체된 경우, 레플리카 간 데이터 동기화가 필요. 무식하게 전체 데이터를 다 비교하면 비싸니 **Merkle Tree**(머클 트리)를 쓴다.

```
        [Root Hash]
        ╱         ╲
   [Hash AB]    [Hash CD]
   ╱      ╲     ╱      ╲
  HA      HB  HC      HD   ← 키-값 쌍 해시 (리프)
```

동작 방식
1. 모든 키를 해시해서 리프 노드로
2. 상위 노드는 자식 노드 해시들의 해시
3. 두 트리의 루트만 비교 → 같으면 동기화 불필요
4. 다르면 차이 나는 가지만 따라 내려가서 불일치 키만 동기화

이런 **anti-entropy** 메커니즘 덕에 동기화 트래픽이 크게 줄어든다.

> Cassandra, DynamoDB가 실제로 Merkle Tree를 써서 노드 간 데이터 일관성을 맞춘다.

면접에서 키-값 저장소 설계가 나왔을 때 챙겨야 할 포인트:

1. **확장성과 성능** — 데이터 분배 전략(샤딩, 일관된 해싱) 명확히
2. **레플리카와 일관성** — 복제 방식, 일관성 모델, 충돌 해결

4. **시스템 발전 가능성** — 버전 관리, 변경 대응

## 5.7 DynamoDB

AWS의 완전 관리형, 서버리스 NoSQL 키-값+문서 DB. SSD 기반, 데이터 센터 3곳에 분산 복제.

### 5.7.1 고정된 스키마가 없다

RDBMS는 인덱스 생성 전에 스키마 정의 필수. NoSQL은 데이터 읽을 때 스키마를 정의 가능 → 유연.

비관계형 DB의 강점:
- **유연성** — 비정형/반정형 데이터 통합 저장
- **확장성** — 문서 형식 → 클러스터 분산 쉬움
- **성능** — 대규모 워크로드에서 안정적
- **가용성** — 노드 교체 시 영향 최소화

### 5.7.3 기본 키 유형

DynamoDB는 두 가지 키 체계 사용.

| 유형 | 설명 |
|------|------|
| **분할 키 (Partition Key)** | 단일 속성으로 항목 고유 식별 |
| **복합 키 (Composite Key)** | 분할 키 + 정렬 키. 같은 분할 키 안에서 정렬 키로 항목 구분 |

```
복합 키 예시:
  분할 키 | 정렬 키    | 속성
  --------|-----------|--------
  1       | Book Id   | title, author
  1       | Movie Id  | ...
  2       | Book Id   | ...
```

분할 키를 해시 → 어느 노드에 저장할지 결정. 정렬 키 → 같은 노드 안에서 항목 구분.

**보조 인덱스**도 지원해서 기본 키 외 다른 키로도 쿼리 가능.

### 5.7.4 처리율 최적화

DynamoDB는 **RCU(Read Capacity Unit)**, **WCU(Write Capacity Unit)** 라는 단위로 처리 용량 관리.

예: 테이블이 20000 RCUs / 5000 WCUs로 설정 → 초당 20000 읽기 / 5000 쓰기 가능 (항목 크기 일정 가정).

**버스팅(Bursting)** — 한 파티션의 트래픽이 일시적으로 폭증할 때, 다른 파티션의 여유 용량을 잠시 빌려쓰는 기능. 단기 폭증을 흡수.

**적응형 용량** — 핫 파티션을 자동 감지해서 추가 용량을 할당.

### 5.7.5 높은 가용성 — WAL + 복제 그룹

쓰기 흐름:
```
1. 클라이언트 → 리더 레플리카
2. 리더가 WAL(Write-Ahead Log)에 기록 + 메모리 색인
3. 리더가 다른 레플리카로 로그+색인 전파
```

복제 그룹 내에서 리더 레플리카는 멀티-Paxos로 선출. 노드 3분의 2 이상 정상이면 쿼럼 형성 → 작동.

DynamoDB는 기본적으로 **최종 일관성**, 옵션으로 강한 일관성도 선택 가능.

---

## 5.8 컬럼 패밀리 데이터베이스

데이터를 **컬럼 패밀리**라는 그룹으로 구성. 수평 확장에 유리.

특징
- 대량의 쓰기/읽기에 적합
- 복잡한 쿼리(조인 등)는 부적합 — **키 기반 조회에 최적화**
- 일관성 수준 조정 가능 — 가용성 vs 일관성 트레이드오프 설정

대표 솔루션
- **Apache Cassandra** — 가용성과 확장성에 중점 (AP)
- **HBase** — 일관성에 중점 (CP)
- **ScyllaDB** — Cassandra 호환 + 성능 개선

> 시계열 데이터, 이벤트 로그, 모니터링 시스템처럼 쓰기가 많고 분석성 쿼리가 단순한 도메인에 적합.

---

## 5.9 HBase

아파치의 오픈소스 분산 컬럼 패밀리 DB. 구글 BigTable에서 영감, **HDFS**(Hadoop 분산 파일 시스템) 기반.

### 데이터 구조

```
테이블
└── 행 (rows)
    └── 컬럼 패밀리 (column families)
        └── 컬럼 (columns)
            └── 키-값 쌍 (셀)
```

행 키로 데이터 정렬. 같은 컬럼 패밀리는 같은 파일에 저장 → 분석 쿼리에 유리.

### 읽기 흐름

```
[Region Server]
  ├── Block Cache (블록 캐시) — 최근 사용한 키-값
  ├── MemStore (메모리 저장소) — 최근 쓰기
  └── HFile (HDFS 데이터 노드) — 디스크 저장소
        ↑
       WAL (Write-Ahead Log)
```

조회 순서:
1. **블록 캐시** — 최근 사용된 키-값 확인
2. **MemStore** — 최근 쓰기 확인
3. 없으면 **블룸 필터** + 인덱스로 어느 HFile에 있을지 추정
4. HFile을 메모리로 로드, 스캐너가 검색

> 블룸 필터 + 인덱스 조합이 디스크 I/O를 크게 줄인다 (3장에서 본 그 블룸 필터).

쓰기는 WAL → MemStore → 주기적 Flush → HFile 순.

---

## 5.10 그래프 기반 데이터베이스

복잡한 관계 데이터를 표현/관리하는 데 최적화. 엔티티 간 연결을 효율적으로 탐색 가능.

### 핵심 개념

```
[노드: Alice]──knows──→[노드: Bob]
        │
       likes
        ↓
   [노드: Pizza]
```

| 개념 | 설명 |
|------|------|
| **노드(Node)** | 사람/상품/장소 같은 개체. 속성(키-값) 포함 |
| **엣지(Edge)** | 노드 간 연결, 두 노드 간 관계 |
| **레이블** | 노드/엣지를 카테고리로 분류 (예: `Person`, `friends`) |
| **Cypher** | 그래프 데이터를 조회/조작하는 쿼리 언어 (Neo4j) |

대표 솔루션: **Neo4j**, AWS Neptune, OrientDB.

> 친구 추천, 사기 탐지, 지식 그래프, 추천 시스템처럼 **관계 자체를 탐색해야 하는** 도메인에 강력함.

---

## 5.11 Neo4j

가장 많이 쓰이는 그래프 DB. 데이터를 노드와 엣지로 직접 저장하는 **네이티브 그래프 저장소**.

### 내부 저장 방식

**노드 저장소** — 각 노드 레코드 15바이트
- 사용 중 표시 (1B)
- 노드 ID (4B)
- 첫 번째 관계 ID (1B)
- 첫 번째 속성 ID (1B)
- 레이블 저장소 (5B)
- 향후 사용 (1B)

**관계 저장소** — 각 관계 레코드 34바이트
- 출발/도착 노드 ID
- 관계 유형 포인터
- 출발/도착 노드 각각의 이전/다음 관계 레코드 포인터 (이중 연결 리스트처럼)

이중 연결 리스트 구조라 한 노드에서 출발하는 관계들을 빠르게 순회 가능 — 그래프 탐색이 빠른 이유.

---

## 5.12 관계형 모델링 vs 그래프 모델링

같은 관계 데이터를 두 방식으로 표현했을 때 차이.

### 관계형 모델링

관계 테이블에 출발-도착을 행으로 저장.

```
follows 테이블:
  src | dst | src_prev_rid | src_next_rid | dst_prev_rid | dst_next_rid
  1   | 2   | NULL         | C            | NULL         | NULL
  1   | 3   | A            | NULL         | NULL         | NULL
```

새 관계 D 추가 시 — 인덱스 갱신, 기존 행의 prev/next 포인터 수정 필요. 관계가 많아질수록 비용 ↑.

### 그래프 모델링

노드 1과 노드 2의 관계는 그냥 엣지로 추가만 하면 됨. 노드 자체에 인접 관계 리스트가 직접 연결돼 있어서 N+M(N=노드, M=관계) 정도로 빠르게 탐색.

> "친구의 친구의 친구" 같은 다단계 관계 탐색은 관계형 DB에서 self-join이 늘어날수록 쿼리가 폭발적으로 느려진다. 그래프 DB는 같은 작업이 노드 포인터 따라가기로 끝나서 훨씬 빠름.
