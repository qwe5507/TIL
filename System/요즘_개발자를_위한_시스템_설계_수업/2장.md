# 2장. 분산 시스템의 속성

분산 시스템은 노드가 여러 개라 좋은 점도 많지만 그만큼 신경 쓸 게 많다.

**8가지 속성은 서로 트레이드오프 관계**다. 전부 충족하는 것은 불가능하고, 요구사항에 맞춰 균형을 잡아야 한다.

---

## 2.1 호텔 객실 예약 시스템 예시

```
사용자1 ──→ [App Server] ──쓰기 요청──→ [db1] ⇄ [db2] ⇄ [db3]
                            (복제 클러스터)
사용자2 ──→ [App Server] ──읽기 요청──→ [db?]
```

- 사용자1(`u1`)이 방(`r1`)을 예약 → `bookRoom(u1, r1)`
- 사용자2(`u2`)는 같은 시각 그 방의 가용 여부 확인 → `isRoomAvailable(r1)`
- DB는 `db1`, `db2`, `db3` 세 개로 복제됨

쓰기를 다른 레플리카에 반영하는 방식은 두 가지.
1. 앱 서버가 모든 레플리카에 직접 쓰기
2. DB 자체 복제 기능 사용 — 앱 서버는 한 번만 쓰고 DB가 알아서 동기화

### 데이터 쓰기 옵션 4가지

| 방식 | 동작 | 지연 | 비고 |
|------|------|------|------|
| 직렬 동기 (serial sync) | db1 → db2 → db3 순서대로 응답 받고 클라이언트 응답 | 매우 김 | 가장 안전, 가장 느림 |
| 직렬 비동기 (serial async) | db1만 쓰고 응답, 나머지는 비동기 | 짧음 | 일관성 약화 |
| 병렬 비동기 (parallel async) | 전체에 동시 쓰기, 일부 응답만 받고 클라이언트 응답 | 짧음 | 스레드 자원 ↑ |
| 메시지 큐 (Kafka 등) | 큐에 기록 후 응답, 소비자가 레플리카에 반영 | 가장 짧음 | 부하 처리에 유리 |

### 데이터 읽기 옵션 3가지

- 하나의 레플리카에서만 읽기 — 빠르지만 옛날 데이터 가능성
- 일부 레플리카(과반수)에서 읽기 — 일관성 검증 후 반환
- 모든 레플리카에서 읽기 — 가장 정확, 가장 느림

이 조합이 결국 **일관성 vs 속도** 트레이드오프의 본질.

---

## 2.2 일관성 (Consistency)

**모든 노드가 항상 같은 데이터를 보여주는가**.

복제된 노드들에 같은 업데이트를 같은 순서로 적용해서, 어느 노드에 물어봐도 동일한 답이 나오게 만드는 것.

분산 시스템에서 다루는 일관성은 크게 두 종류.

### 2.2.1 강한 일관성 (Strong Consistency)

쓰기 직후, 모든 읽기는 무조건 최신 값을 반환한다.

- 모든 노드가 동일 순서로 업데이트 → 시스템이 선형적 상태처럼 보임
- 구현 방법: 분산 트랜잭션, 분산 잠금, Paxos, Raft 같은 합의 알고리즘
- 장점: 예측 가능, 직관적
- 단점: 모든 노드가 동의할 때까지 대기 → 속도 ↓, 가용성 ↓

은행/금융처럼 데이터 불일치가 곧 사고로 이어지는 도메인에 어울린다.

### 2.2.2 최종 일관성 (Eventual Consistency)

일시적 불일치는 OK, 다만 시간이 지나면 결국 모두가 같은 값으로 수렴한다.

- 노드는 비동기로 업데이트 → 잠깐 동안 노드마다 다른 값 가능
- 동기화가 늦어지는 원인: 네트워크 지연, 메시지 전달 속도, 레플리카 동기화 등
- 충돌 해결 기법: Conflict resolution, Replication, Gossip protocol
- 장점: 가용성 ↑, 확장성 ↑, 응답 빠름
- 단점: 일시적 불일치를 처리할 로직이 앱에 필요

### 강한 일관성 vs 최종 일관성 — 야마다 라멘 비유

책에서는 도쿄의 라멘 맛집 "야마다 라멘"의 매운 소스 재고 시스템으로 비유한다.

- **강한 일관성**: 손님 둘이 동시에 마지막 매운 소스를 노릴 때, 시스템이 모든 화면을 동시에 갱신해서 먼저 누른 사람만 성공. 다른 사람 화면엔 즉시 "품절" 표시. 정확한 대신 화면 갱신까지 잠깐 대기가 생긴다.
- **최종 일관성**: 한 명이 추가했는데 다른 화면에는 잠깐 반영이 안 돼서 둘 다 추가 버튼을 눌러버림 → 결국 서버에서 한 명은 자동 취소. 화면 반응은 빠른 대신 일시적 혼란이 생긴다.

강한 일관성은 정확함을 위해 기다리는 쪽, 최종 일관성은 속도를 위해 잠깐의 혼란을 감수하는 쪽이다.

### r + w > n 공식

- `n` = 전체 레플리카 수
- `r` = 읽을 때 참조할 레플리카 수
- `w` = 쓸 때 반영할 레플리카 수

| 설정 | 일관성 | 쓰기 속도 | 읽기 속도 |
|------|--------|-----------|-----------|
| `w=1, r=3` | 강한 일관성 | 빠름 | 느림 |
| `w=3, r=1` | 강한 일관성 | 느림 | 빠름 |
| `w=2, r=2` | 강한 일관성 | 동일 | 동일 |
| `w=1, r=1` | 최종 일관성 | 빠름 | 빠름 |

**`r + w > n`이면 강한 일관성, 아니면 최종 일관성.**

읽기 노드 집합과 쓰기 노드 집합이 반드시 하나 이상 겹쳐야 최신 값을 보장할 수 있기 때문.

## 2.3 가용성 (Availability)

**일부 장애가 있어도 사용자 요청에 계속 응답할 수 있는 능력**.

예약 시스템에서 일부 레플리카만으로도 쓰기/읽기를 처리하면 가용성이 올라간다. 일부 노드가 죽어도 나머지로 서비스 가능하니까.

### 가용성을 보장하는 방법

- **다중화(Redundancy)** — 하드웨어든 소프트웨어든 여러 개 준비
- **복제(Replication)** — Active-Passive(주/대기), Active-Active(전부 동시 작동) 두 패턴
- **로드 밸런싱** — 부하 분산, CPU/메모리/I/O 병목 방지
- **장애 감지 및 복구** — heartbeat, 모니터링, 헬스체크
- **장애 전환 및 복귀(Failover & Failback)** — 문제 노드는 자동으로 백업 노드로 우회, 복구되면 다시 원복

가용성을 높일수록 복잡도가 늘고 자원 소모도 커지고, 일관성 문제까지 따라온다. 또 트레이드오프.

---

## 2.4 파티션 허용성 (Partition Tolerance)

### 2.4.1 네트워크 파티션이란

네트워크 장애로 노드끼리 서로 통신할 수 없게 된 상태. 분산 시스템이 여러 개의 고립된 그룹으로 쪼개지는 상황이다.

원인은 네트워크 장애, 하드웨어 고장, 시스템 버그, 잘못된 네트워크 설정, 네트워크 공격 등.

```
사용자 ──→ [App] ──쓰기──→ [db1] ──×── [db2]   ← db2 고립
                                ↓
                              [db3]
                              
→ db1에 쓴 데이터가 db2로 전달 안 됨
→ db2를 읽으면 옛날 데이터 반환
```

### 2.4.2 파티션 허용성

**네트워크 파티션이 발생해도 시스템이 계속 작동할 수 있는 능력**.

- 고립된 노드도 자기 클라이언트 요청은 독립적으로 처리 가능
- 클라우드, 분산 DB, 대규모 분산 앱처럼 가용성이 중요한 곳에서 필수
- 파티션이 생겨도 성능이 갑자기 죽지 않고 점진적으로 저하되도록 설계

CAP 정리에서 P는 보통 포기 불가로 본다. 분산 시스템 = 네트워크 = 파티션은 언제든 일어나니까. 결국 실질적 선택은 C와 A 사이의 트레이드오프.

---

## 2.5 지연 시간 (Latency)

요청이 들어와서 응답이 나갈 때까지 걸리는 시간. 데이터가 한 지점에서 다른 지점으로 이동하는 데 걸리는 시간.

영향 요인
- 노드 간 거리
- 네트워크 혼잡도
- 각 노드의 처리 시간
- 전송 데이터의 크기와 복잡성

### 지연 시간 줄이는 기법

- **네트워크 최적화** — 빠른 연결, **네트워크 홉(network hop)** 수 줄이기. 홉 = 데이터가 거치는 중간 지점이라 많아지면 그만큼 늦어진다.
- **캐싱** — 메모리, CDN, DB 캐시
- **데이터 지역화** — 사용자 가까이 데이터 배치 (복제, 에지 컴퓨팅, 콘텐츠 분산)
- **비동기 통신** — 메시지 큐, 이벤트 기반 아키텍처로 결합도 ↓
- **성능 튜닝** — 쿼리/알고리즘/코드 최적화

지연 시간을 0으로 만드는 건 불가능하다. 무리하게 줄이려 들면 일관성이나 장애 허용성이 깎인다.

---

## 2.6 내구성 (Durability)

**저장된 데이터가 장애가 나도 사라지지 않게 보장하는 능력**.

분산 시스템은 노드가 많은 만큼 장애 위험도 분산되어 있어서 내구성이 중요해진다.

확보 방법은 두 가지.
- **복제(Replication)** — 여러 노드에 사본
- **백업(Backup)** — 심각한 장애나 사고 대비

금융, 의료처럼 데이터 손실이 치명적인 도메인이나 메시징 플랫폼/소셜 미디어처럼 지속적으로 운영되어야 하는 서비스에서 특히 중요하다.

내구성을 올리려면 복제를 늘려야 하고, 그러면 일관성 유지에 시간이 더 걸린다. 또 트레이드오프.

---

## 2.7 신뢰성 (Reliability)

**다양한 오류·장애 속에서도 시스템이 원래 기능을 꾸준히 제공하는 능력**.

대상 오류는 하드웨어 고장, 네트워크 문제, 앱 버그, 휴먼 에러(사람 실수).

신뢰성을 높이는 방법은 가용성과 거의 겹친다.
- 다중화 — 장비 여러 개
- 자동 감지 및 자가 복구
- 복제
- 로드 밸런싱

가용성과 비슷해 보이지만 미묘하게 다르다.
- **가용성** = "응답을 하느냐"
- **신뢰성** = "응답이 정확하고 일관되게 동작하느냐"

---

## 2.8 장애 허용성 (Fault Tolerance)

**일부 요소가 죽거나 네트워크가 망가져도 시스템이 정상 작동을 이어가는 능력**.

핵심은 사람이 개입하지 않고 자동으로 감지·복구가 가능하도록 설계하는 것.

구현 기술
- 다중화(중복성)
- 복제
- 장애 감지 및 복구 메커니즘 — 응답 없는 노드와의 통신 시도, 백업 노드로 전환 등

장애 허용성은 신뢰성과 가용성을 받쳐주는 토대 역할을 한다.

---

## 2.9 확장성 (Scalability)

1장에서 잠깐 다뤘던 주제. 사용자/데이터가 늘어나도 성능과 신뢰성을 유지하면서 더 많은 작업을 처리할 수 있는 능력.

### 인스턴스(Instance)란

- 일반 프로그래밍에서: 클래스(설계도)로 만든 개별 객체
- 클라우드에서: 서버를 독립적·가상화된 단위로 만든 것 = 인스턴스
- 야마다 라멘 비유로 치면: 소라 휴대폰 화면, 민지 휴대폰 화면이 각각 독립된 인스턴스로 동작.

### 2.9.1 수직 확장성 (Vertical Scaling, Scale-Up)

**한 노드의 자원을 키우는 방식.** CPU 추가, 메모리 추가, 인스턴스 유형 업그레이드 등.

장점
- 단순성 — 아키텍처/SW 변경 최소
- 소규모 작업에 비용 효율적 — 노드 여러 개 관리할 필요 없음

단점
- 하드웨어 한계점 — 결국 물리적 상한이 오고 비용 곡선이 가팔라짐
- 단일 장애점(SPOF) — 그 노드 죽으면 시스템 전체 다운

### 2.9.2 수평 확장성 (Horizontal Scaling, Scale-Out)

**노드/인스턴스를 여러 개 추가하는 방식.** 분산 처리, 병렬화로 용량 확장.

장점
- 용량·성능 ↑
- 장애 허용성 ↑ — 한 노드 죽어도 다른 노드 살아있음

과제
- 분산 조정 — 노드 간 동기화 필요. 재고 카운터를 두 노드가 동시에 차감하면 음수까지 갈 수 있는 경우가 대표적
- 데이터 일관성 — 단일 노드보다 어렵다. 분산 트랜잭션이나 최종 일관성 기법 필요

### 둘은 양자택일이 아니다

수직과 수평은 섞어 쓰는 게 정상이다. 계층마다 적합한 방식이 다르다 — DB는 수직, 앱 서버는 수평 같은 식.

모듈화와 느슨한 결합(loose coupling)이 확장성을 받쳐주는 핵심 구조다. DB·인증·파일 저장 같은 걸 별도 모듈로 두면 한 부분만 확장하거나 교체할 수 있다.

## 2.10 요약

이 장에서 다룬 8가지 속성:

| 속성 | 한 줄 정의 |
|------|------------|
| 일관성 | 모든 노드가 같은 데이터를 보여주는가 |
| 가용성 | 장애가 있어도 응답할 수 있는가 |
| 파티션 허용성 | 네트워크가 쪼개져도 동작할 수 있는가 |
| 지연 시간 | 응답까지 얼마나 걸리는가 |
| 내구성 | 데이터가 사라지지 않는가 |
| 신뢰성 | 정확하고 일관되게 동작하는가 |
| 장애 허용성 | 일부가 죽어도 자동 복구되는가 |
| 확장성 | 부하가 늘어도 처리할 수 있는가 |

핵심은 **이 속성들이 서로 영향을 주고받는다**는 것. 하나를 극단으로 끌어올리면 다른 하나가 깎인다.
