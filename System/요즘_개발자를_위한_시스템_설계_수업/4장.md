
3장에서 이론을 다뤘다면 4장은 실제 분산 시스템의 입구를 책임지는 세 가지 구성 요소를 다룬다.

- **DNS** — 도메인 이름을 IP 주소로 변환
- **로드 밸런서** — 트래픽을 여러 서버에 분산
- **애플리케이션 게이트웨이** — 클라이언트와 백엔드 사이에서 라우팅·보안·인증 등 통합 처리

## 이 장 핵심

요청이 사용자의 브라우저에서 출발해 백엔드 서버까지 도달하는 동안 거치는 인프라 계층을 정리한다. 세 요소 모두 "트래픽을 어떻게 나누고, 어디로 보낼지"의 문제를 다루지만 책임 범위가 다르다.

```
사용자 → DNS → 로드 밸런서 → 애플리케이션 게이트웨이 → 백엔드 서비스
        (이름→IP)  (서버 분산)    (라우팅·보안·통합)
```

## 4.1 DNS 이해

도메인 네임 서버(DNS)는 사람이 기억하기 쉬운 도메인 이름을 컴퓨터가 통신할 수 있는 IP 주소로 변환하는 시스템.

```
1. 사용자: google.com 접속
2. 웹 브라우저 → DNS 쿼리
3. DNS 인프라가 IP 주소 반환
4. 브라우저가 그 IP로 HTTP 요청
```

### DNS 구성 요소

- **네임 서버(Name Server)** — DNS 쿼리에 응답하는 서버. 하나가 아니라 수많은 서버가 네트워크를 이룬다.
- **리소스 레코드(Resource Record, RR)** — 매핑 정보를 저장하는 단위
  - A 레코드: 호스트 이름 → IP 주소
  - NS 레코드: 도메인 → 권한 있는 네임 서버
  - CNAME 레코드: 별칭 → 정식 도메인 (`www.example.com` → `example.com`)
  - MX 레코드: 도메인 → 메일 서버
- **캐싱** — 단계마다 캐시를 둬서 DNS 인프라 부하와 응답 시간을 줄임
- **계층 구조** — 네임 서버가 트리 구조로 구성됨

### 네임 서버 계층

```
        루트 서버
       /    |    \
    .com  .org  .edu
      |
   google.com (권한 있는 네임 서버)
```

| 계층 | 역할 |
|------|------|
| **루트 서버** | 최상위 도메인의 네임 서버 정보를 관리 (전 세계 약 1000개 인스턴스) |
| **최상위 도메인 서버(TLD)** | 해당 도메인의 권한 있는 네임 서버 IP 보유 |
| **권한 있는 네임 서버** | 특정 조직/도메인의 최종 DNS 정보 제공 |

> "권한 있는(Authoritative)"은 영어 직역이라 어색한데, 실제로는 **그 도메인에 대한 최종 결정권을 가진 서버**를 뜻한다.

### 리졸버(Resolver)란

도메인 이름을 IP 주소로 변환해주는 중개자. 사용자/애플리케이션의 요청을 받아 루트 → TLD → 권한 있는 네임 서버 순서로 쿼리를 던지며 IP를 찾아냄. 보통 ISP나 회사 네트워크의 로컬 DNS 서버가 이 역할을 함.

### 4.1.1 DNS 쿼리 — 반복적 vs 재귀적

**반복적 쿼리 (Iterative)**

컴퓨터가 로컬 DNS 리졸버에 모든 과정을 직접 처리하게 한다. 리졸버는 루트 → TLD → 권한 있는 네임 서버에 **순차적으로 직접 쿼리**를 던지며 IP를 찾는다.

```
사용자 → 리졸버: "www.example.com IP 뭐죠?"
리졸버 → 루트:  "최상위 도메인 정보 주세요"
리졸버 → TLD:  "권한 있는 네임 서버 정보 주세요"
리졸버 → 권한 서버: "IP 뭐죠?"
사용자 ← 리졸버: "192.168.1.1"
```

**재귀적 쿼리 (Recursive)**

사용자는 로컬 리졸버에 한 번만 요청하고, **이후 모든 과정은 리졸버가 알아서** 처리한다.

```
사용자 → 리졸버: "www.example.com IP 뭐죠?"
            ↓ (리졸버가 내부적으로 다 처리)
사용자 ← 리졸버: "192.168.1.1"
```

도서관 비유로 말하면
- 반복적: 사서가 "난 모르니 옆 사서한테 가보세요"라고 알려주면 방문자가 직접 옆 사서에게 가서 묻는 방식
- 재귀적: 사서가 다른 사서들에게 직접 물어다가 책을 가져다주는 방식

> 일반적으로 인터넷에서는 **재귀적 방식**을 사용한다. 사용자 입장에서 훨씬 편하고, ISP 리졸버가 캐싱까지 잘해준다.

### IP 직접 사용 vs 도메인 사용

`.env` 파일에 `SERVER_API = https://api.my-server.com` 식으로 도메인 이름을 쓰는 이유.

IP 주소를 직접 박으면 DNS 변환 과정이 생략돼 응답 시간은 줄지만, 다음 이유로 거의 도메인을 쓴다.

- **유연성** — IP가 바뀌어도 도메인은 그대로. 코드/환경변수 수정 불필요
- **로드 밸런싱** — 한 도메인에 여러 IP를 매핑해서 분산 가능
- **보안** — TLS/SSL 인증서가 도메인 기반. IP만 쓰면 인증 검증이 어려움

### DNS 캐싱

자주 접근하는 리소스 레코드를 임시 저장. 캐싱이 들어가는 위치는

- 사용자 웹 브라우저
- 운영 체제
- 사용자 네트워크 내 로컬 네임 서버
- ISP의 리졸버

여러 단계에서 캐싱하면 DNS 인프라 전체 부하가 크게 줄고, 사용자도 빠른 응답을 받는다.

> 책의 노트에서 "DNS 인프라 부하를 줄이려면 어떤 메커니즘이 더 적합?" 질문을 던지는데, 정답은 **반복적 방식이 아니라 재귀적 방식**. 재귀적은 한 번 처리한 결과를 캐싱하기 좋아서 후속 쿼리에 자유로움. 반복적은 매번 새로 처리해서 캐싱 효과가 떨어진다.

## 4.2 DNS의 확장성, 신뢰성, 일관성

### 4.2.1 확장성

루트 서버 → TLD → 권한 있는 네임 서버로 이어지는 **계층 구조 자체가 확장성의 원천**. 전 세계에 약 1000개의 루트 서버 인스턴스가 분산돼 있고, 각 단계가 부하를 나눠 받는다. 한 서버가 죽어도 다른 서버가 대신 처리해서 안정성이 유지됨.

### 4.2.2 신뢰성

- 여러 단계에서 캐싱 → 일부 DNS 서버가 잠깐 죽어도 사용자는 영향 없음
- 전 세계 분산된 중복 서버
- 대부분의 쿼리/응답에 신뢰성 낮은 **UDP 프로토콜** 사용 (속도 우선) — 응답이 없으면 재전송하는 방식으로 보완

### 4.2.3 일관성

DNS는 **강한 일관성 대신 성능을 우선**한다. 캐시 TTL과 점진적 업데이트로 **최종 일관성**을 적용. 업데이트가 모든 서버에 반영되는 데 몇 초~며칠 걸릴 수 있다.

> 그래서 도메인 IP를 바꾼 직후엔 일부 사용자가 아직 옛날 IP로 접속하는 현상이 생긴다. TTL을 짧게 잡으면 빠르게 반영되지만 캐시 효과는 떨어짐.

## 4.3 로드 밸런서

여러 서버에 작업을 고르게 분배해 자원 효율을 높이고, 응답 시간을 줄이고, 과부하를 막는 장치.

<img width="353" height="210" alt="image" src="https://github.com/user-attachments/assets/4a380d66-72ec-4551-bd26-7594fff9e2ac" />

### 핵심 특징

- 트래픽을 여러 서버에 고르게 분산 → 특정 서버 과부하 방지
- 장애가 나거나 느려진 서버의 요청을 정상 서버로 재전송 → 내구성 ↑
- 일부 서버가 죽어도 나머지로 서비스 가능 → 가용성 ↑
- 트래픽 폭증 시 점진적 성능 저하 (완전히 멈추진 않음)
- 서버 풀에 노드를 추가해 수평 확장 가능

### 4.3.1 로드 밸런서 위치

시스템 아키텍처의 여러 지점에 배치 가능.

<img width="506" height="212" alt="image" src="https://github.com/user-attachments/assets/001d7ac2-509a-4f41-a6c2-dad06f478090" />

각 계층 사이에 LB를 두면 계층마다 독립적으로 확장할 수 있고, 한 계층의 장애가 다른 계층으로 번지는 걸 막는다.

### 4.3.2 로드 밸런서의 장점

단순 부하 분산을 넘어서 다음 기능까지 수행한다.

- **헬스 체크(Health Check)** — 백엔드 서버가 정상인지 주기적으로 확인하고, 응답 없는 서버에는 요청을 보내지 않음
- **TLS 종료(TLS Termination)** — 암호화된 트래픽을 LB에서 복호화해서 백엔드로 전달. 백엔드 서버가 복호화에 자원을 안 써도 됨
- **트래픽 패턴 분석 + 효율적 분배 알고리즘** — 자원 활용 극대화

### 4.3.3 전역 로드 밸런싱 vs 로컬 로드 밸런싱

```
                    ┌→[로컬 LB]→ A 지역 서버 랙
네트워크 트래픽 →[전역 LB]→[로컬 LB]→ B 지역 서버 랙
                    └→[로컬 LB]→ C 지역 서버 랙
```

| 종류 | 역할 |
|------|------|
| **전역 서버 LB (GSLB)** | 여러 데이터 센터 사이에 트래픽 분산. 사용자 위치, 데이터 센터 상태, 서버 수 기반 결정 |
| **로컬 LB** | 한 데이터 센터 내부에서 자원 활용도 최적화 |

전역은 데이터 센터 단위 장애·전력 차단 시 다른 센터로 자동 전환. 온프레미스로도 쓰고 LBaaS 형태로도 쓸 수 있음.

### 4.3.4 DNS와 전역 로드 밸런서

DNS도 전역 LB 역할을 어느 정도 할 수 있다 — 한 DNS 쿼리에 여러 IP를 반환해 라운드 로빈으로 분산.

다만 한계가 있다.
- DNS 패킷 크기가 512바이트로 작아 모든 서버 IP를 응답에 못 담음
- 클라이언트가 어떤 IP를 선택할지 DNS는 통제 불가 → 과부하 데이터 센터로 갈 수도
- 가장 가까운 서버를 결정하지 못함 (지리적 위치 기반 연결, 애니캐스팅 같은 기술 필요)
- 장애 발생 시 TTL 길면 캐시된 옛 정보로 인해 복구 지연

> 결론: **DNS 로드 밸런싱만으론 부족하고, 데이터 센터 내부에 별도 로컬 LB를 둬야 한다.**

### 4.3.5 로드 밸런서가 사용하는 알고리즘

| 알고리즘 | 설명 | 특징 |
|---------|------|------|
| **라운드 로빈** | 각 요청을 순차적으로 다음 서버에 할당 | 가장 단순, 서버 차이 미고려 |
| **가중치 라운드 로빈** | 서버 용량에 따라 가중치 부여 | 서버 성능 차이 반영 |
| **최소 연결** | 현재 연결 수가 가장 적은 서버에 할당 | 처리 시간 긴 요청 많을 때 효과적 |
| **최소 응답 시간** | 응답 시간이 가장 짧은 서버에 할당 | 성능 중요 서비스 |
| **IP/URL/일관성 해시** | IP나 URL 기반으로 특정 서버에 매핑 | 같은 클라이언트/URL을 같은 서버로 라우팅 필요할 때 |

#### 정적 알고리즘 vs 동적 알고리즘

- **정적**: 서버 상태를 실시간 반영하지 않고, 미리 설정된 구성에 따라 분배 → 단순한 환경에 적합
- **동적**: 서버의 현재 상태(부하·처리 능력)를 고려해 분배 → 더 복잡하지만 효과적

> 실무에선 동적 알고리즘을 더 많이 쓴다. 부하 분산 효과가 뛰어나기 때문. 서버가 적은 단순 환경에선 정적도 충분.

#### 스테이트풀 vs 스테이트리스

| 종류 | 동작 | 비유 |
|------|------|------|
| **스테이트풀(Stateful)** | 클라이언트와 백엔드 서버 간 연결 상태 정보를 저장. 같은 클라이언트는 같은 서버로 계속 라우팅 | 택배 회사가 특정 고객 담당 기사를 정해두고 그 기사가 항상 배송 |
| **스테이트리스(Stateless)** | 세션 상태를 저장하지 않고, 일관된 해싱 등으로 매번 매핑 | 매번 무작위로 다른 기사가 배송 |

스테이트풀은 가용성·신뢰성이 좋지만 모든 LB가 상태 정보를 공유해야 해서 복잡하고 확장성이 떨어진다. 스테이트리스는 빠르고 확장성이 좋지만 서버 추가/제거 시 일관성 유지가 어려움.

> 실제 운영 시스템은 두 방식을 **혼합**해서 쓴다. VIP 고객은 같은 기사가 담당하고, 일반 고객은 매번 다른 기사가 배송하는 식.

### 4.3.6 OSI 모델의 각 계층에서 로드 밸런싱

| 계층 | 기준 | 특징 |
|------|------|------|
| **전송 계층 (4계층, L4)** | TCP/UDP 연결 | 빠름. TCP 기반 안정성 제공 |
| **응용 계층 (7계층, L7)** | HTTP 헤더, URL, 쿠키, 사용자 ID | 정교한 분배. 속도 제한, HTTP 라우팅, 헤더 재작성 등 가능 |

대부분의 LB 솔루션은 두 계층을 **함께 사용**한다. L4로 기본 분산하고, L7으로 세밀한 제어.

### 4.3.7 로드 밸런서의 배치

데이터 센터 내 LB는 단계별로 배치된다.

| 단계 | 역할 |
|------|------|
| 0단계 | DNS 시스템으로 여러 IP 제공 |
| 1단계 | 특수 라우터가 라운드 로빈/IP 기반 분배 |
| 2단계 | L4 LB가 사용자 세션을 일관 라우팅 |
| 3단계 | L7 LB가 서버 상태 모니터링 + 트래픽 분산 |

하위 계층은 단순 분배, 상위로 갈수록 정보 기반 정교한 분산.

### 4.3.8 로드 밸런서의 구현

| 구현 방식 | 특징 |
|----------|------|
| **하드웨어 LB** | 1990년대 도입. 성능은 뛰어나지만 비싸고 설정 까다로움. 벤더 종속 위험 |
| **소프트웨어 LB** | 유연한 프로그래밍 가능. 비용 효율적, 관리 쉬움 |
| **클라우드 LB (LBaaS)** | 클라우드 업체가 제공. 사용량 기반 비용, 글로벌 트래픽 관리, 모니터링 통합 |

> 요즘은 소프트웨어/클라우드 LB가 대세. 하드웨어 LB는 초고성능이 필요한 일부 환경(통신사·금융권 코어)에서만. 실제 시스템은 여러 종류를 **하이브리드**로 쓰는 경우가 많다.

## 4.4 애플리케이션 게이트웨이

로드 밸런서가 트래픽 분배에 집중한다면, 애플리케이션 게이트웨이(API 게이트웨이)는 **클라이언트와 백엔드 사이에서 라우팅·보안·성능 가속·분석·유연성**을 한 군데서 처리하는 고도화된 프록시.

<img width="421" height="343" alt="image" src="https://github.com/user-attachments/assets/99a5cd6b-d958-411e-8178-ba864e53337f" />

여러 독립 서비스를 하나의 통합 API로 묶어줘서 **마이크로서비스 환경**에서 진가를 발휘한다.

### 4.4.1 애플리케이션 게이트웨이의 기능과 역할

- **고급 요청 라우팅** — 호스트 이름, 경로, 헤더, IP 주소 등 다양한 조건으로 적절한 백엔드로 전달
- **보안** — 인증, 접근 제어, TLS 종료, DDoS 방어. **WAF**(웹 애플리케이션 방화벽) 통합으로 OWASP 위협 대응
- **가속화와 오프로딩** — 캐싱, 압축, TLS 오프로딩으로 백엔드 부담 감소
- **모니터링** — 로그, 메트릭, 트레이스를 한 곳에서 통합 수집
- **적응성** — 레거시와 통신할 때 프로토콜 변환

> 넷플릭스 같은 회사는 마이크로서비스를 수천 개 배포해놓고 게이트웨이로 묶어서 관리한다.

---

## 4.5 마이크로서비스 아키텍처

큰 시스템을 독립적이고 재사용 가능한 여러 서비스로 쪼개는 방식. 서비스끼리는 REST/HTTP 같은 가벼운 프로토콜로 통신.

### 게이트웨이가 마이크로서비스에 주는 가치

- 여러 마이크로서비스를 **하나의 논리적 API**로 묶어 클라이언트에 제공 → 클라이언트는 백엔드 변화에 영향 안 받음
- **서비스 디스커버리**와 동적 요청 라우팅 → 인스턴스 간 LB
- 보안·모니터링·안정성 같은 **횡단 관심사(cross-cutting concern)**를 중앙에서 처리 → 서비스마다 중복 코드 불필요
- 백엔드와 무관하게 게이트웨이 새 버전 배포 가능

> 횡단 관심사 = 인증·로깅·보안처럼 시스템 여러 부분에서 공통으로 필요하지만 개별 기능으로 관리하긴 어려운 관심사.

## 4.6 클라우드 네이티브 애플리케이션 게이트웨이

주요 클라우드 업체별 제공 서비스.

| 업체 | 서비스 |
|------|--------|
| **AWS** | API Gateway (생성·배포·관리·보안), Application Load Balancer |
| **Azure** | Application Gateway (L7 LB, WAF, SSL 오프로딩, 종단 간 TLS, 자동 확장) |
| **GCP** | Cloud Armor (DDoS·방화벽), Cloud CDN |
| **Kubernetes** | Istio, Kong, Traefik, Ambassador 같은 인그레스 컨트롤러 |

서버리스 게이트웨이는 클라우드 네이티브 환경에서 운영 부담을 크게 줄여준다.

## 4.7 온프레미스 옵션

클라우드 외에도 자체 데이터 센터를 운영하는 회사(특히 금융권)는 오픈 소스 게이트웨이를 많이 쓴다.

| 솔루션 | 특징 |
|--------|------|
| **Kong** | 경량 프록시 기반. 플러그인으로 인증·보안·분석 지원 |
| **Tyk** | 강력한 접근 제어, 개발자 포털, REST API 기반 |
| **NGINX** | API 게이트웨이 + LB. 속도 제한, 접근 제어, 서비스 디스커버리 통합 |
| **HAProxy** | 빠르고 경량화된 LB·프록시. 응용 계층 요청 관리 |

## 4.8 요약

이 장에서 다룬 것

- **DNS** — 도메인 → IP 변환. 계층적 구조로 확장성 확보, 캐시 기반 최종 일관성
- **로드 밸런서** — 여러 서버에 트래픽 분산. L4/L7, 정적/동적, 스테이트풀/리스 등 다양한 축으로 분류
- **애플리케이션 게이트웨이** — 라우팅·보안·통합을 한 번에. 마이크로서비스 환경에서 핵심
- 클라우드 네이티브와 온프레미스 옵션 모두 존재
