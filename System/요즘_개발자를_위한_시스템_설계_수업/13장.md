# 13장. 구글 독스 서비스 설계

## 이 장 핵심

여러 사람이 같은 문서를 동시에 편집할 때 "최종 상태를 어떻게 일관되게 유지할 것인가". 책은 **OT (Operational Transformation)** 알고리즘을 핵심 기법으로 제시한다. 그 외에 WebSocket을 통한 양방향 통신, 5개 마이크로서비스 구조가 등장.

```
[Client] ⇄ [WebSocket Server] ⇄ [협업 서비스 + OT 엔진]
              ↑                       ↓
              │                  [분산 캐시]
              │                  [메시지 큐]
         [Client]                     ↓
              ⇅                  ┌──[문서 서비스]
       [API Gateway] ────────────┼──[수정 기록 서비스]
                                 ├──[접근 제어 서비스]
                                 └──[알림 서비스]
```

## 13.1 기능적 요구사항

| 영역 | 요구사항 |
|------|----------|
| **사용자 등록/인증** | 회원가입, 로그인, 세션 관리 |
| **문서 CRUD** | 생성, 편집, 삭제, 풍부한 텍스트 편집기 |
| **실시간 협업 및 동기화** | 동시 편집, 변경 사항 실시간 동기화, 충돌 방지, 다른 사용자 커서 위치 확인 |
| **공유 및 접근 권한** | 공유 링크, 보기 전용 / 댓글 / 편집 권한 구분, 권한 철회 |
| **버전 기록** | 자동 저장, 이전 버전 복원, 변경 이력 추적 |
| **댓글과 수정 제안** | 특정 부분에 댓글, 직접 수정 없이 제안 |
| **검색/문서 정리** | 제목/내용/메타데이터 검색, 폴더/카테고리 분류 |

## 13.2 비기능적 요구사항

- **확장성** — 대규모 사용자/문서, 서비스 독립 확장
- **가용성** — 다중화, 지리적 중복성, 데이터 센터 재해에도 견딤
- **낮은 지연 시간** — 실시간 협업/동기화 필수, 캐싱·통신 최적화
- **데이터 일관성/무결성** — 여러 사람 동시 수정 시 동기화·병합, 충돌 해결

## 13.3 데이터 모델

8개 엔티티. 협업의 핵심 엔티티는 **Revision**과 **CollaboratorPermission**.

<img width="605" height="368" alt="image" src="https://github.com/user-attachments/assets/b171f7fd-e823-4b1a-aa32-2b32d817b235" />

### 데이터 간 관계

**일대다 관계**
- User ↔ Document (한 사용자가 여러 문서 소유)
- Document ↔ Revision (한 문서는 수정 기록 여러 개)
- Document ↔ Comment / Suggestion
- User ↔ Folder

**다대다 관계**
- User ↔ Document (협업 관계 — CollaboratorPermission이 중간)
- Folder ↔ Document (FolderDocument가 중간)

## 13.4 시스템 규모 산정

### 가정

| 항목 | 값 |
|------|------|
| 전체 사용자 | 1000만 |
| 사용자당 평균 문서 | 100개 |
| 문서 하나 평균 크기 | 50KB |
| 문서당 평균 수정본 | 10개 |
| 수정본 하나 평균 크기 | 50KB |
| 문서당 평균 협업자 | 5명 |
| 문서당 평균 댓글 | 20개 (댓글당 1KB) |
| 사용자당 폴더 | 10개 |

### 저장소

| 데이터 | 계산 | 결과 |
|--------|------|------|
| 문서 | 10억 개 × 50KB | **50TB** |
| 수정본 | 100억 개 × 50KB | **500TB** |
| 댓글 | 200억 개 × 1KB | **20TB** |
| 협업 권한 메타데이터 | 10억 × 5 × 1KB | **5TB** |
| 사용자/폴더 메타데이터 | 합계 약 | **110GB** |
| **총합** | | **약 575TB** |

> 수정본(500TB)이 압도적. 모든 편집 이력을 다 저장하면 이게 발생.

### 대역폭

- 문서 업로드: 500GB/일
- 문서 다운로드: 5TB/일
- 협업 업데이트: 5TB/일 (10억 × 협업자 5 × 업데이트당 10KB)
- **총 대역폭: 10.5TB/일**

### 처리량

| 항목 | 계산 | 결과 |
|------|------|------|
| 문서 렌더링 QPS | 1000만 × 10번 / 86,400 | ≈ **1200/초** |
| 문서 수정 QPS | 1000만 × 10 / 86,400 | ≈ **1200/초** |
| **실시간 협업자** | 1000만 × 동시 접속 10% × 협업자 5 | **500만 명** |
| 실시간 업데이트 QPS | 500만 / 60초 | ≈ **83,000/분** |

| 저장소 | 용도 |
|--------|------|
| **관계형 DB** (PostgreSQL) | 사용자 프로필, 문서 메타데이터, 접근 권한, 협업 관계 (ACID 필요) |

| **메시지 큐 (Kafka)** | 협업 이벤트 비동기 처리 |
| **WebSocket** | 클라이언트-서버 양방향 실시간 통신 |

#### 캐싱 및 외부 시스템 연동

- **서버 캐시 (Redis)** — 자주 조회되는 문서/메타데이터
- **클라이언트 캐시 (IndexedDB / localStorage)** — 오프라인 지원, 변경 사항 로컬 저장 후 일정 간격으로 서버 동기화
- **장점**:
  - 반응 속도 향상 (로컬 캐시 즉시 반영)
  - 오프라인 지원 (인터넷 끊겨도 작업 가능, 연결 복구 시 자동 동기화)
  - 복원력 (네트워크 잠시 끊겨도 작업 손실 없음)
  - 서버 부하 감소 (변경 모아서 일정 간격 동기화)

오프라인 편집 시 발생할 수 있는 충돌은 **병합 전략 또는 사용자 수동 해결 UI**로 처리.

#### 오류 처리/일관성

- 메타데이터 수정 시 **트랜잭션** 적용
- 오류 시 자동 재시도 또는 적절한 오류 응답

#### 확장성/성능

- **스테이트리스 마이크로서비스** 형태로 배포, 인스턴스 여러 개 + LB
- 관계형 DB는 **샤딩이나 파티셔닝** (문서 ID 범위로 분할)
- S3는 기본적으로 대용량 데이터 처리 최적화

#### 협업 과정

<img width="609" height="456" alt="image" src="https://github.com/user-attachments/assets/8f537755-94d1-4d46-8f48-d4ef7f150e3c" />

#### OT (Operational Transformation) — 핵심 알고리즘

**개념**: 현재 문서 상태 + 이전에 적용된 작업 이력을 바탕으로, 새로 들어오는 작업을 변환하는 것. **작업 순서와 관계없이 모든 사용자가 동일한 최종 문서 상태**에 수렴하게 한다.

**책의 비유 — 화이트보드**

두 사람이 동시에 'Hello'와 'World'를 적으려고 한다.

```
OT 없으면:
  사람 A 먼저 → "HelloWorld"
  사람 B 먼저 → "WorldHello"
  → 서로 다른 결과

OT 있으면:
  순서와 무관하게 → 동일한 결과로 수렴
```

협업 서비스가 서버 측 문서 상태 유지, 클라이언트에서 보낸 작업을 문서 상태에 적용. 각 클라이언트의 작업 이력 추적하면서 새 작업이 들어오면 변환 → 서버 상태 반영 → 모든 클라이언트에 전파.

#### 접속 상태 관리 (Presence)

- 협업 서비스가 **해시 맵**으로 문서별 접속 사용자 관리
- 일정 간격으로 비활성/끊긴 클라이언트 확인
- 사용자 의도적 오프라인/자리 비움 알림도 처리

#### 확장성/성능

- 스테이트리스 마이크로서비스, LB로 분산
- **Redis 분산 캐시**로 문서 현재 상태/접속 정보 공유
- **Kafka**로 협업 이벤트 비동기 처리 (WebSocket 연결 관리와 이벤트 처리 분리)

#### 오류 처리/보안

- 네트워크 장애/서비스 오류 → 재연결 + 문서 상태 동기화
- 작업을 트랜잭션으로 묶어 중간 일부만 적용되는 일 방지
- HTTPS + WSS로 통신 암호화
- DoS 공격 방어를 위한 Rate Limit

> **DoS vs DDoS 차이** (책 노트): DoS는 한 컴퓨터에서 과도한 요청, DDoS는 봇넷 동원해서 여러 대가 동시에 공격. DDoS는 차단하기 어렵고 피해 규모가 큼.

## 13.7 기타 검토 사항 및 모범 사례

### 성능 최적화

- 여러 계층 캐싱
- **CDN** 활용 — 정적 자산(이미지/JS) 빠르게 전달
- DB 쿼리/인덱스 최적화
- **Lazy Loading** — 문서 내용/공동 작업자 정보를 필요한 순간에 점진적으로
- API 응답에 페이지네이션 적용

### 데이터 일관성/동시성

**낙관적 동시성 제어 vs 비관적 동시성 제어** (책 노트)

협업 편집 같은 환경에선 **낙관적 동시성 제어(Optimistic Concurrency Control)** 가 정석.

| 방식 | 철학 | 동작 |
|------|------|------|
| **낙관적** | "일단 해보고 안 되면 조정하자!" | 작업 진행 → 충돌 시 해결 (자동 또는 수동) |
| **비관적** | "혹시 문제 생길까봐 잠금부터!" | 락 걸고 한 사람씩만 작업 |

낙관적의 동작 흐름
1. 두 사람이 각자 수정한 내용을 서버로 전송
2. 서버: 같은 부분을 수정했는지 감지
3. 누가 먼저 보냈는지, 충돌을 어떻게 해결할지 판단
4. 충돌 시 자동 해결 또는 사용자가 직접 선택

- 버전 번호/타임스탬프로 동시성 제어
- 필요 시 분산 잠금, 동기화 기법 활용
- 가용성/성능 위해 **최종 일관성** 모델도 고려

### 장애 내성과 복원력

- **서킷 브레이커(Circuit Breaker) 패턴** — 일시적 장애가 시스템 전체로 퍼지는 것 방지. 일정 횟수 실패 시 자동 차단 + 일정 시간 후 재시도
- 재시도 메커니즘
- 이중화/복제로 하드웨어/네트워크 장애 대비
- 정기 백업

### 확장성과 탄력성

- 수평 확장 가능한 설계
- 자동 스케일링
- 로드 밸런싱
- 메시지 큐로 결합도 ↓ + 트래픽 폭주 대응

### CI/CD

- 파이프라인으로 빌드/테스트/배포 자동화
- Git으로 버전 관리
- 단위/통합/E2E 테스트 자동화
- 도커 + 쿠버네티스
- 배포 전략: **블루-그린 / 카나리 / 롤링**

## 13.8 요약

- **실시간 협업이 핵심 차별점** — OT 알고리즘으로 동시 편집 충돌 해결
- 5개 마이크로서비스 + WebSocket

## 개인 메모

### OT vs CRDT — 동시 편집의 두 패러다임

책은 OT만 다루는데, 실시간 협업 편집에는 사실 두 가지 큰 흐름이 있다:

| 방식 | 동작 | 사용처 |
|------|------|--------|
| **OT** (Operational Transformation) | 중앙 서버가 작업을 변환해서 모든 클라이언트에 전파 | Google Docs (실제), ShareDB |
| **CRDT** (Conflict-free Replicated Data Type) | 작업 자체가 수학적으로 교환 법칙 성립 → 순서 무관하게 같은 결과 | Yjs, Automerge, Figma 일부, Notion |

**비교**

| 항목 | OT | CRDT |
|------|----|----|
| 중앙 서버 의존 | 강함 | 약함 (P2P 가능) |
| 구현 난이도 | OT 변환 로직이 복잡 | 자료구조 자체가 복잡 |
| 메모리 효율 | 좋음 | 메타데이터 부담 ↑ (단, 최신 CRDT는 개선됨) |
| 오프라인 친화 | 어려움 | 강점 (오프라인 작업 후 합칠 때 자연스러움) |

> **Google Docs의 실제 구현은 OT** — 더 정확히는 "Operations On Documents"라는 자체 변형. 다만 최근 트렌드는 CRDT로 가는 중 (Yjs가 대세). Figma는 자체 CRDT 변형을 사용.

### Y.js, Automerge — 실무에서 자주 보는 CRDT 라이브러리

요즘 협업 기능 구현하면 굳이 OT 직접 짜지 않고 라이브러리 쓰는 게 정석:

- **Y.js** — JavaScript CRDT. 가장 대중적. JupyterLab, Evernote 일부에서 사용
- **Automerge** — JSON-like CRDT. JavaScript/Rust
- **ShareDB** — OT 진영의 대표 라이브러리

### 차등 동기화 (Differential Sync)

책엔 없는데 협업 편집의 또 다른 패턴. 일정 간격으로 변경 diff만 주고받음. Google Docs도 이 기법을 OT와 함께 사용한다고 알려져 있음. 네트워크 효율이 좋아서 모바일에 유리.

### Presence (커서 위치 공유)

책에선 "접속 상태"만 다루는데, 실제 Google Docs/Notion 같은 협업 도구에선:
- **다른 사용자의 커서 위치** 실시간 표시
- **선택 영역(selection)** 표시
- **사용자 색상 매핑**

이것도 WebSocket으로 보내는데, **너무 자주 보내면 성능 부담** → throttle/debounce가 필수.

### Conflict resolution UI

자동 해결 안 되는 충돌은 사용자에게 선택지 보여줘야 함:
- 두 버전을 나란히 보여주고 선택
- 자동 병합 시도 (Git의 3-way merge처럼)
- 가장 단순한 건 "마지막 작성 우선 (Last Write Wins)"

### WebSocket vs Server-Sent Events vs Long Polling

실시간 통신에서 보통 WebSocket인데:

| 방식 | 특징 |
|------|------|
| **WebSocket** | 양방향, 가장 강력. 기본 선택 |
| **SSE** (Server-Sent Events) | 서버→클라이언트 단방향. 가벼움, 알림용 |
| **Long Polling** | HTTP 호환성 ↑, WebSocket 안 되는 환경 폴백 |

### Snowflake ID 활용

문서 ID, 수정본 ID에 Snowflake 같은 시간 기반 ID 쓰면 정렬이 자연스러워서 좋다. 특히 수정본은 시간순으로 정렬해야 하니까.

### 면접에서 빠뜨리면 안 되는 포인트

- **OT vs CRDT 차이와 trade-off**
- **낙관적 vs 비관적 동시성** 선택 이유
- **WebSocket 기반 양방향 통신** 필요성
- **메타데이터 vs 콘텐츠 분리 저장**
- 오프라인 편집 지원과 충돌 해결 UX
- Hot document 문제 (인기 문서 동시 접속자 폭주) — 협업 서비스 수평 확장 + Redis로 상태 공유

### 책에서 다루지 않은 도전 과제

- **End-to-End 암호화** — 콘텐츠를 서버에서도 못 보게 만들면 OT/검색은 어떻게? (이건 진짜 어려운 문제. Apple Notes 같은 서비스가 이 방향)
- **AI 기반 자동 완성/제안** — 최근 Notion AI, Google Docs Smart Compose 같은 기능
- **버전 비교 UI** — Git diff처럼 시각화하는 게 까다로움
- **권한 변경의 일관성** — 협업 세션 중에 권한 철회되면 어떻게? (실시간으로 끊어내야 함)
