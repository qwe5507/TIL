# 8장. 시스템 구성 요소 설계 및 구현: API, 보안, 메트릭

## 이 장 핵심

분산 시스템을 만들 때 **"동작하는 것"** 이상으로 중요한 것들 — 어떻게 통신할지(API), 어떻게 보호할지(보안), 어떻게 들여다볼지(관측성)까지를 한 번에 정리.

```
[Client] →[API: REST or gRPC]→ [Service A] → [Service B]
              ↓                      ↓
         [API 보안]             [관측성 4종]
         - 인증/인가              ├ Logging
         - HTTPS/TLS             ├ Metrics
         - Rate Limit            ├ Alerting
                                 └ Tracing
```

## 8.1 REST API

**REST**(REpresentational State Transfer)는 네트워크 애플리케이션 설계의 아키텍처 스타일. 표준 HTTP 메서드를 그대로 활용해서 구현이 단순하고 직관적이라 가장 널리 쓰인다.

### 8.1.1 설계 원칙

- **리소스 중심** — 객체/데이터/서비스 등을 리소스로 보고 URI로 식별
- **HTTP 메서드 활용** — GET/POST/PUT/DELETE = CRUD에 매핑
- **무상태성(Statelessness)** — 요청에 필요한 모든 정보가 요청 자체에 포함. 서버가 클라이언트 상태를 기억할 필요 없음

### 8.1.2 활용 사례

- 인터넷에 공개되는 공개 API
- CRUD 기반 작업 — 사용자/제품 목록 관리 같은 단순 데이터 조작
- 웹/모바일/서드파티 — HTTP만 받으면 어디서든 쉽게 활용

### 8.1.3 장단점

| 항목 | 내용 |
|------|------|
| 장점 | HTTP 메서드/상태 코드에 익숙하면 직관적, 도구 풍부, 테스트 쉬움 |
| 단점 | 복잡한 다중 요청에는 비효율, JSON 페이로드 커질 수 있음 |

---

## 8.2 gRPC API

**gRPC**(Google Remote Procedure Call)는 구글이 만든 고성능 RPC 프레임워크. HTTP/2 기반.

### 8.2.1 설계 원칙

- **프로토콜 버퍼(protobuf)** 사용 — 언어·플랫폼 독립적인 정형 데이터 직렬화 포맷. JSON보다 빠르고 작음
- **`.proto` 파일로 서비스 정의** — 다양한 언어로 클라이언트/서버 코드 스텁 자동 생성
- 다양한 언어 지원, 인증·로드 밸런싱·**양방향 스트리밍** 지원

### 8.2.2 활용 사례

- 마이크로서비스 간 통신 — 빠르고 빈번한 호출
- 실시간 애플리케이션 — 작은 메시지 크기, 양방향 스트리밍
- 서비스 간 고성능 통신

### 8.2.3 장단점

| 항목 | 내용 |
|------|------|
| 장점 | 페이로드 작고 빠름, 명확한 데이터 구조, 코드 자동 생성 |
| 단점 | 바이너리 형식 + HTTP/2 → 디버깅 어려움. 변경 시 코드 재생성/재배포 필요 |

---

## 8.3 REST vs gRPC 비교

| 기준 | REST | gRPC |
|------|------|------|
| **성능** | JSON 기반, HTTP/1.1 (보통) | protobuf + HTTP/2 → 더 빠름, 페이로드 작음 |
| **사용 편의성** | HTTP 메서드/상태 코드, cURL/Postman으로 쉽게 테스트 | 바이너리 형식, 전용 도구 필요 |
| **호환성** | HTTP만 되면 어디서나 | HTTP/2 필요, 웹 브라우저에서 직접 사용 불가 |
| **스트리밍** | 요청/응답 단방향 | 양방향 스트리밍 지원 |
| **활용 사례** | 공개 API, CRUD 작업 | 마이크로서비스 간 통신, 실시간 |

### 정리

**REST가 좋은 경우**
- 공개 API
- 웹 브라우저에서 직접 호출
- 다양한 클라이언트 호환성 필요
- 단순 CRUD

**gRPC가 좋은 경우**
- 내부 마이크로서비스 간 통신
- 고성능 요구
- 양방향 스트리밍
- 같은 언어/팀 내 강한 타입 계약 필요

> 둘은 양자택일이 아니다. **외부 API는 REST, 내부 서비스 간은 gRPC** 조합이 표준 패턴.

## 8.4 API 보안

API는 외부 공격의 1차 대상. 보안은 선택이 아니라 필수.

### 8.4.1 인증 (Authentication)

**"사용자가 누구인가?"** 를 확인. API 보안의 첫 번째 방어선.

| 방식 | 설명 | 사용처 |
|------|------|--------|
| **API 키** | HTTP 헤더에 키 포함해서 전송. 가장 단순 | 단순 사용. 권한 세밀 제어 어려움 |
| **OAuth 2.0** | 사용자 계정 정보 직접 노출 없이 제3자에게 제한된 권한 부여 | 웹/데스크톱/모바일/IoT 전반. 업계 표준 |
| **JWT** | JSON 형식 클레임을 JWS로 감싼 토큰. MAC으로 무결성 확인 | 시스템·네트워크 간 안전한 데이터 전송 |

#### API 키 = 입장권

특정 API에 접근하는 인증 수단. 단순 인증 용도로는 유용하지만 노출 시 위험. 높은 보안이 필요하면 OAuth/JWT 권장.

**서버 코드 (Python)**
```python
VALID_API_KEYS = {"12345", "67890", "abcde"}

def check_api_key(api_key):
    if api_key in VALID_API_KEYS:
        print("Access granted.")
        return True
    else:
        print("Access denied.")
        return False
```

**클라이언트 코드 (JavaScript)**
```javascript
const API_URL = "http://example.com/api/resource";
const API_KEY = "12345";

const options = {
    method: "GET",
    headers: {
        "Authorization": `Bearer ${API_KEY}`,
        "Content-Type": "application/json",
    },
};

fetch(API_URL, options)
    .then(response => response.json())
    .then(data => console.log("응답 데이터:", data))
    .catch(error => console.error("오류 발생:", error));
```

> `Authorization: Bearer <token>` 형식은 사실상 업계 표준. "이 토큰을 가진 자가 권한 보유자"라는 의미.

### 8.4.2 인가 (Authorization)

**"이 작업을 해도 되는가?"** 인증된 사용자가 특정 리소스에 접근할 권한이 있는지 확인.

| 방식 | 설명 |
|------|------|
| **RBAC** (Role-Based Access Control) | 역할에 권한 부여, 사용자는 역할에 연결. 관리 단순 |
| **ABAC** (Attribute-Based Access Control) | 사용자/리소스/환경 속성을 조합. 더 세밀하지만 복잡 |

> 인증과 인가 헷갈리기 쉬운데 — 인증은 "신원 확인 (로그인)", 인가는 "권한 확인 (이 페이지 보여줄까?)". 영어 약자로 AuthN(인증), AuthZ(인가)로도 구분.

### 8.4.3 보안 통신

- **HTTPS** 필수 — 클라이언트-서버 통신 암호화. 가로채기/도청 방지
- **TLS 최신화** — 새 취약점 발견 시 즉시 업데이트
- 특히 비밀번호/개인정보/금융 데이터는 무조건 HTTPS

### 8.4.4 요청 속도 제한 (Rate Limiting)

특정 시간 내 클라이언트가 보낼 수 있는 요청 수를 제한. **DoS 공격 방어**에 효과적.

#### 알고리즘 3가지

**토큰 버킷(Token Bucket)**
```
   ┌──────────────┐
   │ ●●●●● (토큰) │ ← 일정 간격으로 채워짐
   │              │
   └──────────────┘
        ↓
   요청 1개 = 토큰 1개 사용
   토큰 없으면 요청 거부
```
- 비유: 놀이공원 입장권 — 시간 지나면 새 토큰 채워짐, 토큰 다 떨어지면 잠시 대기

**누수 버킷(Leaky Bucket)**
```
   ┌──────────────┐
   │ ●●●●● ↑추가  │
   │              │
   └──● 새는 구멍 │ ← 일정한 속도로 빠져나감
```
- 비유: 커피숍 직원 — 들어오는 속도는 다르지만 처리 속도는 일정. 줄이 너무 길면 손님 못 받음

**슬라이딩 윈도우(Sliding Window)**
- 특정 시간 내 요청 수 추적. 한도 초과 시 거부
- 비유: 음료 자판기 — 한 시간에 10잔만 판매, 11번째 손님은 "다음 시간에 오세요"

| 알고리즘 | 특징 |
|---------|------|
| 토큰 버킷 | 버스트 허용, 평균 속도 제한 |
| 누수 버킷 | 출력 속도가 항상 일정, 부드러움 |
| 슬라이딩 윈도우 | 정확한 시간 내 요청 수 카운트 |

## 8.5 분산 시스템 로깅

여러 서버에서 다양한 서비스가 동시에 동작하면 **각 서비스가 만든 로그를 한 곳에 모아 보는 게 필수**. 이걸 **중앙 집중형 로깅**이라고 한다.

### 8.5.1 중앙 집중형 로깅

**장점**
- **접근성** — 모든 로그가 한 곳에 모여 검색·분석 쉬움
- **상관관계 분석** — 타임스탬프나 고유 식별자로 여러 서비스 로그 연결, 트랜잭션 전체 흐름 파악
- **데이터 장기 보관** — 컴플라이언스(GDPR, HIPAA 등) 요구사항 충족

### 로그에 꼭 들어가야 할 정보

- **발생 시각** — 정확한 날짜와 시간
- **서비스 이름** — 어디서 생긴 로그인지
- **심각도** — INFO / WARNING / ERROR
- **메시지** — 이벤트 설명
- **추가 정보** — 사용자 ID, 트랜잭션 ID 등 컨텍스트

### 중앙 집중형 로깅 솔루션

| 솔루션 | 특징 |
|--------|------|
| **Logstash** (ELK) | 데이터 수집·변환 후 Elasticsearch로 전달 |
| **Fluentd** | 다양한 소스에서 데이터 수집·통합 |
| **Graylog** | 중앙 집중형 로깅 대표 솔루션, 개방형 표준 기반 |

### 8.5.2 분산 로깅 모범 사례

- **일관된 로그 형식** — 모든 서비스가 같은 형식 따르도록
- **컨텍스트 정보 포함** — 사용자 ID, 트랜잭션 ID 같은 관련 데이터
- **예외를 적절히 처리** — 스택 트레이스 포함
- **적절한 로그 레벨** — 즉시 대응 필요 = ERROR, 참고용 = INFO
- **로그 순환 및 아카이빙** — 저장 공간 관리, 오래된 로그는 삭제/아카이브

> "로그 순환(rotation)" = 냉장고 비유. 음식 계속 넣다 보면 자리가 부족 → 오래된 거 정리. 디스크 가득 차서 시스템 멈추는 사고 방지 + 중요한 데이터 보관.

## 8.6 분산 시스템 메트릭

**메트릭** = 특정 시간 간격 동안 측정한 데이터를 수치로 표현한 것. 시스템 성능과 상태를 정량적으로 평가.

### 메트릭이 답해야 하는 질문

- 시스템이 원활하게 작동 중인가?
- **SLO**(Service-Level Objectives) 충족 중인가?
- 성능 병목이 있는가?
- 예상대로 작동 중인가?

### 8.6.1 메트릭 종류

| 종류 | 예시 |
|------|------|
| **시스템 메트릭** | CPU/메모리/디스크/네트워크 사용량 |
| **애플리케이션 메트릭** | 요청 속도(RPS), 오류 비율, 응답 시간(latency) |
| **비즈니스 메트릭** | 가입자 수, 주문 건수, 매출 |

### 8.6.2 오픈소스 메트릭 솔루션

| 솔루션 | 특징 |
|--------|------|
| **Prometheus** | 다차원 모델링, PromQL 쿼리. 클라우드/마이크로서비스 모니터링 표준 |
| **Graphite** | 시계열 데이터 저장 + 그래프 시각화. 단순한 구조, 대규모 환경에 적합 |
| **Datadog** | 오픈소스는 아니지만 SaaS로 널리 사용. 폭넓은 통합 |

### 8.6.3 메트릭 구현 모범 사례

- **핵심 메트릭 선정** — SLO에 직접 영향 미치는 것에 집중
- **일관된 이름 규칙** — 분산 환경에선 특히 중요
- **오류율/지연 시간 모니터링** — 가장 먼저 문제가 드러나는 신호
- **메트릭 시각화** — 대시보드로 한눈에 보이게

> 메트릭의 **4 golden signals** (책엔 안 나오지만 SRE 표준): Latency, Traffic, Errors, Saturation. 이 네 가지만 잘 봐도 시스템 건강 상태 파악 가능.

## 8.7 분산 시스템에서 알림

알림은 **문제가 심각해지기 전에 잠재적 문제를 미리 감지**하는 적극적 모니터링 수단.

### 8.7.1 효율적인 알림 설계 — 민감도 vs 정확도

너무 민감하면 **거짓 경보(false alarm)** 가 많아져 **알림 피로(alert fatigue)** 가 생긴다. 핵심은 둘 사이 균형.

설계 시 고려할 것
- **중요도** — 모든 알림이 동등하지 않음. 긴급 / 경고 / 정보로 분류
- **실행 가능성** — 실제 조치가 필요한 경우에만. 조치 안 할 거면 알림 만들지 마라
- **문제 파악 가능 정보** — 발생 시간, 영향 서비스, 관련 오류 메시지/로그를 알림에 포함

### 8.7.2 알림 솔루션

| 솔루션 | 특징 |
|--------|------|
| **Prometheus + Alertmanager** | 중복 제거, 그룹화, 적절한 수신자에게 전달 |
| **Grafana** | 이메일, Slack, 웹훅 등 다양한 채널 알림 |
| **PagerDuty** | 사고 관리 플랫폼. 전화/SMS/푸시. 운영팀의 표준 도구 |

### 8.7.3 알림 시스템 모범 사례

- **알림을 너무 많이 발생시키지 말 것** — 알림 피로의 가장 큰 원인
- **알림 테스트** — Chaos Engineering으로 의도대로 발동하는지 검증
- **알림 프로세스 문서화** — 각 알림 유형별 대응 절차, **runbook** 필수

> Runbook = "이 알림 떴을 때 뭐부터 확인하고 어떤 명령어를 실행할지" 절차서. 새벽 3시에 호출돼도 따라할 수 있게.

## 8.8 분산 시스템에서 트레이싱

**트레이싱** = 요청이 여러 서비스를 거치며 처리되는 과정을 추적하는 기법.

분산 시스템에서 하나의 요청이 10개 서비스를 거칠 때, 어디서 느려졌고 어디서 실패했는지 파악하는 가장 강력한 도구.

### 8.8.1 분산 트레이싱 개념

```
[Service A] ─→ [Service B] ─→ [Service C]
   span 1        span 2         span 3
   └────────────────────────────────────┘
              trace 전체
```

- **Span** — 한 서비스 안에서 수행한 작업 단위 (이름, 시작/종료 시각 포함)
- **Trace** — 여러 span을 연결한 전체 요청 흐름

### 장점

- **성능 최적화** — 어디서 병목이 생기는지 시각적으로 파악
- **오류 진단** — 실패한 위치와 원인 추적
- **시스템 동작 파악** — 새 팀원이 시스템 구조 빠르게 이해

### 8.8.2 분산 트레이싱 솔루션

| 솔루션 | 특징 |
|--------|------|
| **Jaeger** | Uber 개발, 오픈소스 공개. Dapper/Zipkin에서 영감. ES/Cassandra/Kafka 백엔드 지원 |
| **Zipkin** | 마이크로서비스 지연 분석. 메모리/MySQL/Cassandra/ES 등 다양한 저장소 지원 |
| **OpenTelemetry** | 트레이스 + 메트릭 통합 표준. Prometheus/Jaeger와 연동. 다양한 언어 지원 |

### 8.8.3 분산 트레이싱 구현 모범 사례

- **코드에 트레이싱 추가** — 라이브러리 적용, 추적할 작업에 span 설정
- **컨텍스트 전파** — HTTP 헤더에 trace context를 담아 서비스 간 전달
- **스팬 이름 명확하게** — 일관성 있게, 어떤 작업인지 파악 가능하게
- **스팬에 주석 추가** — 메타데이터 포함하면 디버깅 ↑

## 8.9 요약

이 장에서 다룬 것

- **API**: REST (단순, 공개 API) vs gRPC (고성능, 마이크로서비스 간)
- **API 보안**: 인증(API 키, OAuth, JWT) + 인가(RBAC, ABAC) + HTTPS + Rate Limit
- **로깅**: 중앙 집중형(ELK), 일관된 형식, 컨텍스트 포함
- **메트릭**: 시스템/앱/비즈니스 메트릭. Prometheus/Grafana
- **알림**: 알림 피로 방지가 핵심. 중요도 분류, runbook 필수
- **트레이싱**: span/trace 개념. Jaeger/Zipkin/OpenTelemetry
