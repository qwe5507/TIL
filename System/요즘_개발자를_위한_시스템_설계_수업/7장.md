# 7장. 발행/구독과 분산 큐

## 이 장 핵심

분산 큐와 발행/구독은 서비스 간 **느슨한 결합**을 만든다. 생산자는 소비자를 신경 쓰지 않고, 소비자도 생산자가 누군지 알 필요 없다. 그 사이를 큐나 브로커가 중개해주는 구조.

```
[생산자] → [큐 / 브로커] → [소비자]
              (중개자)
```

- **분산 큐**: 메시지를 보낸 순서대로 처리하거나, 한 메시지는 한 소비자만 받는 게 보통
- **발행/구독**: 한 메시지를 여러 소비자가 동시에 받는 패턴 (1:N 브로드캐스트)

대표 솔루션은 **Kafka**(범용), **Kinesis**(AWS 매니지드).

---

## 7.1 분산 시스템의 발전 과정

분산 시스템은 모놀리식 → 마이크로서비스로 진화하면서 서비스 간 통신이 중요해졌다. 그 사이를 채워주는 게 메시징 패턴.

### 7.1.1 분산 큐 설계

**큐를 활용하는 이점**

- **부하 관리** — 생산자와 소비자 사이 완충 역할. 메시지 폭주에도 시스템이 깨지지 않음
- **결합도 ↓** — 생산자는 소비자 상태를 신경 안 써도 됨. 모듈화·유지 보수성 ↑
- **신뢰성/일관성** — 소비자가 실패해도 큐가 메시지를 다시 전달

### 분산 큐 설계 시 고려할 것

- **확장성** — 작업량·메시지 크기 증가에 대응
- **장애 허용성** — 일부가 죽어도 큐는 계속 동작
- **메시지 일관성** — 정확히 한 번 전달, 필요시 올바른 순서

### 주요 구성 요소

<img width="569" height="284" alt="image" src="https://github.com/user-attachments/assets/6cf04a5f-cf96-4859-8a6c-dfb66d17205d" />

| 구성 요소 | 역할 |
|----------|------|
| **큐 관리자(Queue Manager)** | 메시지 분배, 순서 유지, 재전송 관리 |
| **메시지 저장소(Message Storage)** | 소비자가 처리하기 전까지 메시지 보관 |
| **로드 밸런싱** | 소비자 작업량·처리 능력 고려해 균형 있는 분배 |
| **장애 허용 및 복구** | 재시도, 우회 경로 |
| **확장성** | 메시지·소비자 증가에 대응 |

### 아키텍처 설계 시 챙겨야 할 것

- **메시지 순서 보장** — 엄격한 순서가 필요한지, "최소 한 번(at-least-once)"인지 "정확히 한 번(exactly-once)"인지
- **데이터 영속성** — 소비자가 ack할 때까지 보관할지, 일부 손실을 허용할지
- **보안** — 큐 접근 권한, 메시지 무결성
- **기술 스택 선택** — RabbitMQ, Kafka, SQS 등
- **메시지 형식 설계** — 크기, 직렬화 방식, 메타데이터
- **장애 허용성** — 재시도 메커니즘, **데드 레터 큐(dead letter queue)** 로 실패 메시지 별도 관리

> 데드 레터 큐는 재시도 N번 실패한 메시지를 따로 모아두는 큐. 운영 중 디버깅에 필수.

## 7.2 발행/구독 시스템 설계

생산자가 특정 **토픽(topic)** 에 메시지를 발행하면, 그 토픽을 구독한 모든 소비자가 받는 구조. 1:N 브로드캐스트.

### 7.2.1 주요 특징

- **결합도 ↓** — 생산자는 소비자를 모름. 유연성 ↑
- **확장성과 효율성** — 대량 메시지를 여러 소비자에게 효율적 분배
- **메시지 처리의 유연성** — 실시간 분석부터 이벤트 기반 아키텍처까지 다양

### 구성

<img width="623" height="208" alt="image" src="https://github.com/user-attachments/assets/0958eb99-5cd4-4075-bf60-dfe7f099040e" />

토픽은 카테고리/채널 역할. 하나의 토픽에 연관된 여러 메시지가 논리적으로 묶임.

### 설계 시 고려할 것

**토픽 관리** — 토픽 생성/관리/삭제 방식 명확히. 동적 생성·삭제도 고려.

**메시지 라우팅** — 적절한 소비자에게 전달하는 메커니즘.

**소비자 관리** — 등록/활성 목록, 토픽별 구독 추가/취소.

**서비스 품질(QoS)** 기술
- **최대 한 번** (at-most-once) — 손실 허용
- **최소 한 번** (at-least-once) — 중복 허용
- **정확히 한 번** (exactly-once) — 둘 다 안 됨, 가장 어려움

**확장성과 로드 밸런싱** — 생산자/소비자 증가에 대응.

### 7.2.2 설계 핵심 — 확장성 + 신뢰성

**확장성**
- **수평 확장 vs 수직 확장** — 보통 수평이 유연하고 안정적 (2장 참조)
- **동적 로드 밸런싱** — 메시지 브로커가 중간에서 부하 조정. 생산자/소비자는 본연 역할에만 집중
- **토픽 파티셔닝** — 하나의 토픽을 더 작은 파티션으로 나눠 부하 분산

**신뢰성**
- **메시지 전달 보장** — at-most/at-least/exactly-once 중 선택
- **장애 허용성** — 노드에 복제, 재시도 메커니즘
- **메시지 순서** — 시간적 순서 유지

### 기본적인 발행/구독 시스템 구축 단계

1. **적절한 도구와 기술 선택** — Apache Kafka, RabbitMQ, Google Pub/Sub 등
2. **인프라 설정** — 토픽 생성, 메시지 저장·전달 인프라
3. **생산자와 소비자 구현** — 발행 코드 + 구독 코드 + 오류 처리/재시도 로직
4. **테스트 및 최적화** — 로드 밸런싱과 라우팅 튜닝

### 라디오 방송 비유

책에서 발행/구독을 라디오 방송으로 비유한 게 가장 직관적이다.

- **생산자 = DJ (방송국)** — 매일 콘텐츠(메시지) 송출
- **소비자 = 청취자** — 누군 음악, 누군 뉴스를 듣고 싶어함
- **토픽 = 채널(주파수)** — 102.1MHz는 음악, 105.3MHz는 뉴스
- **차이점** — 라디오는 그때 안 들으면 사라지지만, 발행/구독은 큐가 메시지를 잠시 보관

## 7.3 Kafka

Apache Kafka는 대용량 실시간 데이터 피드 처리를 위해 설계된 분산 스트리밍 플랫폼. 발행/구독과 큐 기반 메시징을 둘 다 구현 가능.

### 핵심 개념

| 개념 | 설명 |
|------|------|
| **레코드(Record)** | Kafka의 데이터 단위. 키, 값, 타임스탬프 포함 |
| **토픽(Topic)** | 레코드들의 카테고리/채널 |
| **생산자(Producer)** | 특정 토픽에 데이터 전송 |
| **소비자(Consumer)** | 토픽에서 데이터를 읽어 처리 |
| **브로커(Broker)** | Kafka 클러스터를 구성하는 서버. 데이터 저장·전달·소비자 관리 |
| **파티션(Partition)** | 토픽을 여러 조각으로 나눈 것. 병렬 처리와 확장성 핵심 |
| **복제(Replication)** | 파티션을 여러 브로커에 복제 → 장애 대비, 처리량 ↑ |
| **오프셋(Offset)** | 각 파티션에서 메시지가 저장된 위치(숫자). 소비자가 어디까지 읽었는지 추적 |

### Kafka의 토픽 = 라디오와 비슷하지만

라디오의 채널과 같은 개념이긴 한데, Kafka에서는 토픽을 **여러 파티션으로 쪼개서 저장**한다. 이게 라디오와의 차이점이자 Kafka의 강점.

### 아키텍처

<img width="597" height="388" alt="image" src="https://github.com/user-attachments/assets/7836cc4e-757b-48ed-8d3e-849e0757817f" />

**리더와 팔로워**
- 각 파티션마다 **리더 브로커** 1개 + **팔로워 브로커** N개
- 모든 읽기/쓰기는 리더가 처리, 팔로워는 복제만
- 리더 죽으면 팔로워 중 하나가 리더 승격

**Zookeeper의 역할**
- 브로커 메타데이터 관리
- 리더 선출 조정
- (Kafka 2.8부터는 **Kafka Raft (KRaft)** 로 Zookeeper 의존도 ↓, 컨트롤러 브로커가 그 역할)

### Kafka 공항 비유

책에서 Kafka 구조를 데이터 공항으로 비유한다.

- **활주로 = 파티션** — 노선마다 활주로 여러 개 → 비행기 여러 대 동시 이착륙
- **항공사 = 생산자** — 비행기(메시지)에 데이터 실어 보냄
- **승객 = 소비자** — 자기가 필요한 노선의 비행기 기다림
- **관제탑 = 브로커** — 비행기가 어느 활주로로 갈지 결정, 승객에게 전달
- **운항 관리 센터 = Zookeeper** — 관제탑들 조율
- **탑승권 = 오프셋** — 승객이 어디까지 받았는지 기억

### 7.3.1 Kafka 배포 단계

1. **클러스터 설정** — 브로커 개수 정하고 부하 분산
2. **Zookeeper 통합** (또는 KRaft 모드)
3. **토픽 생성 및 설정** — 파티션 수, 복제 계수 (replication factor) 설정
4. **생산자/소비자 구현** — 오프셋 관리, 동시성 처리
5. **모니터링과 유지 보수** — 클러스터 상태/성능 지표 추적
6. **성능 튜닝** — 네트워크, 디스크 I/O 최적화

> 복제 계수(replication factor)는 보통 3으로 설정. 한 노드 죽어도 데이터 손실 0, 두 노드까지 죽어도 가용성 유지.

## 7.4 Kafka Streams

Kafka 위에서 데이터를 처리하는 스트림 처리 라이브러리. **별도의 스트림 처리 프레임워크(Spark/Flink) 없이** Kafka 안에서 바로 작업 가능.

### 주요 특징

- **Kafka와 완전 통합** — 토픽에서 자연스럽게 읽고 쓰기. 파티션과 복제 기능 활용
- **고수준 DSL** — `map`, `filter`, `join`, `aggregate` 같은 연산 지원. 선언적으로 처리 로직 표현
- **저수준 Processor API** — 세밀한 제어 필요할 때 사용
- **상태 기반(Stateful) 처리** — 시간 지난 후에도 상태 유지/업데이트 가능
- **정확히 한 번 처리(Exactly-once)** — Kafka의 트랜잭션 기능 활용
- **확장성 + 장애 허용성** — Kafka 자체 메커니즘 그대로 활용

### 스트림 처리 토폴로지

```
[입력 토픽(소스)] → [Processor 1] → [Processor 2] → [출력 토픽(싱크)]
                                         │
                                    [Processor 3] → [출력 토픽]
```

- **노드** = 프로세서 (데이터 처리 단위)
- **엣지** = 스트림 (데이터 흐름)
- DSL이나 Processor API로 토폴로지 정의

### 활용 사례

- **실시간 데이터 분석** — 스트리밍 데이터에서 이상 징후 감지
- **이벤트 기반 아키텍처** — 마이크로서비스 간 통신
- **데이터 보강/변환** — 외부 데이터 결합, 형식 변환
- **부정행위 감지** — 실시간 거래/행동 분석
- **스트리밍 ETL** — Extract/Transform/Load를 실시간으로

## 7.5 Kinesis (AWS)

AWS가 제공하는 클라우드 기반 실시간 스트리밍 서비스. Kafka의 대안 격이지만 **완전 관리형**이라 운영 부담이 적다.

### 구성 요소

| 구성 요소 | 역할 |
|----------|------|
| **Kinesis Data Streams** | 핵심 서비스. 대규모 데이터 실시간 수집·처리 |
| **Kinesis Data Firehose** | 스트림 데이터를 S3, Redshift, Elasticsearch, Splunk 등으로 자동 적재 |
| **Kinesis Data Analytics** | 표준 SQL로 스트리밍 데이터 실시간 분석 |

### Kafka vs Kinesis 차이

책에서 직접 비교하진 않지만, 정리하면

| 항목 | Kafka | Kinesis |
|------|-------|---------|
| 운영 방식 | 자체 운영(또는 MSK 등 매니지드) | 완전 관리형 |
| 처리량 조정 | 브로커/파티션 추가 | 샤드(Shard) 추가 |
| 데이터 보존 | 설정에 따라 무제한 | 기본 24시간, 최대 365일 |
| 비용 모델 | 인프라 비용 | 사용량 기반 |
| AWS 통합 | 별도 설정 필요 | 네이티브 통합 (S3/Redshift/Lambda) |

> AWS 환경에서 빠르게 시작하려면 Kinesis, 멀티클라우드/온프레미스/세밀한 튜닝이 필요하면 Kafka.

## 7.6 요약

이 장에서 다룬 것

- **분산 큐** — 부하 관리, 결합도 ↓, 신뢰성. 데드 레터 큐로 실패 메시지 처리
- **발행/구독** — 1:N 브로드캐스트. 토픽 기반 라우팅, QoS(at-most/at-least/exactly-once)
- **Kafka** — 토픽/파티션/브로커/오프셋. 발행/구독과 분산 큐를 모두 구현
- **Kafka Streams** — Kafka 위에 얹어 실시간 스트림 처리
- **Kinesis** — AWS의 완전 관리형 스트리밍 솔루션
