## 14.1 기능적 요구사항

| 영역 | 요구사항 |
|------|----------|
| **사용자 등록/인증** | 이메일/비밀번호 회원가입, 세션 관리, 여러 기기 끊김 없이 |
| **콘텐츠 탐색/검색** | 영화/TV 프로그램 둘러보기, 카테고리/장르/추천, 작품명/배우/감독 검색 |
| **비디오 재생/스트리밍** | 끊김 없는 재생, 최소 버퍼링, 고품질, 다양한 해상도, 자동 화질 조정 |
| **사용자 프로필/개인 맞춤** | 한 계정에 여러 프로필, 자녀 보호 |
| **추천/개인화** | 시청 기록·취향·유사 사용자 분석, ML 기반, 피드백 학습 |
| **시청 목록/시청 기록** | 나중에 볼 콘텐츠 추가, 여러 기기 동기화, 중단 지점부터 이어보기 |
| **오프라인 시청** | 다운로드, 라이선스 계약 준수, 일정 기간 후 자동 만료 |

## 14.2 비기능적 요구사항

- **확장성/성능** — 비디오 라이브러리 수, 동시 접속자 수 폭증 대응, 수평 확장
- **가용성/안정성** — 데이터 복제, 부하 분산, 자동 장애 복구(failover)
- **콘텐츠 전달/스트리밍 품질** — **CDN** 활용, **적응형 비트레이트(adaptive bitrate)**, 버퍼링 최소화
- **다수 동시 스트리밍** — 다양한 포맷/코덱 지원

> **적응형 비트레이트(ABR)** = 사용자의 인터넷 속도와 기기 성능을 실시간 감지해서 비디오 품질을 자동 조절. WiFi에선 고화질, 데이터 신호 약해지면 자동 저화질로 전환.

## 14.3 데이터 모델

9개 엔티티. 이전 사례들과 비교하면 가장 많고 구조가 복잡한 편.

<img width="596" height="515" alt="image" src="https://github.com/user-attachments/assets/4c697992-8d1c-4e87-bb8e-21692d2ed675" />

| 엔티티 | 역할 |
|--------|------|
| **User** | 핵심 엔티티. 한 계정에 여러 프로필 연결 가능 |
| **Profile** | 계정 내 개별 프로필. 시청 선호도/설정 |
| **Movie** | 독립적인 콘텐츠 (메타데이터 자체 보유) |
| **TVShow** | 에피소드 여러 개로 구성된 시리즈 |
| **Episode** | TVShow에 속하는 개별 콘텐츠 단위 |
| **WatchHistory** | 프로필별 시청 활동 기록. 이어보기 핵심 |
| **Watchlist** | 나중에 볼 콘텐츠 |
| **Rating** | 프로필이 영화/TV에 매기는 평점 |
| **ContentMetadata** | 비디오 파일 정보 (화질/파일 크기/언어). **적응형 스트리밍의 기반** |

### 핵심 관계

| 관계 | 설명 |
|------|------|
| User → Profile (1:N) | 한 계정에 프로필 여러 개 |
| Movie/TVShow/Episode → ContentMetadata (1:N) | 한 영화당 여러 화질/언어 메타데이터 |
| TVShow → Episode (1:N) | TV 시리즈는 에피소드 여러 개 |
| Profile → WatchHistory/Watchlist/Rating (1:N) | 프로필별 시청 활동 |

> **ContentMetadata가 1:N** 인 게 핵심. 영화 하나에 여러 비트레이트/해상도/언어 메타데이터를 매달아두는 구조. 이게 적응형 스트리밍의 기반이 됨.

## 14.4 시스템 규모 산정

### 가정

| 항목 | 값 |
|------|------|
| 전체 사용자 | 5000만 명 |
| DAU | 1000만 명 |
| 사용자당 일별 시청 | 3편 |
| 비디오 한 편 평균 길이 | 1.5시간 |
| HD 화질 기준 파일 크기 | 3GB |

### 비디오 저장 공간

```
영화 1만 편 + (TV 5000개 × 평균 30화) = 영화 1만 + 에피소드 15만 = 총 16만 편
16만 × 3GB = 480PB
```

### 사용자/메타데이터 저장 공간

- 사용자 기본 정보: 5000만 × 1KB = **50GB**
- 프로필: 5000만 × 5개 × 1KB = **250GB**
- 시청 기록/평점: 5000만 × 5 × 1MB = **250TB**
- 콘텐츠 메타데이터: 16만 × 10KB = **1.6GB**
- 합계: **약 250TB**

### **총 저장**: 비디오 480PB + 메타데이터 250TB ≈ **480PB**

> 비디오가 압도적. 메타데이터는 양념 수준.

### 대역폭

**일일 비디오 스트리밍 대역폭**
- 일평균 스트리밍 횟수: 1000만 DAU × 3편 = 3000만 회
- 일일 전체 대역폭: 3000만 × 3GB = **약 90PB/일**

**피크 시간대 최대 대역폭**
- 동시 접속자 100만 명 × 3GB ÷ 1.5시간 ≈ **2TB/초**

### 처리량

| 항목 | 계산 | 결과 |
|------|------|------|
| 비디오 인코딩 시간 | 100편/일 × 7.5시간 (5가지 비트레이트) | **750시간/일** |
| 추천 요청 처리 | 1000만 × 10회 × 100ms | **약 2.7시간/일** |

### 14.7.3 적응형 비트레이트 스트리밍 ⭐

CDN이 사용자 환경에 맞는 최적 화질 제공.

```
1. 비디오를 여러 비트레이트로 인코딩 → 작은 세그먼트 단위로 나눠 저장
   (예: 240p/360p/480p/720p/1080p/4K)
2. 비디오 재생 중 네트워크 상태 지속 모니터링
3. 상황에 따라 다음 세그먼트의 비트레이트 결정 → 에지 서버에 요청
4. 에지 서버: 해당 요청에 맞는 비디오 세그먼트 전달
5. 비트레이트 실시간 조정 → 끊김 없는 시청 경험
```

**핵심 메커니즘**
- 비디오를 **수 초 단위 세그먼트**로 쪼개서 저장 (예: 6초)
- 매 세그먼트마다 클라이언트가 네트워크 상태 평가
- "지금 1080p 받기 무리네 → 다음 세그먼트는 720p로" 식으로 동적 조절
- 사용자는 화질이 부드럽게 바뀌는 것만 경험

### 14.7.4 콘텐츠 보안 및 DRM

```
[비디오 콘텐츠]
   ↓
[DRM 시스템으로 암호화]
   ├── Microsoft PlayReady
   ├── Google Widevine
   └── Apple FairPlay
   ↓
[CDN] ─→ 암호화된 콘텐츠 전달
   ↓
[Client]
   ↓
[유효한 DRM 라이선스 확인]
   ↓
[정식 사용자만 재생 가능]
```

- 비디오 콘텐츠는 **DRM (Digital Rights Management)** 으로 암호화
- CDN은 DRM 서비스와 연동: 콘텐츠를 암호화 상태로 안전 전달 + 사용자 라이선스 검증
- 무단 접근/복제 방지, 라이선스 조건/지역 제한 준수

### 다른 서비스와의 연결

CDN은 단순 콘텐츠 전송 그 이상으로 핵심 서비스들과 유기적으로 엮여 있음:

- **비디오 서비스** — 인코딩/저장 후 빠른 배포
- **추천 서비스** — 사용자 맞춤 콘텐츠 빠른 전달
- **분석 서비스** — 콘텐츠 전송 성능 지표 → CDN 운영 최적화
- **DRM 서비스** — 라이선스 검증 + 암호화 콘텐츠 안전 전송

## 14.8 요약

이 장에서 다룬 것

- **9개 엔티티** — User/Profile/Movie/TVShow/Episode/WatchHistory/Watchlist/Rating/ContentMetadata
- **저장 480PB** — 비디오가 압도적
- **피크 2TB/초** — 동시 100만 명 스트리밍
- **6개 마이크로서비스 + CDN** — Video/User/Recommendation/Search/Billing/Analytics
- **CDN 자체 설계** — 콘텐츠 배포 + 요청 라우팅 + 적응형 비트레이트 + DRM
- **비디오 인코딩 파이프라인** — 원본 → 트랜스코딩(MP4/HLS/DASH) → 청크 분할 → CDN 배포
- **적응형 비트레이트 (ABR)** — 네트워크 상태에 따라 실시간 화질 조정

| 핵심 도전 | 1:N 팬아웃 | 이미지 파이프라인 | 동시 편집 충돌 | 스트리밍 + 적응형 품질 |
| 저장 규모 | 365PB | 182.5PB | 575TB | 480PB |
| 통신 패턴 | 비동기 (Kafka) | 비동기 + CDN | **양방향 WebSocket** | **CDN + ABR 스트리밍** |
| 일관성 | 최종 일관성 | 최종 일관성 | 실시간 강한 일관성 | 최종 일관성 |
| 핵심 기법 | Fan-out, 캐싱 | 비동기 처리 + CDN | OT/CRDT | 비디오 인코딩 + CDN |

### Netflix Open Connect (책에 없는 진짜 핵심)

책에서 CDN을 일반론으로만 설명하는데, **실제 Netflix는 자체 CDN인 Open Connect**를 운영한다:

- **OCA (Open Connect Appliance)** — Netflix가 직접 만든 CDN 박스. ISP의 데이터 센터 안에 무료로 설치
- 인기 콘텐츠를 **사용자가 잠들기 전 시간에 미리** 캐싱 (Fill window)
- 결과: ISP 백본 트래픽 90% 이상 절약
- 일반 CDN(CloudFront)도 쓰지만 Open Connect가 메인

> 이게 Netflix가 다른 스트리밍 서비스보다 화질이 안정적인 비밀. 트래픽이 ISP 백본을 거치지 않고 같은 ISP 안에서 끝남.

### 비디오 인코딩 파이프라인 — 실무 상세

책엔 "여러 비트레이트로 인코딩"만 짧게 나오지만 실제로는 굉장히 복잡:

1. **포맷 분석** — 입력 파일의 코덱/해상도/프레임레이트 파악
2. **트랜스코딩** — 코덱 변환 (H.264, H.265, AV1, VP9)
3. **다해상도 생성** — 240p/360p/480p/720p/1080p/4K (보통 6-10개 변형)
4. **청크 분할** — HLS/DASH용 세그먼트 (예: 6초 단위)
5. **오디오 분리** — 비디오와 오디오 따로 인코딩 (다국어 지원)
6. **자막 처리** — VTT/SRT 변환
7. **DRM 패키징** — Widevine/PlayReady/FairPlay 키로 암호화
8. **매니페스트 생성** — m3u8 (HLS), mpd (DASH)
9. **QC (Quality Control)** — 인코딩 결과 자동 검증
10. **CDN 배포** — origin → edge 전파

전부 **DAG 형태로 병렬 처리**. Netflix는 자체 인코딩 클러스터 운영 (이전엔 AWS EC2 활용).

### HLS vs DASH 차이

| 항목 | HLS | DASH |
|------|-----|------|
| 개발사 | Apple | 표준 (MPEG) |
| 매니페스트 | .m3u8 (텍스트) | .mpd (XML) |
| 청크 포맷 | TS, fMP4 | fMP4 |
| 브라우저 지원 | Safari 네이티브, 다른 브라우저는 hls.js | dash.js 라이브러리 |
| 코덱 호환 | 보수적 | 유연 (코덱 무관) |

Netflix는 둘 다 지원. iOS는 HLS, Android/웹은 DASH.

### ABR 알고리즘

책엔 "네트워크 상태 보고 조절"이라고만 나오는데, 실제 알고리즘 종류:

- **Throughput-based** — 최근 다운로드 속도 측정해서 결정. 단순하지만 변동에 민감
- **Buffer-based** — 버퍼 잔량 기반. 버퍼 충분하면 화질 ↑, 부족하면 ↓
- **Hybrid (BOLA, MPC)** — 둘 다 고려. 학술적으로 검증됨
- **ML-based** — 강화학습. Netflix가 일부 도입 중

### 추천 시스템 — Netflix Prize의 유산

Netflix는 2009년 100만 달러 상금의 **Netflix Prize**를 통해 추천 시스템 혁신을 일으켰음. 책에 짧게만 다루는데 실무에선:

- **Matrix Factorization** — 사용자-콘텐츠 행렬을 잠재 요인으로 분해
- **Deep Learning** — Two-tower 모델 (사용자 임베딩 × 콘텐츠 임베딩)
- **Multi-armed Bandit** — 새 콘텐츠 탐색 vs 안전한 추천 트레이드오프
- **Personalized Ranking** — 같은 추천 풀이라도 사용자마다 순서 다름

### 비디오 압축 — 코덱 트레이드오프

| 코덱 | 압축률 | 인코딩 비용 | 디바이스 호환성 |
|------|--------|-------------|----------------|
| H.264 (AVC) | 기준 | 낮음 | 거의 모든 기기 |
| H.265 (HEVC) | 30~50% ↑ | 5~10배 ↑ | 라이선스 복잡 |
| VP9 | H.265 수준 | 비슷 | Google 진영 |
| AV1 | 50~70% ↑ | 매우 ↑ | 최신 기기만 |

Netflix는 콘텐츠/시청 환경별로 코덱 다르게 선택. **인코딩 비용 vs 대역폭 비용** 트레이드오프.

### 면접에서 빠뜨리면 안 되는 포인트

- **CDN이 필수** — 전 세계 동시 스트리밍은 단일 origin으로 절대 불가
- **적응형 비트레이트** — 네트워크 환경에 따라 자동 조절
- **여러 포맷/해상도 사전 인코딩** — 1080p, 720p, 480p 등
- **HLS/DASH** — 청크 기반 스트리밍 프로토콜
- **DRM** — 콘텐츠 보호. Widevine/PlayReady/FairPlay
- **추천 시스템** — Cold start 문제, 다양성 유지

### 책에서 다루지 않은 도전 과제

- **Live Streaming** (Twitch, YouTube Live) — VOD와 다른 도전. **저지연 LL-HLS, LL-DASH** 필요
- **콘텐츠 사전 캐싱 전략** — Netflix Open Connect의 Fill window
- **Trick play** — 빨리감기, 되감기, 썸네일 미리보기 (별도 인코딩)
- **다중 오디오 트랙 동기화** — 더빙, 자막 싱크
- **TV 앱 vs 모바일 vs 웹** — 각 플랫폼별 다른 디코더, 다른 DRM
- **국가별 규제** — 지역별 콘텐츠 차단, 콘텐츠 등급
- **다운로드 후 시청** — 모바일에서 미리 다운받아 오프라인 시청 (책에 짧게만 나옴)
